# 📚 项目二：日志收集系统

## 项目目标

这个项目的目标不是单纯把几个 YAML 跑起来，而是带你理解：

- Kubernetes 中应用日志通常从哪里来
- 为什么不能把日志收集逻辑直接塞进业务容器
- DaemonSet、Volume、ConfigMap、Service 这些对象如何一起工作
- 一个“节点采集 -> 聚合存储 -> 检索分析”的日志链路该怎么拆

如果你能把这个项目理解清楚，就会对日志系统在 Kubernetes 里的部署方式有比较扎实的认识。

## 这个项目会模拟什么

我们用一个简化但很典型的日志架构来学习：

```text
业务 Pod 日志
  -> 节点上的日志文件
  -> 日志采集器（DaemonSet）
  -> 日志聚合后端
  -> 查询或可视化入口
```

可以把它理解为：

- 每个节点都需要一个“收集员”
- 集群里还需要一个“集中存放和查询的地方”

## 推荐学习重点

做这个项目时，建议优先关注下面这些问题：

1. 为什么日志采集器通常用 `DaemonSet`
2. 为什么采集器要挂载宿主机目录
3. 业务日志、节点日志、容器 stdout/stderr 日志之间是什么关系
4. 采集失败时应该优先查容器、卷挂载还是网络连通

## 会用到哪些 Kubernetes 对象

| 对象 | 作用 |
|------|------|
| `Namespace` | 隔离日志系统资源 |
| `DaemonSet` | 保证每个节点都有一个日志采集器 |
| `ConfigMap` | 存放采集规则、输出目标、解析配置 |
| `Volume` | 挂载宿主机日志目录和配置文件 |
| `Deployment` / `StatefulSet` | 运行日志聚合后端 |
| `Service` | 为日志后端或查询入口提供稳定访问方式 |
| `Ingress` | 如果需要浏览器访问 UI，可暴露 HTTP 入口 |

## 推荐部署顺序

1. 创建命名空间
2. 部署日志后端存储或聚合服务
3. 部署日志采集器配置
4. 部署日志采集器 `DaemonSet`
5. 如需可视化，再部署 UI
6. 生成测试日志并验证采集链路

## 最小验证思路

即使你现在还没有完整的日志平台，也可以先验证日志收集思路是否正确：

### 步骤 1：准备一个持续输出日志的应用

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: log-demo
spec:
  containers:
  - name: app
    image: busybox
    command:
    - sh
    - -c
    - while true; do echo "$(date) hello from kubernetes"; sleep 5; done
```

### 步骤 2：观察日志是否持续输出

```bash
kubectl logs log-demo -f
```

### 步骤 3：让采集器去读取节点上的容器日志目录

这一步的重点不是背具体路径，而是理解：

- 采集器需要“看到节点上的日志文件”
- 所以它通常需要挂载宿主机目录
- 这也是为什么这类组件常常用 `DaemonSet`

## DaemonSet 为什么特别适合日志采集

因为日志采集往往是“每个节点都必须有一个实例”的问题。

这与 Deployment 很不同：

- Deployment 关注副本数
- DaemonSet 关注节点覆盖率

日志采集器适合 DaemonSet 的原因：

- 每个节点都能采集本机日志
- 节点新增时会自动补一个实例
- 节点下线时实例也会随之消失

## 日志系统最常见的几种数据来源

### 1. 容器标准输出

这是 Kubernetes 中最常见、也最推荐的应用日志方式：

- 应用把日志写到 stdout/stderr
- 容器运行时和节点负责落盘
- 采集器再从这些日志文件中读取

### 2. 应用文件日志

有些老应用仍会把日志写到容器内部文件中：

- 需要额外挂载卷
- 采集器需要能访问该卷
- 管理复杂度通常更高

### 3. 节点系统日志

例如：

- kubelet
- container runtime
- node system logs

这类日志更偏平台运维视角。

## 常见排查问题

### 问题 1：采集器 Pod 没起来

优先检查：

- `kubectl get ds -A`
- `kubectl describe ds <name>`
- `kubectl get pods -o wide`
- 是否有节点污点或镜像拉取问题

### 问题 2：采集器起了，但没有日志

优先检查：

- 宿主机日志目录是否挂载成功
- 采集配置是否正确
- 采集路径是否匹配实际日志路径
- 应用是否真的在输出日志

### 问题 3：日志后端收不到数据

优先检查：

- 采集器到后端的网络连通性
- Service 名称、端口、DNS 是否正确
- 后端鉴权或输入协议是否匹配

### 问题 4：日志量很大，节点压力高

优先检查：

- 采集器资源配置是否合理
- 是否存在重复采集
- 是否对高频无价值日志做过滤
- 是否做了限流或采样

## 最佳实践

- 优先让应用输出到 stdout/stderr，而不是写复杂文件路径
- 日志采集器优先使用 `DaemonSet`
- 配置要通过 `ConfigMap` 管理，而不是写死在镜像里
- 日志后端通常需要持久化存储
- 高噪声日志要尽早过滤，否则成本会很高
- 生产环境要区分访问日志、应用日志、审计日志、系统日志

## 做完这个项目后你应该掌握什么

完成这个项目后，至少应该能讲清楚下面这些问题：

- 为什么日志采集器通常是 DaemonSet
- 为什么日志系统通常不直接和业务容器绑死
- 为什么日志后端往往要配持久卷
- 日志从业务 Pod 到集中检索系统，大致会经过哪些环节
