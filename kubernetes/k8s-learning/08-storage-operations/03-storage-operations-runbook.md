# 🛠️ 存储运维手册：扩容、迁移、备份、排障

## 为什么存储运维和普通对象运维完全不是一个难度

Deployment 出问题时，你很多时候可以：

- 回滚
- 重建
- 扩副本
- 删除再拉

但存储和数据问题不一样。  
因为一旦操作失误，可能带来的不是“服务短暂不可用”，而是：

- 数据损坏
- 数据丢失
- 无法回滚
- 恢复窗口过长
- 业务状态不一致

所以存储运维最重要的原则不是“快”，而是：

> 先判断风险，再决定动作；先保护数据，再处理系统。

## 1. 一条总原则：先搞清楚你操作的是哪一层

存储运维时常见对象包括：

- Pod
- StatefulSet
- PVC
- PV
- StorageClass
- 底层云盘 / 文件系统 / 存储集群

你必须先判断问题在哪一层，否则很容易“在错误的层做正确的操作”，结果依然出事故。

例如：

- Pod 起不来，不一定是应用问题，也可能是卷没挂上
- PVC Pending，不一定是 YAML 写错，也可能是 CSI 供给器异常
- 服务数据丢失，不一定是业务 bug，也可能是卷被回收了

## 2. 存储运维前的通用检查清单

在做任何可能影响数据的操作前，建议先确认：

1. 业务是否允许中断
2. 当前数据是否已有备份或快照
3. PV / PVC 的绑定关系是否清楚
4. 回收策略是不是 `Delete`
5. 底层存储是否有额外自动清理策略
6. 操作对象是测试环境还是生产环境

如果这些都没确认清楚，先不要动。

## 3. 常见运维动作 1：PVC 扩容

### 3.1 扩容前先确认什么

PVC 扩容看起来简单，但不是所有环境都支持。  
至少要确认：

- StorageClass 允许扩容
- 后端存储支持扩容
- CSI 驱动支持扩容
- 文件系统扩容链路可用

### 3.2 基本做法

通常是修改 PVC 请求容量：

```yaml
spec:
  resources:
    requests:
      storage: 200Gi
```

然后：

```bash
kubectl apply -f pvc.yaml
kubectl describe pvc <pvc-name>
```

### 3.3 扩容时的典型风险

- 只是 API 字段变了，底层卷没真正扩容
- 底层卷扩容成功，但文件系统没扩
- 应用没识别新容量
- 正在高负载写入时扩容，影响业务性能

所以扩容完成后，除了看资源对象，还要看应用实际是否识别容量变化。

## 4. 常见运维动作 2：卷迁移

卷迁移通常发生在这些场景：

- 更换 StorageClass
- 更换存储后端
- 集群迁移
- 从本地卷迁到云盘
- 从低性能卷迁到高性能卷

### 4.1 卷迁移的核心步骤

常见思路通常是：

1. 创建新卷
2. 停止或冻结写入
3. 把旧卷数据复制到新卷
4. 切换业务到新卷
5. 验证后再回收旧卷

### 4.2 最危险的点

最危险的通常不是复制本身，而是：

- 业务在迁移时仍持续写入
- 切换窗口没有控制住
- 验证不充分就删旧卷

所以对数据库类应用，迁移前通常还要考虑：

- 是否需要只读窗口
- 是否有复制机制
- 是否需要应用级别一致性保障

## 5. 常见运维动作 3：备份与恢复

这部分是最关键的生产能力之一。

### 5.1 备份不等于复制 PVC

很多人会误以为：

- 有 PVC
- 或者有 PV
- 就算有备份能力

这通常是不够的。

真正的备份能力要回答：

- 数据在哪个时间点被保存
- 能否恢复到指定时刻
- 恢复需要多久
- 恢复流程是否演练过

### 5.2 常见备份手段

#### 卷快照

适合：

- 块存储支持快照
- 需要快速保留某个时间点状态

但要注意：

- 快照能力取决于 CSI 和后端存储
- 快照不等于业务一致性

#### 应用级备份

例如：

- MySQL dump
- PostgreSQL base backup
- Redis RDB/AOF

这类方式更贴近业务数据结构，但恢复流程通常更依赖应用本身。

#### 文件级备份

适合共享文件或文件型业务，但恢复粒度和一致性要单独评估。

### 5.3 恢复前最该确认什么

恢复不是“把数据拉回来”这么简单。  
你要先确认：

- 恢复目标环境在哪
- 恢复后是否覆盖现网
- 是否允许数据回退
- 应用版本是否兼容历史数据
- 是否需要先停业务流量

## 6. 常见运维动作 4：删除与回收

这是最容易出事故的一类动作。

### 6.1 删除 Pod 和删除 PVC 的区别

- 删除 Pod：通常只是删除计算实例
- 删除 PVC：可能影响真实卷和数据生命周期

所以你可以随手删 Pod，但绝对不能同样心态去删 PVC。

### 6.2 删除 PVC 前必须确认

- 这块数据是不是关键数据
- PVC 当前绑定的是哪个 PV
- PV 回收策略是不是 `Delete`
- 底层卷是否会被自动删
- 是否已有快照或备份

## 7. 有状态工作负载升级时怎么做

数据库、中间件、有状态服务升级时，最重要的不是先改镜像，而是先判断升级对数据有什么影响。

### 常见检查项

- 数据格式是否兼容
- 是否支持滚动升级
- 是否需要停机窗口
- 升级失败如何回滚
- 升级前是否有快照 / 备份

### 典型风险

- 镜像升级了，但存储数据格式不兼容
- 旧副本和新副本无法同时工作
- 回滚镜像后数据却已被升级过，无法回退

这类问题非常常见。

## 8. 存储排障总链路

当你遇到存储问题时，可以按下面顺序查：

1. 业务表现是什么
2. Pod 当前状态是什么
3. PVC 是不是 `Bound`
4. PV 是否正常、回收策略是什么
5. StorageClass 和 CSI 是否正常
6. 节点是否成功挂载卷
7. 底层存储系统是否健康

这条链条能帮助你避免只盯一层。

## 9. 常见场景排障手册

### 场景 1：PVC 一直 `Pending`

优先命令：

```bash
kubectl get pvc
kubectl describe pvc <pvc-name>
kubectl get storageclass
kubectl get pv
```

常见原因：

- 没有默认 StorageClass
- 指定的 StorageClass 不存在
- 动态供给器异常
- 容量申请不合理
- accessModes 不匹配

### 场景 2：Pod 卡在 `ContainerCreating`

优先命令：

```bash
kubectl describe pod <pod-name>
```

重点看：

- 挂载卷事件
- 节点 attach/mount 错误
- PVC 是否真的已经 `Bound`

### 场景 3：Pod 起来了，但应用报“磁盘不可写”

优先检查：

- 卷是否只读挂载
- 容器用户权限是否匹配
- 文件系统权限或属主问题
- 后端卷是否被异常保护成只读

### 场景 4：数据疑似丢失

优先检查：

- 当前挂载的是否还是原来的 PVC
- PVC 是否重新绑定到了不同卷
- 原 PV 是否已经被回收
- 应用是不是写到了临时卷

### 场景 5：节点故障后 Pod 起不来

优先检查：

- 卷是否能重新附加到新节点
- 是否有拓扑限制
- 是否是本地卷导致无法迁移
- 后端存储是否支持多节点或跨区恢复

## 10. 最常用的诊断命令清单

```bash
# 看 Pod
kubectl get pods -A
kubectl describe pod <pod-name>

# 看 PVC / PV
kubectl get pvc -A
kubectl describe pvc <pvc-name>
kubectl get pv
kubectl describe pv <pv-name>

# 看 StorageClass
kubectl get storageclass
kubectl describe storageclass <sc-name>

# 看事件
kubectl get events --sort-by=.metadata.creationTimestamp

# 看节点
kubectl get nodes -o wide
kubectl describe node <node-name>
```

## 11. 生产环境特别需要避免的操作

- 未确认备份就删除 PVC
- 在高峰期直接改存储类或迁移卷
- 未验证恢复流程就认为“有快照就安全”
- 对数据库类应用只看 Kubernetes 状态，不看应用内部一致性
- 看到 Pod 起不来就直接删资源重建

这些操作在无状态服务里有时还能补救，在有状态服务里很容易出大事故。

## 12. 推荐的运维习惯

- 关键卷操作前固定执行“备份/快照确认”
- 把 PV/PVC/StorageClass 关系记录清楚
- 给关键业务设计标准化恢复手册
- 重大变更先在测试环境做全流程演练
- 让应用团队和平台团队对数据责任边界达成一致

## 13. 一页总结

- 存储运维首先要分清操作层级：Pod、PVC、PV、StorageClass、后端存储
- 扩容、迁移、备份、恢复都不是单纯改 YAML 那么简单
- 删除 PVC 前必须确认回收策略和数据风险
- 存储问题排查不能只盯 Kubernetes 对象，还要看底层存储和业务一致性
- 关键数据场景里，真正重要的是“恢复能力”，不只是“能不能创建卷”
