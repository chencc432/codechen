# 🗄️ Kubernetes 存储与运维专题

## 这个专题解决什么问题

Kubernetes 学到后面，真正最容易让人踩坑、最影响生产稳定性的，往往不是 Deployment 或 Service，而是：

- 存储
- 数据
- 卷挂载
- 扩容迁移
- 备份恢复
- 故障处理

很多应用“能跑起来”，但并不代表“能长期稳定运行”。  
一旦涉及：

- 数据库
- 日志系统
- 消息队列
- 文件共享
- 备份恢复

你就必须真正理解 Kubernetes 的存储体系，以及围绕存储展开的运维方式。

这个专题就是为了解决这些问题。

读完这个专题后，你应该能讲清楚这些问题：

- `Volume`、`PV`、`PVC`、`StorageClass`、`CSI` 是怎么串起来的
- `RWO`、`ROX`、`RWX` 各自适合什么场景
- 为什么有些卷能扩容，有些不行
- 删除 PVC 之前为什么一定要看 `reclaimPolicy`
- 有状态应用上云或迁移时，最该先确认什么
- 存储问题排查时，为什么不能只盯着 Pod

## 阅读时建议重点盯住的几个问题

如果你想把这部分真正学扎实，建议每篇都带着下面这些问题去读：

1. 这类存储对象是谁声明的，谁负责供应，谁负责消费
2. 数据会不会因为 Pod 重建而丢失
3. 存储故障时，风险到底发生在 Pod、PVC、PV、节点还是后端存储
4. 这个方案更适合开发环境，还是生产环境

## 一条最重要的理解主线

如果你只先记住一句话，那就是：

> Kubernetes 管的是“如何声明、绑定、挂载和维护存储使用关系”，真正的数据可靠性仍然依赖后端存储系统和你的运维策略。

换句话说：

- Kubernetes 帮你编排存储使用
- 但不自动替你解决所有数据可靠性问题

这点非常关键。

## 新增进阶阅读

为了让这个专题更接近“教科书式”的体系化内容，当前还额外补充了两类很关键的章节：

- **方案对比类**：帮助你在不同存储后端之间建立选型能力，而不是只会背概念
- **恢复演练类**：帮助你把“有备份”升级成“真的会恢复、真的演练过”
- **场景差异类**：帮助你理解为什么数据库、中间件、日志系统不能套用同一套存储思路
- **故障剧本类**：帮助你把“知道对象”升级成“遇到事故时知道第一步查什么”

这些部分通常是很多团队真正进入生产阶段后，最需要但最容易缺失的能力。
