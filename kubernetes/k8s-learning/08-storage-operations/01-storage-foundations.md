# 🧱 存储基础与架构认知

## 为什么说存储是 Kubernetes 里最容易“看似简单、实际复杂”的部分

很多人初学 Kubernetes 时，最先上手的是：

- Pod
- Deployment
- Service

这些对象确实很重要，但它们主要在解决“计算和流量”的问题。  
一旦开始部署数据库、缓存、日志、对象处理、文件服务，就会立刻进入另一个难度层级：

- 数据怎么保留
- 卷怎么挂载
- Pod 重建后数据还在不在
- 节点换了以后数据能不能跟着走
- 删除 PVC 会不会把数据删掉

所以存储之所以难，不是因为它字段多，而是因为它同时涉及：

- Kubernetes 对象层
- 节点层
- 后端存储系统层
- 业务数据一致性层

## 1. 先建立整体心智模型

Kubernetes 存储可以先拆成 5 层来看：

1. **容器使用层**：容器通过 `volumeMounts` 使用存储
2. **Pod 定义层**：Pod 通过 `volumes` 声明卷
3. **资源绑定层**：PVC 申请存储，PV 提供存储
4. **供应策略层**：StorageClass 决定怎么动态提供卷
5. **后端实现层**：NFS、云盘、Ceph、EBS、CSI 驱动等

可以画成：

```text
应用容器
  -> volumeMounts
  -> Pod volumes
  -> PVC
  -> PV
  -> StorageClass
  -> 后端存储系统
```

### 这条链为什么重要

因为实际排障时你经常需要判断：

- 问题出在容器挂载
- 还是 PVC 没绑定
- 还是 PV 不匹配
- 还是 StorageClass 没动态供给
- 还是底层存储本身就异常

## 2. 临时卷与持久卷

### 2.1 临时卷

最常见代表是 `emptyDir`。

特点：

- 跟着 Pod 生命周期走
- Pod 删除后数据消失
- 适合缓存、临时文件、中间结果

适用场景：

- 临时处理目录
- 多容器共享中间文件
- 非关键短期数据

不适用场景：

- 数据库数据目录
- 业务持久数据
- 需要跨 Pod 保留的数据

### 2.2 持久卷

持久卷的目标是：

- Pod 删除重建后仍可保留数据
- 存储与计算解耦
- 集群可以用统一方式管理数据卷

这时就需要：

- `PV`
- `PVC`
- `StorageClass`

## 3. Volume、PV、PVC 的关系

### 3.1 先用一句最简单的话记忆

- `Volume`：Pod 里的卷入口
- `PVC`：业务对存储的申请单
- `PV`：集群里的实际卷资源

### 3.2 用生活类比理解

| 对象 | 类比 |
|------|------|
| `PVC` | 你提出“我要一块 100G 磁盘”的申请 |
| `PV` | 仓库里真实存在的一块盘 |
| `StorageClass` | 说明“这类盘应该按什么规格创建” |
| `Volume` | 最终挂到你房间里的那根数据线接口 |

## 4. 为什么 Pod 不直接使用 PV

这是一个很重要但经常被忽视的设计点。

Kubernetes 不鼓励业务直接依赖某个固定 PV，而是通过 PVC 做一层抽象，好处有：

- 业务只声明需求，不关心底层卷细节
- 存储管理员和业务使用者职责分离
- 更方便动态供应和统一治理
- 更利于迁移与标准化

所以通常是：

- 业务写 PVC
- Pod 用 PVC
- 系统再把 PVC 绑定到 PV

## 5. Access Modes 到底怎么理解

访问模式是存储设计和排障里非常关键的一环。

| 模式 | 含义 | 常见使用理解 |
|------|------|--------------|
| `ReadWriteOnce` | 单节点读写 | 最常见，块存储常见 |
| `ReadOnlyMany` | 多节点只读 | 静态内容分发、共享只读 |
| `ReadWriteMany` | 多节点读写 | 共享文件系统 |
| `ReadWriteOncePod` | 单 Pod 独占读写 | 更强隔离语义 |

### 最容易混淆的一点

`ReadWriteOnce` 不是“只能一个 Pod 用”，而更接近：

> 同一时刻通常只能被一个节点以读写方式挂载。

所以很多人误以为：

- RWO 卷完全不能被多个 Pod 使用

其实更准确地说，要看这些 Pod 是否位于同一节点、底层存储是否支持、调度是否允许。

## 6. `reclaimPolicy` 为什么删除前一定要看

存储出事故时，很多时候不是“创建失败”，而是“误删了”。  
而误删里最危险的一个点就是没看 `reclaimPolicy`。

常见策略：

| 策略 | 含义 | 典型风险/场景 |
|------|------|---------------|
| `Retain` | 删除 PVC 后保留 PV 和数据 | 需要人工回收，适合关键数据 |
| `Delete` | 删除 PVC 后连后端卷一起删 | 适合临时环境，但风险大 |
| `Recycle` | 旧机制，基本不用 | 不建议依赖 |

最重要的实践建议：

- 删除 PVC 前一定先确认数据重要性
- 再确认 PV 的回收策略
- 再确认底层存储会不会被自动清理

## 7. 存储方案按后端类型大致怎么分

### 7.1 本地路径类

例如 `hostPath`。

特点：

- 简单
- 适合开发测试
- 强依赖单节点

不适合生产核心场景，因为：

- 节点坏了数据风险高
- Pod 调度到别处就可能挂不上原数据

### 7.2 共享文件系统类

例如 NFS、CephFS。

特点：

- 适合 `RWX`
- 多节点共享方便
- 常用于文件共享类场景

### 7.3 云块存储类

例如云盘、EBS、云硬盘等。

特点：

- 常见于云环境
- 往往适合 `RWO`
- 性能和可靠性通常比本地路径更可控

### 7.4 分布式存储 / CSI 生态

现代 Kubernetes 更常见的是通过 CSI 接入各种存储系统。

这带来的好处是：

- 存储接入标准化
- 动态供给更方便
- 不同供应商能力更容易统一接入

## 8. 有状态应用为什么更难

无状态应用的本质是：

- 实例没了，再拉一个就行

有状态应用则不同：

- 数据要保留
- 身份可能要稳定
- 主从、选主、备份、恢复都更复杂

这也是为什么数据库、中间件往往需要同时考虑：

- `StatefulSet`
- `PVC`
- `StorageClass`
- 备份恢复
- 扩容与迁移

## 9. 存储与调度为什么会互相影响

很多人会把存储和调度分开学，但在真实场景里它们经常是耦合的。

例如：

- 卷只能挂某个区域
- 节点不在同一可用区
- `WaitForFirstConsumer` 会影响卷创建时机
- 本地卷绑定了特定节点

这意味着：

- PVC 问题有时看起来像调度问题
- 调度问题有时根因其实是存储约束

## 10. 存储问题为什么不能只看 Pod

这是很重要的一点。

当存储出问题时，Pod 往往只是最后表现出症状的一层。  
真正的问题可能出在：

- PVC 一直 `Pending`
- PV 绑定失败
- StorageClass 没有供应器
- CSI 驱动异常
- 节点无法附加卷
- 后端存储本身故障

所以排障时通常应该一起看：

```bash
kubectl get pod
kubectl get pvc
kubectl get pv
kubectl get storageclass
kubectl describe pod <name>
kubectl describe pvc <name>
```

## 11. 一套最常见的存储排障思路

### 场景 1：Pod 卡在 `ContainerCreating`

优先怀疑：

- 卷还没挂成功
- 节点附加卷失败
- PVC 未绑定
- CSI 驱动问题

### 场景 2：PVC 一直 `Pending`

优先怀疑：

- 没有匹配的 PV
- StorageClass 配置不对
- 动态供给器异常
- 访问模式或容量不匹配

### 场景 3：Pod 重建后数据没了

优先怀疑：

- 用的是 `emptyDir`
- 绑定到了新卷
- 原 PVC / PV 被删
- 底层卷已被回收

## 12. 一页总结

- Kubernetes 存储不是单一对象，而是一条从容器到后端存储的完整链路
- `Volume/PVC/PV/StorageClass` 各自职责不同，不能混看
- 数据可靠性不只取决于 Kubernetes，也取决于后端存储系统和运维策略
- 有状态应用比无状态应用更依赖正确的存储设计
- 存储排障不能只看 Pod，必须连 PVC、PV、StorageClass 一起看
