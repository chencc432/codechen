# 第四册：后训练数据工程与评测体系

> 如果只选一个最值得投入精力的方向，那通常不是“再换一个训练算法”，而是“把数据工程和评测体系做好”。
>
> 因为大多数后训练失败，并不是方法名字不够高级，而是：
>
> - 数据脏
> - 数据不对目标
> - 数据风格冲突
> - 评测不完整
> - 只看 loss，不看业务效果

---

## 1. 数据为什么是后训练的核心

大模型后训练的一个基本现实是：

> 模型的下限由数据决定，训练方法很多时候只是把数据里的规律放大。

如果数据本身有问题，那么训练往往会放大这些问题：

- 风格不统一 -> 输出越来越乱
- 错误答案太多 -> 幻觉和错误更稳
- 重复数据过高 -> 模型复读、过拟合
- 格式混乱 -> 推理阶段不听模板

---

## 2. 后训练数据的分类

### 2.1 按训练目标分类

| 类型 | 适用方法 | 核心内容 |
|------|----------|----------|
| 指令问答数据 | SFT | 输入与标准输出 |
| 领域文本数据 | CPT | 纯文本语料 |
| 偏好对数据 | DPO / ORPO / SimPO | chosen / rejected |
| 标签反馈数据 | KTO | 好坏标签 |
| 奖励可验证数据 | GRPO | 问题、回答、程序化打分 |

### 2.2 按来源分类

- 人工编写
- 业务日志加工
- 知识库抽取
- 程序规则生成
- 强模型蒸馏
- 用户反馈回流

### 2.3 按质量等级分类

建议至少把数据按等级分层：

- `S 级`：高价值、人工确认、可作为核心训练集
- `A 级`：高质量自动生成或人工抽检通过
- `B 级`：可用但噪声偏大
- `C 级`：只适合探索，不应直接进核心训练

---

## 3. 构建训练集时的关键原则

### 3.1 目标先行，而不是数据先行

很多团队的错误顺序是：

```text
先抓到很多数据 -> 再想这些数据能训什么
```

正确顺序应该是：

```text
先定义目标任务 -> 再决定要收集什么数据
```

你必须先明确：

- 训练后模型要提升什么能力
- 提升的标准是什么
- 上线成功标准是什么

### 3.2 高质量小数据，通常优先于低质量大数据

尤其在 SFT 阶段，这个原则非常重要。

因为模型不是没有基础能力，而是要学习你的目标行为。此时：

- 高质量样本能快速塑造行为
- 低质量样本会污染原本不错的模型

### 3.3 要覆盖“成功场景”，也要覆盖“失败边界”

很多数据集只覆盖理想样本，却没有覆盖：

- 模糊问题
- 易混淆问题
- 恶意输入
- 格式陷阱
- 过长上下文
- 工具调用失败情况

没有这些边界样本，模型上线后很容易在真实环境里出问题。

---

## 4. SFT 数据应该怎么设计

### 4.1 一个好样本的基本结构

好样本至少应该满足：

- 用户问题真实
- 任务目标清晰
- 输出与目标一一对应
- 风格稳定
- 结构可复用

### 4.2 单轮任务要注意什么

对于抽取、分类、摘要、翻译这类任务，最关键的是：

- 指令表达清楚
- 输出边界清楚
- 格式尽量稳定

例如你要训 JSON 输出，建议样本中明确统一：

- 字段名
- 字段顺序
- 空字段处理方式
- 异常值表达方式

### 4.3 多轮对话要注意什么

多轮对话数据最容易出问题的地方：

- 历史轮次缺失
- system 指令不统一
- 用户上下文与回复不匹配
- 中间轮次逻辑断裂

因此多轮数据一定要保证：

- 上下文完整
- 角色顺序正确
- 每一轮的语义衔接自然

### 4.4 长上下文任务要注意什么

如果你的任务是文档问答、长文本摘要、长对话记忆，那么训练样本必须包含足够多的长上下文样本，否则模型上线时会出现：

- 长上下文下格式漂移
- 关键信息漏提
- 无法正确引用前文

---

## 5. 偏好数据应该怎么设计

### 5.1 好的偏好对不是“差距越大越好”

很多人做 DPO 数据时，会把 chosen 写得很好，rejected 写得极差。

这样虽然容易标，但问题是：

- 模型学到的是粗粒度区分
- 学不到“两个都还行时，哪个更优”

更有价值的偏好对通常是：

- 两个答案都看起来像样
- 但 chosen 在正确性、完整性、简洁性、安全性中至少一项更好

### 5.2 偏好标注标准必须写成规则

不能只靠“标注员感觉哪个好”。

应该明确：

- 优先正确性还是优先风格
- 冗长是否扣分
- 幻觉是否一票否决
- 安全问题权重多大

### 5.3 偏好数据的覆盖面

偏好数据不能只集中在一个领域。

至少要覆盖：

- 常规任务
- 高风险任务
- 容易幻觉的任务
- 容易格式错乱的任务
- 容易过度拒答的任务

---

## 6. 数据清洗的完整流程

### 6.1 基础清洗

必须做的基础项：

- 编码统一
- JSON 结构校验
- 空值过滤
- 超长样本检查
- 特殊字符清理
- 标签字段一致性检查

### 6.2 语义清洗

进一步建议做：

- 去重
- 相似样本聚类
- 错误答案过滤
- 低信息量样本过滤
- 风格异常样本过滤

### 6.3 业务清洗

和业务强相关的检查：

- 是否泄露隐私
- 是否包含敏感字段
- 是否包含过期知识
- 是否违反业务规范
- 是否和最新产品能力冲突

---

## 7. 去重为什么重要

### 7.1 重复样本的危害

- 模型记住少数模板
- 表达风格僵化
- 训练 token 浪费
- 评测结果虚高

### 7.2 去重至少分两层

1. 精确去重
2. 语义相似去重

精确去重解决完全重复问题，语义去重解决“换个说法但本质一样”的问题。

### 7.3 去重不是越狠越好

如果去重策略太激进，也会误删真正有价值的不同样本。

因此更合理的做法是：

- 先高置信精确去重
- 再人工抽样核查语义去重策略

---

## 8. 数据混配怎么做

很多团队最终都不是单一数据源，而是多源混合：

- 通用指令数据
- 业务数据
- 安全数据
- 格式数据
- 工具调用数据

### 8.1 混配的核心问题

不同数据源权重怎么分配。

如果业务数据占比太低：

- 模型学不到业务能力

如果业务数据占比太高：

- 可能损害通用能力

### 8.2 实战建议

不要一开始就追求完美配比，可以采用：

1. 先做一个可解释的初始配比
2. 分任务做验证集
3. 根据评测结果再迭代配比

### 8.3 常见混配错误

- 全靠直觉，不做验证
- 所有数据一股脑混进来
- 不同模板数据混用却不统一模板
- 训练集混配了，验证集却没分任务统计

---

## 9. 训练集、验证集、测试集怎么拆

### 9.1 为什么必须拆分

因为你必须区分：

- 模型是否学会训练数据
- 模型是否能泛化到未见数据
- 模型是否真的能在业务场景成功

### 9.2 三者的角色

| 集合 | 作用 |
|------|------|
| 训练集 | 用来更新参数 |
| 验证集 | 用来选模型、选超参数、早停 |
| 测试集 | 用来做最终效果判断 |

### 9.3 最常见的错误

- 测试集泄漏进训练集
- 用验证集反复调到“过拟合验证集”
- 测试集覆盖面太窄

### 9.4 一个实用建议

测试集最好分层构造：

- 主流程样本
- 难样本
- 边界样本
- 线上高频真实样本

---

## 10. 为什么 loss 好看，效果却不好

这是训练里最常见的误区之一。

### 10.1 可能原因 1：目标函数和业务目标不一致

loss 优化的是训练目标，不一定等于业务 KPI。

比如你真正关心的是：

- 结构化字段准确率
- 用户任务完成率
- 工具调用成功率

而训练 loss 只是 token 级拟合程度。

### 10.2 可能原因 2：数据分布偏了

如果训练集过于单一，loss 当然可以降得很漂亮，但一到真实场景就暴露问题。

### 10.3 可能原因 3：评测集不对

如果评测集本身和线上任务不一致，你看到的“效果好”只是评测假象。

---

## 11. 评测体系应该怎么搭

### 11.1 最低限度的四层评测

建议至少建立以下四层：

1. 训练中监控
2. 离线自动评测
3. 人工评测
4. 线上业务评测

### 11.2 训练中监控看什么

- train loss
- eval loss
- 学习率曲线
- 梯度异常
- 显存占用
- 吞吐

### 11.3 离线自动评测看什么

根据任务不同选择：

- 分类准确率
- 抽取 F1
- 代码单测通过率
- 数学题正确率
- 格式正确率
- 事实一致性

### 11.4 人工评测看什么

人工评测要补上自动指标看不到的东西：

- 是否真正有帮助
- 是否自然
- 是否幻觉
- 是否啰嗦
- 是否安全

### 11.5 线上评测看什么

最终要落到业务指标：

- 用户满意度
- 人工介入率
- 工具调用成功率
- 工单完成率
- 投诉率

---

## 12. 一个可执行的评测模板

### 12.1 对问答模型

建议至少打这几类标签：

- 正确性
- 完整性
- 简洁性
- 幻觉情况
- 风格符合度

### 12.2 对结构化输出模型

建议至少看：

- JSON 是否可解析
- 字段是否齐全
- 字段值是否正确
- 异常输入时是否稳定

### 12.3 对代码模型

建议至少看：

- 编译是否通过
- 单元测试通过率
- 安全风险
- 冗余代码比例

---

## 13. 数据工程中的常见问题

### 13.1 过拟合

常见表现：

- 训练集指标高
- 测试集表现差
- 模型复读或模板化严重

常见原因：

- 数据太少
- 重复过高
- epoch 太多

### 13.2 风格漂移

常见原因：

- 数据源风格不统一
- 混配比例失衡
- 对话模板不统一

### 13.3 任务互相冲突

例如：

- 一部分数据要求详细解释
- 另一部分要求只输出 JSON

如果没有明确区分场景和指令，模型就会学得很混乱。

---

## 14. 一套推荐的数据工程流水线

```text
明确目标
-> 列出任务类型
-> 设计数据 schema
-> 采集原始数据
-> 规则清洗
-> 人工抽检
-> 去重
-> 分层打标
-> 划分训练/验证/测试集
-> 小规模训练验证
-> 分析失败样本
-> 回流修订数据
```

这里面最重要的一步是：

> 分析失败样本并回流数据。

因为高质量数据集通常不是“一次写出来的”，而是“多轮迭代磨出来的”。

---

## 15. 本册总结

- 数据工程决定后训练效果的下限和稳定性
- 高质量样本通常比大量脏样本更有价值
- 偏好数据的标注标准必须明确
- loss 不能替代完整评测
- 训练集、验证集、测试集必须角色分离
- 离线评测和线上业务指标必须打通
