# 第二册：后训练核心：SFT 与参数高效微调

> 这是全套教材最重要的一册。
>
> 如果说预训练是在“造一个大脑”，那么后训练就是在“把这个大脑训练成一个能工作的员工”。现实里的大模型业务落地，绝大多数工作都发生在这里。

---

## 1. 什么是后训练

后训练（Post-Training）是指：

> 在一个已经训练好的基础模型或指令模型上，继续用任务数据做进一步训练，让它更适合特定用途。

### 1.1 后训练到底解决什么问题

基础模型虽然能力强，但通常存在以下问题：

- 不够听指令
- 输出风格不稳定
- 不知道你的领域术语
- 不能稳定输出你想要的结构化格式
- 回答可能啰嗦、答非所问、拒答不稳

后训练的任务，就是把这些“通用能力”变成“业务可用能力”。

### 1.2 后训练的常见目标

- 指令遵循
- 格式对齐
- 领域知识增强
- 偏好对齐
- 安全边界控制
- 推理能力优化
- 工具调用习惯塑造

---

## 2. 后训练的主要类型

```text
后训练
  ├─ SFT：监督微调
  ├─ CPT：继续预训练
  ├─ PEFT：参数高效微调（LoRA / QLoRA / DoRA 等）
  └─ Preference / RL：偏好对齐与强化学习
```

这一册先讲最核心、最常见、最实用的部分：

- `SFT`
- `Full Fine-Tuning`
- `LoRA`
- `QLoRA`
- `DoRA`
- `AdaLoRA`

---

## 3. SFT 是什么，为什么它最重要

SFT（Supervised Fine-Tuning）就是监督微调。

最朴素的理解：

> 给模型看“输入应该对应什么输出”，让模型学会模仿这种映射关系。

### 3.1 一个最简单的例子

```json
{"instruction": "请把下面这句话翻译成英文：我喜欢学习大模型训练。", "output": "I enjoy learning large model training."}
```

训练时，模型看到这条数据，会尝试生成输出。然后它的输出和标准答案比较，计算 loss，再更新参数。

### 3.2 SFT 训练的本质

SFT 不是让模型“记住这一条答案”，而是让它学会：

- 输入长什么样
- 任务目标是什么
- 输出格式应该怎么组织
- 哪种表达是更符合预期的

### 3.3 SFT 最适合哪些任务

- 问答
- 信息抽取
- 分类
- 摘要
- 改写
- 翻译
- 工具调用格式学习
- 结构化 JSON 输出
- 企业知识库风格回答

### 3.4 SFT 不擅长单独解决什么

SFT 可以提升很多能力，但它不是万能的。单靠 SFT 往往很难彻底解决：

- 偏好对齐
- 安全边界长期稳定性
- 多候选答案之间的好坏排序
- 数学/代码任务里复杂的搜索式推理能力

这些更适合在 SFT 之后再叠加偏好优化或 RL 类训练。

---

## 4. SFT 的训练流程

```text
准备样本
-> 统一模板
-> tokenize
-> 截断/拼接
-> 送入模型
-> 计算目标 token 的 loss
-> 反向传播
-> 更新参数
-> 定期评测与保存 checkpoint
```

### 4.1 模板化为什么很重要

同一条对话数据，模板不同，模型看到的真实 token 序列就不同。

例如同样的问答，有的模型模板是：

```text
<|system|>你是助手
<|user|>什么是 LoRA
<|assistant|>...
```

有的模板是：

```text
[INST] 什么是 LoRA [/INST]
...
```

如果模板不匹配，会直接导致：

- 训练效果变差
- 特殊 token 使用混乱
- 推理时行为异常

### 4.2 label mask 是什么

在很多 SFT 场景中，并不是所有 token 都参与 loss 计算。

典型做法是：

- 用户输入部分只作为上下文
- 助手输出部分才作为预测目标

这样做的目的是让模型重点学习“回答应该怎么写”，而不是去拟合用户输入本身。

---

## 5. 全量微调是什么

全量微调（Full Fine-Tuning）指：

> 训练时更新模型的全部参数。

### 5.1 优点

- 可塑性最强
- 理论上效果上限更高
- 能更彻底改变模型行为

### 5.2 缺点

- 显存需求极高
- 训练成本高
- checkpoint 很大
- 更容易发生灾难性遗忘
- 多任务切换不灵活

### 5.3 什么叫灾难性遗忘

灾难性遗忘是指：

> 模型为了适应新任务，过度修改原有参数，导致原来学会的通用能力被破坏。

常见表现：

- 语言能力退化
- 拒答策略失衡
- 某些通用问答能力明显变差
- 只会围绕训练数据的风格回答

### 5.4 全量微调什么时候值得用

- 资源非常充足
- 数据量足够大且质量很高
- 任务与原模型差异很大
- 你追求极致效果并接受更高成本

---

## 6. 为什么 LoRA 会成为主流

LoRA（Low-Rank Adaptation）是参数高效微调里最常见的方法。

它的核心思想是：

> 冻结原模型参数，只在部分线性层旁边新增一小组可训练的低秩参数，用很小的训练开销实现任务适配。

### 6.1 直观理解

你可以把全量微调理解为“把整个大脑重新训练一遍”，而 LoRA 更像是：

- 大脑主体不动
- 在少数关键回路上外挂一个小模块
- 只训练这个小模块

### 6.2 为什么它好用

- 显存占用低
- 训练速度快
- 容易切换不同任务 adapter
- 存储成本小
- 适合企业快速实验

### 6.3 LoRA 的数学直觉

原始权重更新如果写成：

```text
W' = W + ΔW
```

LoRA 假设这个增量矩阵 `ΔW` 可以用低秩分解近似：

```text
ΔW = B × A
```

其中：

- `A` 和 `B` 是两个更小的矩阵
- `rank` 决定中间维度大小
- rank 越大，表达能力越强，但显存和参数量也会上去

### 6.4 LoRA 不是“白送效果”

LoRA 虽然很实用，但它不是没有代价：

- 某些复杂任务上限可能低于全量微调
- 对 target modules、rank、alpha 比较敏感
- 数据质量差时，LoRA 同样救不了你

---

## 7. LoRA 的关键超参数

### 7.1 `lora_rank`

最重要的 LoRA 参数之一。

- 小 rank：参数更少，更省资源，但表达能力有限
- 大 rank：效果可能更好，但显存和训练不稳定性增加

经验上常见起点：

- `8`、`16`：轻量实验
- `32`、`64`：业务常用
- `128`：追求更强表达时尝试

### 7.2 `lora_alpha`

它控制 LoRA 更新的缩放强度。

常见经验值：

```text
lora_alpha = 2 * lora_rank
```

但这不是死规则，关键还是要结合任务和学习率看效果。

### 7.3 `lora_dropout`

用于减少过拟合。

常见设置：

- `0`
- `0.05`
- `0.1`

如果数据很小、重复高，适当 dropout 往往更稳。

### 7.4 `target_modules`

指定 LoRA 加在哪些层。

常见目标层：

- `q_proj`
- `k_proj`
- `v_proj`
- `o_proj`
- `gate_proj`
- `up_proj`
- `down_proj`

不同模型命名不同，但本质都是注意力和 MLP 里的关键线性层。

---

## 8. QLoRA 是什么

QLoRA 可以理解为：

> 在 LoRA 的基础上，把基座模型做低比特量化，从而进一步降低显存占用。

### 8.1 它解决什么问题

LoRA 已经省显存了，但有时仍然不够，尤其在单卡或消费级 GPU 场景下。

QLoRA 的思路是：

- 基座模型用 4-bit 或类似低比特方式加载
- 可训练部分仍然是 LoRA adapter
- 用更低显存跑更大模型

### 8.2 代价是什么

- 训练更复杂
- 速度不一定更快
- 精度可能略有损失
- 某些场景稳定性不如常规 LoRA

### 8.3 什么时候优先选 QLoRA

- 显存明显不够
- 你只能用单卡或少量显卡
- 任务以快速验证为主

### 8.4 什么时候不要一上来就 QLoRA

如果你资源够，其实通常建议先普通 LoRA：

- 配置更简单
- 稳定性往往更好
- 结果更容易解释

---

## 9. DoRA、AdaLoRA 是什么

### 9.1 DoRA

DoRA 是对 LoRA 的一种改进，核心是把权重更新分成“方向”和“幅度”去建模，希望得到更好的参数表达能力。

适合：

- 想在 LoRA 框架下争取更好效果
- 愿意接受更复杂配置

### 9.2 AdaLoRA

AdaLoRA 强调“rank 自适应分配”。

它不再给所有层固定相同 rank，而是让模型把有限预算分配给更重要的层。

适合：

- 想在有限参数预算下优化效果
- 更偏研究和实验场景

### 9.3 实际建议

如果你是业务团队：

- 第一选择通常是 `LoRA`
- 显存更紧张时用 `QLoRA`
- 有明确实验需求再考虑 `DoRA` 或 `AdaLoRA`

## 10. 后训练的数据设计细节

后训练成败，很多时候不是输在算法，而是输在数据。

### 10.1 一个好 SFT 数据集应具备什么特征

- 目标明确
- 风格统一
- 标签稳定
- 难度有梯度
- 覆盖真实业务场景
- 包含边界样本

### 10.2 常见错误数据

- 问题含糊，答案也含糊
- 同样问题给出风格冲突的答案
- 训练样本大量重复
- 把不该输出的思维过程强行作为标准答案
- 用很脏的爬虫数据直接做 instruction

### 10.3 为什么“格式一致性”很重要

如果你希望模型稳定输出 JSON，但训练数据里一会儿是 JSON，一会儿是自然语言解释，一会儿是 Markdown 表格，那么模型会学到混乱的输出习惯。

这也是很多人说“模型不听话”的根本原因之一。

### 10.4 多任务混训时最容易犯的错

多个任务一起训练时，最常见的问题不是“不能训”，而是：

- 任务配比失衡
- 一类任务把另一类任务冲掉
- 任务模板不统一
- 样本长度差异过大

因此多任务训练一定要关注：

- 配比
- 模板
- 长度分布
- 验证集分任务拆开看

## 11. 后训练常见超参数与调参逻辑

### 11.1 学习率 `learning_rate`

这是最重要的超参数之一。

常见现象：

- 太大：loss 抖动、发散、NaN、灾难性遗忘
- 太小：训练很慢、loss 几乎不动、效果提升不明显

通常经验：

- LoRA 可以比全量微调用更大的学习率
- 对齐阶段通常比 SFT 用更小学习率

### 11.2 `batch_size` 与 `gradient_accumulation_steps`

很多人只盯 `batch_size`，其实真正关键的是“有效 batch size”。

```text
有效 batch size = per_device_batch_size × gradient_accumulation × GPU 数
```

影响：

- 太小：梯度噪声大，训练不稳
- 太大：显存吃紧，泛化未必更好

### 11.3 `max_length`

它决定每条样本最多保留多长 token 序列。

影响非常大：

- 太小：长样本被截断，关键信息丢失
- 太大：显存大增，吞吐下降

很多任务训练效果不好，其实不是模型不行，而是你把真正重要的信息截掉了。

### 11.4 `num_train_epochs`

它决定数据集被完整看多少遍。

典型风险：

- 太少：学不进去
- 太多：过拟合、复读、风格僵化

### 11.5 `warmup_ratio`

它用于训练初期平滑学习率，降低刚开始训练时的震荡风险。

对中大型训练任务尤其重要。

## 12. 后训练里常见的错误理解

### 12.1 “loss 降了就一定效果变好了”

不对。

loss 降只是说明模型更会拟合训练目标，但不代表业务目标一定提升。

可能出现：

- 训练集上变好，测试集没变
- 格式正确了，但事实性变差
- 某一类任务提升了，另一类退化了

### 12.2 “数据越多越好”

不对。

低质量、重复、风格冲突的数据越多，副作用越大。

### 12.3 “LoRA 一定不如全量微调”

也不对。

在很多真实业务场景里：

- 数据量没那么大
- 任务差异没那么极端
- 资源又有限

这时候 LoRA 往往是性价比最高的选择。

### 12.4 “换个最新方法就能显著提升”

很多时候并不能。

如果你的数据没清洗、目标没定义、评测没建好，那么方法换得再花哨，提升也很有限。

## 13. 一个实用的后训练选型方法

### 13.1 如果你是第一次做业务微调

建议：

1. 先选一个成熟的开源 Instruct 模型
2. 先做高质量小规模 SFT
3. 优先用 LoRA
4. 建一个可靠的验证集和测试集
5. 先看业务效果，再谈复杂方法

### 13.2 如果模型不懂你的领域

优先考虑：

- 先补领域数据
- 必要时做 `CPT`
- 再做 SFT

### 13.3 如果模型会回答，但回答“质量不够好”

优先考虑：

- 补高质量 SFT 数据
- 做偏好数据
- 再考虑 DPO / PPO / GRPO

## 14. 一套推荐的新手后训练流程

```text
第 1 步：定义目标
  我要提升什么任务

第 2 步：准备小规模高质量样本
  不要急着堆数量

第 3 步：用 LoRA 做第一轮 SFT
  先跑通、先拿基线

第 4 步：做评测
  看格式、事实性、稳定性、业务命中率

第 5 步：分析失败样本
  确定是数据问题、方法问题还是超参数问题

第 6 步：逐步扩数据、改配比、调参数

第 7 步：需要时再进入偏好对齐阶段
```

## 15. 本册总结

### 15.1 你必须记住的结论

- 后训练是业务大模型落地的主战场
- SFT 是所有后训练工作的基础
- LoRA 是最常见、最实用的工程方案
- 数据设计、模板匹配、评测闭环，重要性往往高于“追新方法”
- 全量微调不是默认最优解，而是高成本选项

### 15.2 你真正该具备的能力

学完这册后，你应该不仅知道名词，还应该能做判断：

- 什么时候该用 SFT
- 什么时候用 LoRA 而不是全量
- 为什么某批数据会把模型训坏
- 为什么训练日志看起来正常，但结果却不对
