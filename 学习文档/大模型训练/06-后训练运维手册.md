# 第六册：后训练运维手册

> 这一册面向训练平台、MLOps、AI Infra、模型工程和运维同学。
>
> 它的目标不是讲一个“理想训练流程”，而是给出一套能真正用于生产环境的后训练运维手册。
>
> 你可以把它理解为：
>
> - 训练任务怎么稳
> - 训练产物怎么管
> - 故障来了怎么顶
> - 版本怎么发
> - 整个团队怎么形成规范

---

## 1. 后训练运维到底在运维什么

后训练运维不是只盯 GPU 是否在线，而是在维护一个完整训练生产系统：

```text
数据 -> 模型 -> 训练代码 -> 镜像 -> 调度 -> 存储 -> 监控 -> checkpoint
   -> 评测 -> 发布 -> 回滚 -> 审计
```

### 1.1 运维对象清单

至少包括：

- 训练数据与元数据
- 基座模型与 adapter
- 训练镜像与依赖版本
- GPU / 网络 / 存储资源
- 训练作业
- checkpoint 和日志
- 评测结果
- 上线版本和回滚版本

### 1.2 运维目标

一个成熟的后训练系统，至少要做到：

- 可复现
- 可监控
- 可告警
- 可恢复
- 可追责
- 可回滚

---

## 2. 训练任务生命周期

一个训练任务在运维视角下通常会经历：

1. 需求提出
2. 数据冻结
3. 配置冻结
4. 训练前检查
5. 提交任务
6. 训练中监控
7. 训练完成验收
8. 评测与准入
9. 上线或归档
10. 版本留档与复盘

建议每一步都有对应记录，否则一旦出问题就容易进入“谁也说不清”的状态。

---

## 3. 训练前检查清单

训练前检查是最容易被忽视，但最省成本的一步。

### 3.1 数据检查

提交任务前至少确认：

- 数据路径存在
- 数据权限正确
- schema 校验通过
- 样本量符合预期
- 去重完成
- 训练集 / 验证集 / 测试集已划分
- 敏感字段已脱敏
- 数据版本已冻结

### 3.2 模型检查

- 基座模型路径正确
- tokenizer 与模型匹配
- 模型版本有唯一标识
- adapter 与基座模型兼容
- 量化版本与训练方式兼容

### 3.3 配置检查

- 方法类型明确：SFT / DPO / GRPO 等
- learning rate 是否在安全范围
- batch size 是否与显存匹配
- max length 是否与任务匹配
- save_steps 是否合理
- eval 策略是否开启

### 3.4 环境检查

- 镜像版本固定
- 框架版本固定
- CUDA / 驱动 / NCCL 版本兼容
- 依赖包版本固定
- 启动命令与参数模板验证通过

### 3.5 资源检查

- GPU 数量足够
- 显存估算通过
- CPU 和内存足够
- 存储空间足够
- 网络带宽足够
- checkpoint 输出路径可写

---

## 4. 资源规划方法

### 4.1 先估显存，再申请资源

一个很常见的错误是先提交任务，再看会不会 OOM。

更正确的做法是：

1. 明确模型大小
2. 明确训练方式
3. 明确精度
4. 明确序列长度
5. 明确 batch size
6. 根据经验或基线估算显存

### 4.2 影响显存的主要因素

- 模型参数量
- 是否全量微调
- 是否 LoRA / QLoRA
- optimizer 状态
- 激活值
- sequence length
- batch size
- 并行策略

### 4.3 影响训练吞吐的主要因素

- dataloader 吞吐
- 磁盘读写
- 网络存储性能
- token 长度分布
- 多机通信效率
- gradient checkpointing 开销

### 4.4 容量规划建议

对团队来说，最好维护一份“标准任务画像”：

- 7B LoRA
- 14B LoRA
- 7B 全量
- 32B QLoRA
- DPO 任务
- GRPO 任务

每类任务给出：

- 推荐 GPU 数
- 推荐 batch 范围
- 常见运行时长
- 常见故障点

这样新任务就不是从零试错。

## 5. 镜像与依赖管理

### 5.1 为什么镜像必须版本化

训练任务最怕“环境漂移”。

如果今天和昨天用的是不同镜像，但没人记录，那么出现差异时几乎无法排查。

因此建议：

- 镜像采用不可变 tag
- 每次训练记录镜像 digest
- 不允许生产训练任务使用 `latest`

### 5.2 镜像中应该固定什么

- Python 版本
- PyTorch 版本
- CUDA 版本
- NCCL 版本
- Transformers 版本
- PEFT / TRL / DeepSpeed 版本
- 自研训练脚本版本

### 5.3 升级原则

框架升级不要直接覆盖全量任务，建议分阶段：

1. 新镜像构建
2. 小任务 smoke test
3. 单机验证
4. 多卡验证
5. 历史任务回归验证
6. 灰度切换

---

## 6. 训练任务命名与元数据规范

建议每个训练任务都有明确命名规则，例如：

```text
<业务线>-<模型>-<方法>-<数据版本>-<日期>-<序号>
```

例子：

```text
search-qwen2.5-7b-lora-v20260324-001
```

### 6.1 至少记录哪些元数据

- 任务 ID
- 提交人
- 业务目标
- 基座模型版本
- 数据版本
- 训练方法
- 关键超参数
- 镜像版本
- 代码版本
- 输出路径
- 评测结果

### 6.2 为什么这一步很重要

因为没有元数据，后面这些事情都会失控：

- 复现失败
- 故障定位失败
- 回滚失败
- 谁改坏了说不清

---

## 7. checkpoint 管理

checkpoint 是训练运维的核心资产。

### 7.1 必须解决的四个问题

1. 存在哪
2. 保存多频繁
3. 保留多少份
4. 怎么恢复

### 7.2 保存策略建议

根据任务类型选择：

- 短任务：每较少步数保存一次
- 长任务：定步数 + 定时双策略
- 高价值任务：保留关键里程碑 checkpoint

### 7.3 checkpoint 命名建议

建议带上：

- step
- epoch
- 时间
- 数据版本
- 训练方法

### 7.4 checkpoint 旁边必须存什么

不要只存权重，还要存：

- 训练配置
- tokenizer
- adapter 配置
- 训练日志摘要
- 评测结果
- 恢复说明

### 7.5 恢复演练

真正成熟的团队不是“理论可恢复”，而是“做过恢复演练”。

建议定期做：

- 从最新 checkpoint 恢复
- 从中间 checkpoint 恢复
- 单节点失败后恢复
- 存储路径切换恢复

---

## 8. 训练中监控什么

### 8.1 模型训练指标

必看：

- train loss
- eval loss
- learning rate
- grad norm
- tokens/s
- steps/s

### 8.2 资源指标

必看：

- GPU 利用率
- GPU 显存
- CPU 利用率
- 内存占用
- 磁盘 IO
- 网络吞吐

### 8.3 分布式指标

多机多卡时要看：

- 各 rank 是否齐
- 通信耗时
- barrier 等待时间
- 节点间网络抖动

### 8.4 存储与日志指标

- checkpoint 保存是否成功
- 日志是否持续输出
- 输出目录是否接近写满
- event 文件是否正常生成

---

## 9. 告警体系怎么建

### 9.1 建议的告警分级

可以按严重级别分：

- P1：任务中断、checkpoint 丢失、存储不可写、整批任务失败
- P2：loss 异常、训练严重降速、某节点掉线
- P3：吞吐波动、显存偏高、评测未按时完成

### 9.2 高优先级告警场景

- 任务状态异常退出
- 连续多次 OOM
- loss NaN
- 多机通信超时
- checkpoint 连续保存失败
- 输出目录空间不足

### 9.3 告警后的动作必须标准化

每种告警都应该有对应 runbook：

- 谁响应
- 多久内响应
- 是否自动重试
- 是否需要人工兜底
- 是否需要升级处理

---

## 10. 故障处理手册

### 10.1 OOM 故障

处理顺序建议：

1. 看最近变更的是模型、数据还是参数
2. 降 batch size
3. 降 max length
4. 打开或确认 gradient checkpointing
5. 切换 LoRA / QLoRA
6. 重新估算资源

### 10.2 训练发散故障

优先处理：

- 降学习率
- 检查异常样本
- 检查精度设置
- 检查梯度裁剪

### 10.3 多机通信故障

优先处理：

- 查是否某节点先异常退出
- 查网络连通性
- 查 NCCL 和驱动版本
- 缩回单机验证

### 10.4 存储故障

优先处理：

- 确认挂载是否正常
- 检查权限
- 检查剩余容量
- 检查 checkpoint 是否局部损坏

---

## 11. 训练任务恢复策略

### 11.1 自动恢复

适合：

- 短暂节点抖动
- 非致命中断
- 调度器重试可恢复的场景

### 11.2 手动恢复

适合：

- checkpoint 需要人工确认
- 配置需要修正
- 训练方法或数据版本需要切换

### 11.3 恢复前检查

恢复之前必须确认：

- checkpoint 完整
- 基座模型没变
- tokenizer 没变
- 配置差异可接受
- 输出路径不会覆盖错误结果

---

## 12. 评测与发布准入

训练完成并不等于可以上线。

### 12.1 最低准入条件

至少通过：

- 离线自动评测
- 人工抽检
- 关键风险场景测试
- 回归测试

### 12.2 上线前对比

建议与至少一个基线版本做对比：

- 当前线上版本
- 上一轮训练版本
- 原始基座或 SFT 基线

### 12.3 灰度发布建议

不要直接全量切换，建议：

1. 小流量灰度
2. 对比核心指标
3. 观察异常场景
4. 确认后再扩大范围

---

## 13. 回滚机制

### 13.1 为什么回滚必须预先设计

一旦模型上线出现：

- 幻觉暴增
- 拒答异常
- 结构化输出出错
- 用户投诉显著上升

你不能临时想回滚方案。

### 13.2 回滚至少需要什么

- 稳定可用的上一个版本
- 明确的版本映射关系
- 快速切换机制
- 回滚触发阈值

### 13.3 回滚触发信号

建议提前定义：

- 线上错误率超阈值
- 关键任务成功率跌破阈值
- 安全事件触发
- 幻觉率显著上升

---

## 14. 日常巡检清单

建议团队建立日巡检或周巡检。

### 14.1 资源巡检

- GPU 节点健康
- 驱动异常
- 显存碎片情况
- 存储容量
- 网络延迟

### 14.2 任务巡检

- 是否有长时间运行异常任务
- 是否有频繁失败的配置模板
- 是否有 checkpoint 堆积
- 是否有日志缺失

### 14.3 版本巡检

- 是否有过期镜像仍在被用
- 是否有废弃模型仍在生产训练
- 是否存在未归档实验任务

---

## 15. 安全与审计

后训练不仅是工程问题，也涉及数据和模型资产安全。

### 15.1 数据安全

- 敏感字段脱敏
- 权限分级
- 训练集访问审计
- 导出与下载审计

### 15.2 模型资产安全

- 基座模型权限控制
- 训练产物访问控制
- adapter 和完整权重分类管理
- 发布链路留痕

### 15.3 操作审计

至少记录：

- 谁提交任务
- 谁改了配置
- 谁发布了模型
- 谁执行了回滚

## 16. 团队协作规范

成熟团队通常会把训练和运维协作流程固定下来。

建议角色分工如下：

- 数据负责人：保证数据质量和版本冻结
- 模型工程师：负责方法与参数选择
- 平台 / 运维：负责资源、调度、监控、恢复
- 评测负责人：负责准入标准
- 发布负责人：负责灰度和回滚

### 16.1 每次训练最好都产出什么

- 任务说明
- 数据版本
- 配置文件
- 训练日志摘要
- 评测报告
- 结论和建议

这能极大降低团队沟通成本。

---

## 17. 一个推荐的生产化 Runbook

```text
需求确认
-> 数据冻结
-> 训练配置评审
-> 小样本 smoke test
-> 正式训练
-> 训练中监控和告警
-> 训练完成后自动评测
-> 人工抽检
-> 灰度发布
-> 线上观测
-> 正式发布或回滚
-> 复盘归档
```

这条链路的重点是：

> 训练任务必须像生产变更一样管理，而不是像个人实验一样随意。

## 18. 最终总结

### 18.1 对运维同学最重要的结论

- 后训练运维的核心不是“起服务”，而是“维护一条可复现的训练生产线”
- 镜像、数据、模型、配置都必须版本化
- checkpoint 不是附属品，而是恢复和回滚的核心资产
- 告警、恢复、回滚都必须标准化

### 18.2 对团队最重要的结论

- 训练不是单人行为，而是跨角色协作
- 训练成功不等于交付成功
- 能稳定复现、稳定回滚、稳定上线，才叫工程成熟
