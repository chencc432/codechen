# 第五册：后训练常见问题与系统排障

> 这一册不是讲“理想世界里的训练”，而是讲真实世界里训练任务为什么会失败、怎么定位、怎么修。
>
> 训练工程师最重要的能力之一，不是把任务提起来，而是出了问题能快速判断：
>
> - 是数据问题
> - 是参数问题
> - 是模型问题
> - 是代码问题
> - 还是环境和分布式系统问题

---

## 1. 先建立一套排障总思路

遇到训练问题时，不要一上来就乱改参数。

更好的顺序是：

```text
先确认问题现象
-> 再确认问题发生在训练哪一层
-> 再缩小范围
-> 最后做最小代价修复
```

建议把问题先分成五大类：

1. 数据层问题
2. 模型与方法层问题
3. 超参数层问题
4. 训练框架层问题
5. 系统与基础设施层问题

---

## 2. 数据层问题

### 2.1 数据格式错误

典型现象：

- 任务启动后立刻失败
- 日志里出现 JSON decode error
- 某些样本 tokenize 时报错
- role 字段缺失或非法

常见原因：

- JSONL 某一行格式损坏
- 字段名拼错
- 多轮消息顺序错误
- content 类型不一致

排查方法：

1. 先抽样读原始数据
2. 做 schema 校验
3. 统计异常样本比例
4. 单独跑数据预处理脚本

### 2.2 样本过长被截断

典型现象：

- 模型训练正常，但效果很怪
- 长任务表现极差
- 模型总是漏掉后半段信息

常见原因：

- `max_length` 太小
- 长样本占比过高
- 模板额外占用很多 token

排查方法：

- 统计 token 长度分布
- 看截断比例
- 单独检查失败样本的实际输入长度

### 2.3 样本重复导致过拟合

典型现象：

- loss 很快下降
- 模型开始复读
- 风格越来越模板化

解决思路：

- 去重
- 减少 epoch
- 增加数据多样性

### 2.4 标签冲突

典型现象：

- 模型行为不稳定
- 同类问题回答风格反复横跳
- 某些格式任务时而正确时而错误

根因通常是：

- 同样输入对应多个风格冲突的输出
- 标注团队标准不统一

---

## 3. 超参数层问题

### 3.1 学习率过大

典型现象：

- loss 震荡
- loss 爆炸
- 出现 NaN 或 Inf
- 模型性能反而退化

为什么会这样：

- 参数每次更新幅度太大
- 优化器在局部最优附近来回跳
- 数值稳定性变差

建议处理：

- 先把学习率降一个量级试
- 检查 warmup 是否过短
- 检查混合精度设置

### 3.2 学习率过小

典型现象：

- loss 很慢才动
- 跑了很久效果几乎不变

建议处理：

- 适当增大学习率
- 确认是否被 scheduler 很快衰减掉

### 3.3 batch 太大导致 OOM

典型现象：

- 刚跑起来就显存溢出
- 某些特定 step 才溢出

建议处理：

- 降 `per_device_batch_size`
- 提高 `gradient_accumulation`
- 降 `max_length`
- 切换 LoRA / QLoRA

### 3.4 epoch 太多导致过拟合

典型现象：

- 训练集变好
- 验证集不涨甚至下降
- 模型越来越僵硬

解决思路：

- 减少 epoch
- 增加验证频率
- 早停

## 4. 训练过程中的典型异常

### 4.1 OOM：显存不足

这是最常见的训练故障之一。

常见诱因：

- 模型太大
- `max_length` 太长
- batch 太大
- 开了额外缓存或评测
- 全量微调占用过高

优先排查顺序：

1. 模型规模
2. max length
3. batch size
4. gradient checkpointing 是否开启
5. 是否能切 LoRA / QLoRA

### 4.2 loss 不下降

典型原因：

- 数据格式不对
- label mask 错了
- 学习率不合适
- target modules 设错
- 数据和任务目标不匹配

### 4.3 loss 变成 NaN

典型原因：

- 学习率过大
- fp16 数值溢出
- 样本里有异常 token 或异常值
- 梯度爆炸

建议：

- 优先用 `bf16`
- 降学习率
- 检查异常样本
- 开梯度裁剪

### 4.4 训练吞吐突然下降

可能原因：

- 数据加载跟不上
- 磁盘 IO 抖动
- 网络存储带宽不足
- 某节点通信异常

排查方向：

- 看 GPU 利用率
- 看 dataloader worker
- 看磁盘和网络监控

## 5. 分布式训练常见问题

### 5.1 NCCL 报错

常见症状：

- 多卡任务卡住
- 训练一直不动
- 日志里出现 NCCL timeout 或 communicator error

常见根因：

- 节点间网络问题
- 驱动 / CUDA / NCCL 版本不兼容
- 某个 rank 提前退出
- 拓扑或环境变量配置错误

排查顺序：

1. 先确认是不是单卡可跑
2. 再确认单机多卡可跑
3. 最后确认多机多卡问题

### 5.2 某个 rank 卡死

常见原因：

- 某张卡 OOM
- 某个进程遇到异常但没正确退出
- 数据加载不一致导致 barrier 等待

处理思路：

- 看每个 rank 的日志
- 查是否某个进程先报错
- 看样本分片逻辑是否一致

### 5.3 checkpoint 恢复失败

常见原因：

- 框架版本变了
- LoRA adapter 与基座模型不匹配
- optimizer state 损坏
- 多机保存不完整

建议：

- 保存训练元信息
- 记录镜像版本、框架版本、模型版本
- 恢复前先做结构一致性校验

## 6. 模型效果类问题

### 6.1 模型开始复读

表现：

- 喜欢重复句子
- 回答尾部循环
- 模板化严重

常见原因：

- 过拟合
- 重复样本太多
- 输出模板过于单一

### 6.2 模型格式错乱

表现：

- 本来要求 JSON，却输出自然语言
- 少字段、乱字段、混 Markdown

常见原因：

- 训练数据格式不统一
- 提示词风格不一致
- 评测没盯格式正确率

### 6.3 模型幻觉变多

表现：

- 答得更流畅，但错得更多

常见原因：

- 数据里错误知识被放大
- 对齐阶段更强调“像样”而非“正确”
- 缺乏事实性评测

### 6.4 模型拒答变多

常见原因：

- 安全数据占比过高
- 对齐训练过于保守
- chosen / rejected 标准偏向保守策略

## 7. 如何判断问题属于哪一层

### 7.1 一个非常实用的判断法

如果问题是“任务启动就挂”，大概率先看：

- 数据格式
- 环境依赖
- 路径
- 框架参数

如果问题是“能跑但 loss 异常”，大概率先看：

- 学习率
- 数据质量
- label mask
- 数值精度

如果问题是“训练看起来正常，但效果不好”，大概率先看：

- 数据目标是否对齐
- 评测是否有效
- 模板和格式是否统一
- 是否过拟合或遗忘

## 8. 一个标准排障 Runbook

下面是一套适合团队共用的排障流程。

### 第一步：记录现象

至少记录：

- 什么任务
- 哪个模型
- 哪个方法
- 哪个 checkpoint
- 从第几步开始异常
- 异常日志原文

### 第二步：先复现

先确认问题是否稳定复现。

如果不能稳定复现，优先怀疑：

- 环境波动
- 数据分片问题
- 分布式时序问题

### 第三步：缩小规模

把问题缩小到：

- 单卡
- 小数据集
- 固定随机种子
- 关闭复杂优化项

如果缩小后问题消失，通常说明根因在：

- 分布式
- 数据规模
- 长样本
- 系统资源

### 第四步：逐项回加

把以下因素一个个加回来：

- 长样本
- 多卡
- 混合精度
- checkpoint 恢复
- 评测

谁加回来后出问题，谁就是重点怀疑对象。

## 9. 一组高频故障速查表

| 现象 | 最先怀疑什么 | 第一处理动作 |
|------|--------------|--------------|
| 启动即失败 | 数据路径/参数/依赖 | 看首个异常堆栈 |
| CUDA OOM | batch/max_length/模型大小 | 降 batch 或长度 |
| loss NaN | lr 过大/精度问题 | 降 lr，改 bf16 |
| 多卡卡住 | NCCL/某个 rank 异常 | 查各 rank 日志 |
| 训练成功但效果差 | 数据与评测问题 | 抽失败样本分析 |
| 模型复读 | 过拟合/重复样本 | 去重、减 epoch |
| 格式不稳定 | 数据模板不统一 | 统一输出格式数据 |
| 拒答过多 | 对齐过强 | 回看安全数据配比 |

## 10. 团队层面的预防性措施

排障最好的方式，其实不是事后救火，而是事前预防。

建议团队建立：

- 数据 schema 校验
- 训练前 smoke test
- 标准化训练配置模板
- checkpoint 命名规范
- 固定评测集
- 故障分级和复盘制度

特别重要的一点是：

> 每一次训练失败都应该留下可复用的经验，而不是只靠某个人临时记忆。

---

## 11. 本册总结

- 训练问题要按层分解，不能乱猜
- OOM、NaN、NCCL、过拟合、格式漂移是最常见的五大类问题
- 训练成功不代表模型可用，效果问题很多时候是数据与评测问题
- 最有效的排障方法是：缩小规模、稳定复现、逐项回加
