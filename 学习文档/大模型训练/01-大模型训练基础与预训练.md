# 第一册：大模型训练基础与预训练

> 这一册解决三个问题：
>
> 1. 大模型训练到底在干什么
> 2. 预训练和后训练的边界在哪里
> 3. 为什么企业团队大多数时候不自己做预训练

---

## 1. 大模型训练到底是什么

一句最朴素的话：

> 训练就是让模型不断犯错、不断改错，直到它在大量样本上形成稳定的参数分布。

如果把模型当成一个巨大的函数，那么训练过程就是不断调整这个函数里的参数，让它在面对输入时输出更接近目标答案。

```text
输入文本 -> 分词(Tokenize) -> 模型前向计算 -> 产生预测
       -> 和标准答案比较 -> 得到 loss -> 反向传播 -> 更新参数
       -> 重复很多很多次
```

### 1.1 训练里最常见的基本术语

| 术语 | 含义 | 通俗理解 |
|------|------|----------|
| 参数 Parameters | 模型内部的数值 | 模型脑子里的“连接强度” |
| 权重 Weights | 参数的一部分常见叫法 | 本质上和参数一起理解即可 |
| Token | 文本被切分后的基本单元 | 模型真正处理的不是字，而是 token |
| Loss | 预测和目标的差距 | 错多少 |
| Gradient | loss 对参数的变化方向 | 应该往哪改 |
| Step | 一次参数更新 | 做完一小批题并改一次 |
| Epoch | 整个数据集完整看一遍 | 把整本教材过一遍 |
| Batch Size | 每一步处理多少条样本 | 一次做几题 |
| Learning Rate | 每次改参数的幅度 | 改得激进还是保守 |
| Checkpoint | 训练中间保存的快照 | 游戏存档 |

### 1.2 一个你必须有的直觉

训练不是“记住答案”，而是“形成统计规律”。

比如你给模型看大量翻译样本，它不是把每一句话都背下来，而是在参数里形成了“这种模式通常应该这样输出”的倾向。

所以：

- 数据分布决定模型学到什么
- 训练目标决定模型朝哪个方向学
- 超参数决定它学得快不快、稳不稳

## 2. 大模型训练的两大阶段

```text
预训练 Pre-Training -> 后训练 Post-Training -> 部署与迭代
```

### 2.1 预训练做什么

预训练的目标是：

- 学会语言规律
- 学会世界知识的统计分布
- 学会代码、问答、段落结构、常见推理模式
- 形成一个通用的“基座能力”

预训练后的产物通常叫做：

- `Base Model`
- 基座模型
- 原始模型

它通常还不够“会聊天”，也不够“听指令”，只是很会做下一 token 预测。

### 2.2 后训练做什么

后训练的目标是：

- 让模型更会遵循指令
- 让模型掌握特定业务风格和任务
- 让模型回答更符合偏好、安全和格式要求
- 让模型在特定领域表现更稳定

后训练后的产物通常叫：

- `Instruct Model`
- `Chat Model`
- 领域模型
- 对齐后的模型

### 2.3 一句话区分

- 预训练：让模型“会语言、懂世界”
- 后训练：让模型“按你的要求做事”

---

## 3. 预训练的本质

### 3.1 预训练最核心的目标函数

大部分语言模型预训练，本质上都是做“下一个 token 预测”。

比如一句话：

```text
今天 北京 的 天气 很 _
```

模型需要预测下一个 token 可能是：

- 好
- 冷
- 热
- 不错

它看的是概率分布，而不是“理解式”地做人类推理。

但正因为这种目标在海量文本上反复进行，模型最后会获得非常强的语言能力和知识压缩能力。

### 3.2 为什么下一 token 预测这么强

因为语言里蕴含了几乎所有人类知识的结构：

- 语法
- 事实
- 风格
- 逻辑关系
- 代码模式
- 问答套路

当模型能在足够多的文本里反复预测下一个 token，它就会逐步学会：

- 什么词常一起出现
- 什么句式常出现在什么语境
- 什么问题后面通常跟什么答案
- 什么代码上下文接什么实现

这也是为什么“只是预测下一个 token”最后能涌现出复杂能力。

## 4. 预训练的数据是什么样

### 4.1 数据来源

典型预训练语料包括：

- 网页文本
- 书籍
- 论文
- 新闻
- 论坛
- 代码仓库
- 问答数据
- 合成数据

### 4.2 真正难的不是“找数据”，而是“做数据工程”

预训练并不是把互联网所有文本一股脑喂进去就行。真正困难的是：

- 去重
- 去脏
- 去广告
- 去低质量翻译文
- 识别垃圾内容和采样农场内容
- 过滤违法违规内容
- 多语言配比
- 代码和自然语言配比
- 长短文本配比
- 不同领域样本的温度采样

### 4.3 为什么要去重

如果重复样本太多，会带来几个问题：

- 模型记忆过强，泛化变差
- 某些语料分布被异常放大
- 训练 token 被浪费
- 数据泄漏风险增大

### 4.4 为什么要做数据配比

不同来源数据质量差异极大。

例如：

- 高质量百科可能少但价值大
- 网页语料多但噪声高
- 代码数据稀缺但对代码能力关键

因此预训练数据不是“谁多谁权重大”，而是“谁更值得被看到，就给更合理的采样概率”。

## 5. 预训练的工程成本为什么极高

### 5.1 算力成本

从零训练一个主流大模型通常需要：

- 数百到数千张高端 GPU
- 数周到数月训练周期
- 高速网络互联
- 持久化存储和 checkpoint 管理体系

### 5.2 工程复杂度

预训练时你要面对的不只是一个 PyTorch 脚本，而是一整套超大型分布式系统：

- 多机多卡调度
- 数据并行、张量并行、流水并行
- 通信优化
- checkpoint 切分与恢复
- 训练稳定性控制
- 容错与重启
- 数据管道吞吐

### 5.3 预训练常见技术栈

- `PyTorch`
- `CUDA`
- `NCCL`
- `Megatron-LM`
- `DeepSpeed`
- `FSDP`
- `Flash Attention`
- 自研数据流水线

## 6. 为什么业务团队通常不做预训练

因为预训练对大多数企业来说，既不划算，也不必要。

### 6.1 不划算

原因很现实：

- 成本极高
- 周期极长
- 风险极大
- 复现门槛高

而且即便你砸了很多钱，也未必训练出一个比当前开源模型更强的基座模型。

### 6.2 不必要

因为现成的开源基座模型已经非常强，很多业务问题并不是“模型完全不会”，而是：

- 不够懂你的领域
- 不会按你的格式输出
- 安全边界不符合你的要求
- 对你关心的任务没有被充分优化

这些问题主要靠后训练解决，而不是重新预训练。

### 6.3 什么时候才需要考虑继续预训练

只有在下面这类场景，才会认真考虑做 `CPT` 或更大规模的领域预训练：

- 领域语言极强，例如法律、医疗、金融、工业控制
- 有海量未标注文档，但高质量指令数据少
- 基础模型在该领域几乎完全不懂行话和知识结构

---

## 7. 从预训练到后训练，中间发生了什么

一个更贴近现实的链路如下：

```text
海量通用语料
  -> 预训练得到 Base 模型
  -> 指令微调(SFT)让它学会听话
  -> 偏好对齐(DPO/PPO/GRPO等)让它更像“好助手”
  -> 评测和部署
```

这意味着：

- 预训练解决“有没有底子”
- 后训练解决“能不能胜任工作”

## 8. 大模型训练的三个关键瓶颈

### 8.1 不是只有算力瓶颈

很多人一提训练，只想到 GPU 不够。实际上真正常见的瓶颈有三个：

1. 数据瓶颈
2. 工程瓶颈
3. 评测瓶颈

### 8.2 数据瓶颈

常见表现：

- 样本很多，但有用信息少
- 样本格式不一致
- 重复高
- 标签不稳
- 和目标任务不对齐

### 8.3 工程瓶颈

常见表现：

- 训练经常中断
- 显存估算不准
- checkpoint 恢复失败
- 多机通信异常
- 数据加载速度跟不上 GPU

### 8.4 评测瓶颈

常见表现：

- loss 降得很好，但效果不涨
- 离线分数高，线上体验差
- 某些边缘场景退化没人发现

---

## 9. 你必须建立的训练闭环观

真正成熟的大模型训练不是“训练一次就完”，而是一个闭环：

```text
定义目标
-> 准备数据
-> 选择方法
-> 跑训练
-> 做评测
-> 分析失败样本
-> 回到数据或方法继续迭代
```

这里最关键的一点是：

> 训练不是目的，稳定提升目标任务表现才是目的。

如果没有评测闭环，再复杂的训练方法都可能只是“看起来很努力”。

---

## 10. 本册你应该记住的结论

### 10.1 核心结论

- 预训练是在海量通用数据上做基础能力构建
- 后训练是在已有模型基础上做任务适配和行为对齐
- 业务团队最常做的是后训练，不是从零预训练
- 预训练的最大难点不是某个单点算法，而是海量数据和超大规模工程系统

### 10.2 学习上的结论

如果你是新手，不要一上来就沉迷研究超大规模预训练细节。你更应该先学会：

- 看懂训练链路
- 看懂后训练任务
- 学会设计数据和评测
- 学会跑通稳定的训练流程
