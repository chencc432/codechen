# 第三册：偏好对齐：DPO、PPO、GRPO 等方法详解

> 当一个模型通过 SFT 已经“能回答问题”之后，下一个问题往往是：
>
> 为什么它回答得还不够好？
>
> 所谓“不够好”，通常不是指完全不会，而是指：
>
> - 不够稳
> - 不够像人类偏好的高质量回答
> - 不够安全
> - 不够有帮助
> - 不够会推理或不够会选择更优答案

这就是偏好对齐阶段要解决的问题。

---

## 1. 为什么 SFT 之后还需要对齐

SFT 教模型“应该怎么答”，但很多任务根本没有唯一标准答案。

例如问题：

```text
请给我一份三天杭州旅游建议。
```

这类问题可能有多个合理答案：

- 简洁版
- 详细版
- 面向亲子游
- 面向预算有限用户

这时真正重要的不是“唯一正确答案”，而是“哪个答案更好、更有帮助、更符合偏好”。

### 1.1 SFT 的局限

SFT 通常依赖单个参考答案，因此它更擅长：

- 学会任务格式
- 学会某种风格
- 学会指令映射

但它不天然擅长：

- 排序多个候选答案的优劣
- 对复杂行为偏好进行精细对齐
- 在多个可行回答里稳定倾向更优那个

### 1.2 对齐阶段在优化什么

对齐阶段主要在优化：

- helpfulness：是否真正有帮助
- harmlessness：是否安全、无害
- honesty：是否更真实、不过度幻觉
- style：是否符合期望风格
- preference：是否更符合人类或规则偏好

---

## 2. 偏好对齐的核心对象

### 2.1 偏好数据

偏好数据不是传统意义上的“标准答案”，而是关于“哪个更好”的信息。

典型形式有三种：

1. `chosen / rejected` 成对偏好
2. 单个回答的好坏标签
3. 按规则自动打分的奖励信号

### 2.2 奖励模型

奖励模型（Reward Model, RM）本质上是一个“打分器”。

它输入：

- 问题
- 回答

输出：

- 一个分数，代表回答有多好

它的作用是把“人类偏好”变成一个可优化的训练信号。

### 2.3 参考模型

参考模型（Reference Model）是为了防止训练时模型偏离原始行为过远。

简单理解：

- policy model：当前正在被训练的模型
- reference model：一个相对稳定的参照物

很多对齐算法会通过比较当前模型和参考模型的差异，防止模型学偏。

---

## 3. 对齐方法全景图

```text
偏好对齐
  ├─ PPO：经典 RLHF
  ├─ DPO：直接偏好优化
  ├─ GRPO：组相对策略优化
  ├─ KTO：基于好坏标签的优化
  ├─ ORPO：把 SFT 和偏好优化合并
  └─ SimPO：不依赖参考模型的简化偏好优化
```

---

## 4. PPO：最经典，也最复杂

PPO（Proximal Policy Optimization）是 RLHF 里最经典的一条路线。

### 4.1 PPO 的典型三阶段流程

```text
阶段 1：SFT
  先得到一个能正常回答问题的模型

阶段 2：奖励模型训练
  用偏好数据训练一个 RM，负责打分

阶段 3：PPO 优化
  让模型输出拿到更高奖励，同时不要偏离参考模型太多
```

### 4.2 PPO 的优势

- 理论上表达能力强
- 能直接结合奖励信号优化策略
- 适合做更完整的 RLHF 流程

### 4.3 PPO 的难点

- 链路长
- 组件多
- 显存开销大
- 超参数敏感
- 训练不稳定

现实里它通常需要维护多个对象：

- policy model
- reference model
- reward model
- 有时还包括 value / critic 相关组件

### 4.4 PPO 常见问题

- KL 约束不稳，模型飘得太远
- 奖励黑客（reward hacking）
- 奖励模型本身学偏
- 回答看起来更“讨好打分器”，但不一定更真实

### 4.5 什么时候用 PPO

适合：

- 算法研究
- 高投入对齐项目
- 需要复杂奖励设计的任务

不太适合：

- 团队刚起步
- 资源有限
- 评测体系还不成熟

---

## 5. DPO：工程上最流行的偏好优化方法之一

DPO（Direct Preference Optimization）之所以流行，是因为它把很多 RLHF 的复杂度折叠掉了。

### 5.1 DPO 的核心思想

它不显式训练奖励模型，而是直接利用偏好对：

- 同一个问题
- 一个更好的回答 `chosen`
- 一个更差的回答 `rejected`

训练目标是：

> 让模型更倾向生成 `chosen`，更不倾向生成 `rejected`。

### 5.2 为什么 DPO 受欢迎

- 不需要单独训练奖励模型
- 比 PPO 简单很多
- 训练稳定性通常更好
- 工程链路更短

### 5.3 DPO 的数据长什么样

```json
{
  "messages": [
    {"role": "user", "content": "如何提升睡眠质量？"}
  ],
  "chosen": {
    "role": "assistant",
    "content": "可以从规律作息、减少咖啡因摄入、控制晚间屏幕时间、改善卧室环境几方面入手。"
  },
  "rejected": {
    "role": "assistant",
    "content": "早点睡就行。"
  }
}
```

### 5.4 DPO 的关键前提

偏好对必须可信。

如果你的 chosen 和 rejected 本身标得很乱，那么 DPO 会把模型往错误方向推。

### 5.5 DPO 的常见问题

- 偏好对质量不高
- chosen 和 rejected 差异太明显，模型学不到细粒度偏好
- beta 设置不合理，导致优化过强或过弱
- 对齐后风格提升了，但事实性可能退化

### 5.6 什么时候优先选 DPO

- 已经有一版不错的 SFT 模型
- 拿得到高质量偏好对
- 想提升回答质量和偏好一致性
- 不想上来就做复杂 PPO

---

## 6. GRPO：适合“有明确可评分规则”的任务

GRPO（Group Relative Policy Optimization）近年很受关注，尤其在数学、代码、可验证推理任务里。

### 6.1 它的核心直觉

对同一个问题，模型生成一组回答，然后根据规则或程序判分，做组内相对比较。

换句话说：

- 它不一定依赖人工偏好对
- 它更依赖“可验证奖励”

### 6.2 最适合的任务类型

- 数学题
- 代码题
- 有标准执行结果的任务
- 可程序验证正确性的结构化任务

### 6.3 为什么它在推理任务中有优势

因为这类任务最重要的不是“像人类觉得好”，而是“答案真的对不对”。

例如代码任务里，可以直接：

- 执行单元测试
- 统计通过率
- 将通过率作为奖励

这种奖励比纯人工偏好更客观。

### 6.4 GRPO 的难点

- 采样开销大
- 奖励函数设计很关键
- 如果规则不完整，会把模型引向投机策略

### 6.5 常见风险

- 模型学会“投机拿分”
- 奖励过于单一，导致风格和可读性下降
- 只追求最终答案正确，过程和安全性退化

---

## 7. KTO、ORPO、SimPO 分别在解决什么

### 7.1 KTO

KTO 不要求成对偏好，而是使用单条回答的“好 / 坏”标签。

优点：

- 标注成本比 DPO 更低
- 更容易从已有审核数据迁移

缺点：

- 信息密度通常不如成对偏好高
- 对标签质量依赖强

### 7.2 ORPO

ORPO 的思路是把监督学习和偏好优化结合得更紧，让链路更短。

优点：

- 训练流程更简洁
- 可能减少两阶段训练的复杂性

缺点：

- 调参和收益解释不一定像 DPO 那么直观

### 7.3 SimPO

SimPO 试图进一步简化偏好优化，减少对参考模型的依赖。

优点：

- 显存压力更低
- 流程更轻

缺点：

- 在不同任务和模型上收益稳定性可能不一

---

## 8. 这些方法到底怎么选

### 8.1 一个最实用的结论

如果你是业务团队，选型通常不是“谁最前沿”，而是“谁最匹配你的数据和评测条件”。

### 8.2 选型表

| 方法 | 需要什么数据 | 工程复杂度 | 适合什么 |
|------|--------------|------------|----------|
| PPO | 偏好数据 + 奖励模型 | 最高 | 高投入 RLHF 项目 |
| DPO | chosen/rejected 偏好对 | 中等 | 通用高质量偏好优化 |
| GRPO | 可程序评分的奖励 | 中等 | 数学、代码、可验证推理 |
| KTO | 单条好坏标签 | 中等偏低 | 已有审核标签数据 |
| ORPO | 偏好数据 | 中等 | 想缩短链路的实验场景 |
| SimPO | 偏好数据 | 中等偏低 | 想降低显存和流程复杂度 |

### 8.3 一个实际决策树

```text
你手里有什么？

1. 只有问答对
  -> 先做 SFT

2. 有高质量 chosen/rejected
  -> 优先试 DPO

3. 有明确的程序可验证奖励
  -> 优先试 GRPO

4. 有成熟奖励模型团队和高投入资源
  -> 可做 PPO

5. 只有单条审核好坏标签
  -> 试 KTO
```

## 9. 对齐训练中的细节问题

### 9.1 对齐不是越强越好

很多团队做对齐时会犯一个错误：

> 觉得“更对齐”一定意味着“更好”。

实际上对齐过度可能导致：

- 模型更保守
- 拒答变多
- 创造性下降
- 回答模板化、官腔化

### 9.2 偏好数据的来源决定上限

常见来源：

- 人工标注
- 用户反馈
- 审核团队打标
- 规则程序自动生成
- 更强模型蒸馏出的偏好

不同来源的问题也不同：

- 人工标注贵但细腻
- 用户反馈真实但噪声大
- 程序规则稳定但容易单一
- 强模型蒸馏高效但可能传播偏差

### 9.3 偏好数据最常见的坑

- 标注标准不一致
- chosen 和 rejected 质量差距过大
- 数据覆盖面窄
- 只关注“礼貌”，忽略“正确”
- 奖励函数只看形式，不看实质

## 10. 训练后怎么判断对齐是否真的有效

不能只看训练 loss。

对齐阶段至少要看四类指标：

### 10.1 偏好胜率

同一个问题，让新旧模型都回答，再做人审或自动评估，看谁更常胜出。

### 10.2 安全与合规指标

- 有害内容拒答率
- 越权回答率
- 幻觉率
- 敏感场景稳定性

### 10.3 任务能力指标

对齐后不应只看风格提升，还要检查基础任务能力是否退化：

- 数学
- 代码
- 事实问答
- 长文本理解

### 10.4 用户体验指标

上线后还要看：

- 用户接受率
- 任务完成率
- 投诉率
- 人工兜底率

---

## 11. 一套推荐的工程顺序

现实里最稳妥的顺序通常是：

```text
高质量基座模型
-> SFT 建立任务能力
-> 建立评测集
-> 收集偏好数据
-> DPO 或 GRPO 做第二阶段优化
-> 小流量验证
-> 上线监控
```

为什么不是一开始就 PPO？

因为很多团队在进入 PPO 前，连下面这些都没做稳：

- 评测集
- 人工审核规范
- 偏好数据质量控制
- 上线回滚流程

如果这些没有，PPO 往往只会把复杂度推高。

---

## 12. 偏好对齐的典型问题案例

### 案例 1：回答更礼貌了，但正确率下降

原因通常是：

- 偏好数据过度偏向礼貌和长回答
- 没有把正确性纳入评测闭环

### 案例 2：模型更安全了，但拒答太多

原因通常是：

- 安全偏好优化过强
- 训练集里的“拒答样本”覆盖过宽

### 案例 3：代码任务 DPO 效果一般，GRPO 更好

原因通常是：

- 代码任务更适合看程序执行结果
- 程序可验证奖励通常比人工主观偏好更稳定

---

## 13. 这一册最重要的结论

- SFT 解决“会不会做”
- 偏好对齐解决“做得好不好”
- PPO 是经典完整路线，但复杂度最高
- DPO 是目前工程里最常见、最实用的偏好优化路线之一
- GRPO 更适合数学、代码、可验证推理任务
- 没有好的偏好数据和评测体系，再高级的方法也可能收益有限
