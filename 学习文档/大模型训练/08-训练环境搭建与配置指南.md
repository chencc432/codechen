# 第八册：训练环境搭建与配置指南

> 这一册解决一个最基础、但极容易被忽视的问题：
>
> **训练环境到底怎么搭？**
>
> 很多人一上来就想着跑训练脚本，结果卡在环境上几天都跑不通。驱动不对、CUDA 不匹配、NCCL 报错、PyTorch 装不上、多机通信不通……这些问题看起来都不是"算法问题"，但它们能直接决定你的训练任务能不能启动。
>
> 这册的目标是：让你从一台裸机或一个空容器出发，能一步步搭出一个稳定、可复现、可用于生产的大模型训练环境。

---

## 1. 先建立一个全局认知：训练环境由什么组成

大模型训练环境不是装一个 Python 包就行的事。它是一个多层栈：

```text
┌───────────────────────────────────────────┐
│            训练框架 & 工具层               │
│  ms-swift / LLaMA-Factory / TRL /         │
│  Megatron-LM / 自研训练代码               │
├───────────────────────────────────────────┤
│            Python 核心库层                 │
│  PyTorch / Transformers / PEFT /          │
│  Datasets / DeepSpeed / Accelerate /      │
│  Flash Attention / vLLM（推理验证用）      │
├───────────────────────────────────────────┤
│            Python 环境层                   │
│  Python 3.10+ / conda or venv / pip       │
├───────────────────────────────────────────┤
│            GPU 计算栈                      │
│  CUDA Toolkit / cuDNN / NCCL              │
├───────────────────────────────────────────┤
│            驱动层                          │
│  NVIDIA Driver                            │
├───────────────────────────────────────────┤
│            操作系统层                      │
│  Linux (Ubuntu / CentOS / Rocky)          │
├───────────────────────────────────────────┤
│            硬件层                          │
│  GPU / CPU / 内存 / NVLink / InfiniBand / │
│  高速 SSD / 网络存储                      │
└───────────────────────────────────────────┘
```

每一层出问题，都会导致训练失败。所以搭建环境的正确方式是：**自底向上、逐层验证、每一层都确认通过后再往上走。**

## 2. 硬件层：你需要什么样的机器

### 2.1 GPU 选型

大模型训练对 GPU 要求非常高，不是"有显卡就行"。

| GPU 型号 | 显存 | 适用场景 | 说明 |
|----------|------|----------|------|
| A100 80GB | 80GB | 主流生产训练 | 目前产业界最常见的训练卡 |
| A100 40GB | 40GB | 中等规模训练 | 显存受限，大模型需更多卡 |
| A800 80GB | 80GB | 国内常见替代 | 通信带宽受限，但显存充足 |
| H100 / H800 | 80GB | 新一代训练卡 | 更快的 Tensor Core 和互联 |
| L40S | 48GB | 中小模型训练 | 无 NVLink，适合单机场景 |
| RTX 4090 | 24GB | 个人实验 | 显存小，不支持 NVLink |
| RTX 3090 | 24GB | 个人实验 | 同上，性价比不错 |

核心结论：

- 生产训练优先选 A100/A800 80GB 或更高规格
- 个人实验和快速验证，RTX 4090 / 3090 可以凑合
- 显存是最核心的瓶颈指标

### 2.2 CPU 与内存

- CPU：不是瓶颈，但太弱会拖慢数据预处理和加载
- 内存：建议至少是 GPU 显存总量的 1.5~2 倍。例如 8 卡 A100 80GB = 640GB 显存，建议主机内存至少 1TB
- 内存不足时，数据加载、模型加载、checkpoint 保存都可能出问题

### 2.3 存储

存储性能直接影响训练吞吐：

| 存储类型 | 用途 | 建议 |
|----------|------|------|
| 本地 NVMe SSD | 训练数据缓存、checkpoint 写入 | 每节点至少 2~4TB |
| 分布式存储（Ceph / Lustre / GPFS） | 共享数据集、模型权重 | 带宽要够，延迟要低 |
| 对象存储（S3 / OSS / MinIO） | 归档 checkpoint、长期存储 | 不建议直接用于训练 IO |

常见踩坑：

- 用网络存储直接做训练 IO，导致 dataloader 跟不上 GPU
- checkpoint 保存到慢存储，导致训练卡住
- 存储空间不够，训练跑到中途 checkpoint 写满磁盘

### 2.4 网络

单机训练对网络要求不高，但多机训练时网络是关键：

- **NVLink / NVSwitch**：机内 GPU 间高速互联，A100 / H100 通常标配
- **InfiniBand（IB）**：多机间高速通信，200Gbps 或 400Gbps 是主流
- **RoCE**：基于以太网的 RDMA，成本更低但配置更复杂
- **普通以太网**：只适合小规模实验，多机训练性能极差

核心结论：

- 单机训练：网络不是瓶颈
- 多机训练：没有 IB 或高速 RoCE，通信效率会大幅下降，训练吞吐可能只有理论值的一半甚至更低

## 3. 操作系统层

### 3.1 推荐系统

大模型训练几乎全在 Linux 上跑。

| 系统 | 建议 |
|------|------|
| Ubuntu 20.04 / 22.04 | 最推荐，社区支持最好，驱动兼容性最佳 |
| CentOS 7 | 老系统，很多团队仍在用，但依赖老旧 |
| Rocky Linux 8/9 | CentOS 替代，适合企业 |
| Debian | 可以用，但社区资源稍少 |

不建议用 Windows 做训练环境，WSL2 只适合极小规模的个人实验。

### 3.2 系统基础配置

拿到一台新机器后，建议先做以下配置：

```bash
# 更新系统
sudo apt update && sudo apt upgrade -y

# 安装基础依赖
sudo apt install -y build-essential git curl wget vim \
  software-properties-common ca-certificates \
  lsb-release gnupg2 net-tools htop tmux

# 如果需要编译安装某些库
sudo apt install -y cmake ninja-build pkg-config
```

### 3.3 关键系统参数调优

以下参数在大规模训练时可能需要调整：

```bash
# 增大打开文件数限制（避免多进程场景报 "Too many open files"）
# 在 /etc/security/limits.conf 追加：
* soft nofile 1048576
* hard nofile 1048576

# 增大共享内存（PyTorch DataLoader 多 worker 场景）
# 确认 /dev/shm 足够大，建议至少 64GB
df -h /dev/shm

# 如果不够，临时调大：
sudo mount -o remount,size=128G /dev/shm

# 永久修改：在 /etc/fstab 中设置
# tmpfs /dev/shm tmpfs defaults,size=128G 0 0
```

## 4. NVIDIA 驱动安装

驱动是整个 GPU 计算栈的地基。驱动不对，后面全白搭。

### 4.1 查看当前状态

```bash
# 查看是否有 NVIDIA GPU
lspci | grep -i nvidia

# 查看当前驱动版本（如果已安装）
nvidia-smi
```

### 4.2 安装驱动

推荐使用 NVIDIA 官方仓库安装，不建议手动下载 `.run` 文件：

```bash
# Ubuntu 安装示例
# 添加 NVIDIA 仓库
sudo apt install -y linux-headers-$(uname -r)
distribution=$(. /etc/os-release;echo $ID$VERSION_ID | sed -e 's/\.//g')
wget https://developer.download.nvidia.com/compute/cuda/repos/$distribution/x86_64/cuda-keyring_1.1-1_all.deb
sudo dpkg -i cuda-keyring_1.1-1_all.deb

# 安装驱动
sudo apt update
sudo apt install -y nvidia-driver-550
# 版本号根据你的 GPU 和 CUDA 需求选择

# 重启
sudo reboot
```

### 4.3 验证驱动

```bash
nvidia-smi
```

正常输出应该能看到：

- Driver Version
- CUDA Version（这是驱动支持的最高 CUDA 版本）
- 每张 GPU 的型号、温度、显存占用

### 4.4 驱动版本选择原则

- 驱动版本决定了你能用的 CUDA 版本上限
- 太老的驱动可能不支持新版 CUDA
- 太新的驱动可能在老内核上不稳定
- 生产环境建议使用 NVIDIA 推荐的稳定版本，不追最新

| CUDA 版本 | 最低驱动版本 |
|-----------|-------------|
| CUDA 11.8 | ≥ 520.61.05 |
| CUDA 12.1 | ≥ 530.30.02 |
| CUDA 12.2 | ≥ 535.54.03 |
| CUDA 12.4 | ≥ 550.54.14 |
| CUDA 12.6 | ≥ 560.28.03 |

## 5. CUDA Toolkit 安装

### 5.1 CUDA 是什么

CUDA Toolkit 是 NVIDIA 提供的 GPU 编程工具集，包含：

- CUDA 运行时
- nvcc 编译器
- cuBLAS、cuFFT 等计算库
- 调试与性能分析工具

PyTorch 底层依赖 CUDA 做 GPU 加速计算。

### 5.2 安装方式选择

CUDA 有两种安装方式：

1. **系统级安装**：装到 `/usr/local/cuda`，全局生效
2. **通过 PyTorch 自带 CUDA**：PyTorch 预编译包会自带对应的 CUDA 运行时

实际建议：

- 如果你用 Docker/容器：容器镜像通常已经内置 CUDA，不需要在宿主机装 CUDA Toolkit（只需要宿主机有驱动）
- 如果你在裸机上跑：安装系统级 CUDA
- 如果你只是快速实验：可以只依赖 PyTorch 自带的 CUDA runtime

### 5.3 系统级安装 CUDA Toolkit

```bash
# 以 CUDA 12.4 为例
wget https://developer.download.nvidia.com/compute/cuda/12.4.0/local_installers/cuda_12.4.0_550.54.14_linux.run
sudo sh cuda_12.4.0_550.54.14_linux.run

# 安装时注意：
# - 如果你已经装了驱动，不要勾选 "Driver"
# - 勾选 "CUDA Toolkit"
# - 默认安装路径 /usr/local/cuda-12.4
```

### 5.4 配置环境变量

```bash
# 在 ~/.bashrc 或 ~/.zshrc 追加
export CUDA_HOME=/usr/local/cuda
export PATH=$CUDA_HOME/bin:$PATH
export LD_LIBRARY_PATH=$CUDA_HOME/lib64:$LD_LIBRARY_PATH

# 生效
source ~/.bashrc
```

### 5.5 验证 CUDA

```bash
nvcc --version
# 应该输出你安装的 CUDA 版本

# 运行一个简单测试
cd /usr/local/cuda/samples/1_Utilities/deviceQuery
sudo make
./deviceQuery
# 应输出 Result = PASS
```

## 6. cuDNN 安装

### 6.1 cuDNN 是什么

cuDNN（CUDA Deep Neural Network library）是 NVIDIA 的深度学习加速库，PyTorch 的 CNN、RNN、Transformer 等操作底层依赖它。

### 6.2 安装

```bash
# 推荐用 apt/yum 包管理器安装
sudo apt install -y libcudnn8 libcudnn8-dev

# 或者从 NVIDIA 官网下载 tar 包手动安装
tar -xf cudnn-linux-x86_64-9.x.x.x_cuda12-archive.tar.xz
sudo cp cudnn-*-archive/include/cudnn*.h /usr/local/cuda/include
sudo cp -P cudnn-*-archive/lib/libcudnn* /usr/local/cuda/lib64
sudo chmod a+r /usr/local/cuda/include/cudnn*.h /usr/local/cuda/lib64/libcudnn*
```

### 6.3 验证

```bash
# 检查版本
cat /usr/local/cuda/include/cudnn_version.h | grep CUDNN_MAJOR -A 2
# 或者
python -c "import torch; print(torch.backends.cudnn.version())"
```

## 7. NCCL 安装

### 7.1 NCCL 是什么

NCCL（NVIDIA Collective Communications Library）是 NVIDIA 的多 GPU 通信库。多卡训练、分布式训练全靠它：

- AllReduce
- AllGather
- Broadcast
- ReduceScatter

### 7.2 为什么 NCCL 很关键

因为分布式训练的核心瓶颈之一就是通信。NCCL 直接影响：

- 多卡同步梯度的速度
- 训练吞吐
- 扩展效率

### 7.3 安装

```bash
# apt 安装
sudo apt install -y libnccl2 libnccl-dev

# 验证
python -c "import torch; print(torch.cuda.nccl.version())"
```

### 7.4 NCCL 关键环境变量

在多机训练时，以下环境变量很常用：

```bash
# 指定通信接口（多机训练时必须设对）
export NCCL_SOCKET_IFNAME=eth0
# 或者用 IB 接口名，如 ibp3s0

# 指定使用 IB 或 Socket
export NCCL_IB_DISABLE=0       # 0 表示启用 IB
export NCCL_NET_GDR_LEVEL=5    # GPUDirect RDMA 级别

# 调试时可以开日志
export NCCL_DEBUG=INFO
export NCCL_DEBUG_SUBSYS=ALL

# 设置超时时间（秒），防止某个 rank 卡死导致整个训练挂起
export NCCL_TIMEOUT=1800
```

## 8. Python 环境搭建

### 8.1 Python 版本选择

当前主流推荐：

- **Python 3.10** 或 **Python 3.11**
- 不建议用 3.8 及以下（很多新库已不支持）
- 3.12+ 可能存在某些库兼容问题，需要留意

### 8.2 用 Conda 还是 venv

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| Conda (Miniconda) | 管理 Python 版本方便，能管 CUDA 运行时 | 体积大，有时 resolve 慢 | 裸机开发、多版本共存 |
| venv + pip | 轻量，标准 | 不能管 Python 版本本身 | 容器内、已有 Python |
| Docker 容器 | 完全隔离，可复现 | 需要容器化基础设施 | 生产训练 |

### 8.3 Conda 安装与配置

```bash
# 下载 Miniconda
wget https://repo.anaconda.com/miniconda/Miniconda3-latest-Linux-x86_64.sh
bash Miniconda3-latest-Linux-x86_64.sh -b -p $HOME/miniconda3

# 初始化
$HOME/miniconda3/bin/conda init bash
source ~/.bashrc

# 创建训练环境
conda create -n train python=3.10 -y
conda activate train
```

### 8.4 pip 镜像源配置

国内网络环境下，建议配置国内镜像源以加速下载：

```bash
# 临时使用
pip install torch -i https://mirrors.aliyun.com/pypi/simple/

# 永久配置
pip config set global.index-url https://mirrors.aliyun.com/pypi/simple/
pip config set global.trusted-host mirrors.aliyun.com
```

常用国内镜像：

| 镜像源 | 地址 |
|--------|------|
| 阿里云 | https://mirrors.aliyun.com/pypi/simple/ |
| 清华 | https://pypi.tuna.tsinghua.edu.cn/simple/ |
| 中科大 | https://pypi.mirrors.ustc.edu.cn/simple/ |
| 豆瓣 | https://pypi.doubanio.com/simple/ |

## 9. PyTorch 安装

PyTorch 是大模型训练最核心的框架，装不对就全白搭。

### 9.1 安装原则

**PyTorch 版本必须和 CUDA 版本匹配。**

这是最常见的环境问题根源。例如你系统装了 CUDA 12.4，但 PyTorch 编译时用的是 CUDA 11.8，那么很多功能可能不正常。

### 9.2 查看兼容矩阵

去 PyTorch 官网 https://pytorch.org/get-started/locally/ 选择你的配置。

### 9.3 安装命令示例

```bash
# PyTorch 2.4+ with CUDA 12.4（示例，具体版本请以官方为准）
pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu124

# 如果用 CUDA 12.1
pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu121

# 如果用 CUDA 11.8
pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu118
```

### 9.4 验证 PyTorch + CUDA

```bash
python -c "
import torch
print('PyTorch version:', torch.__version__)
print('CUDA available:', torch.cuda.is_available())
print('CUDA version:', torch.version.cuda)
print('cuDNN version:', torch.backends.cudnn.version())
print('GPU count:', torch.cuda.device_count())
for i in range(torch.cuda.device_count()):
    print(f'  GPU {i}: {torch.cuda.get_device_name(i)}')
    print(f'    Memory: {torch.cuda.get_device_properties(i).total_mem / 1024**3:.1f} GB')
"
```

如果 `CUDA available` 输出 `False`，说明 PyTorch 和 CUDA 没对上，需要排查。

### 9.5 一个快速的 GPU 运算验证

```bash
python -c "
import torch
x = torch.randn(1000, 1000, device='cuda')
y = torch.randn(1000, 1000, device='cuda')
z = torch.matmul(x, y)
print('GPU computation OK, result shape:', z.shape)
"
```

## 10. 核心 Python 库安装

### 10.1 Hugging Face 生态

这是大模型后训练最常用的库集合：

```bash
# Transformers：模型加载、推理、训练的核心库
pip install transformers

# Datasets：数据集加载和处理
pip install datasets

# PEFT：参数高效微调（LoRA、QLoRA 等）
pip install peft

# TRL：强化学习训练（DPO、PPO、GRPO 等）
pip install trl

# Accelerate：分布式训练封装
pip install accelerate

# Tokenizers：高性能分词
pip install tokenizers

# safetensors：安全的模型权重格式
pip install safetensors
```

### 10.2 DeepSpeed

DeepSpeed 是微软的分布式训练优化框架，用于 ZeRO、流水并行等场景：

```bash
pip install deepspeed

# 验证
ds_report
```

`ds_report` 会输出 DeepSpeed 对各种 ops 的编译状态，有些 ops 需要在首次使用时 JIT 编译，如果你想提前编译：

```bash
DS_BUILD_OPS=1 pip install deepspeed --global-option="build_ext"
```

### 10.3 Flash Attention

Flash Attention 是加速 Transformer 注意力计算的关键库，能显著降低显存占用和提高训练速度：

```bash
pip install flash-attn --no-build-isolation
```

注意：

- Flash Attention 安装需要 CUDA 和 `ninja`，编译时间较长
- 版本需要和 PyTorch / CUDA 匹配
- Ampere（A100）及更新架构才支持

如果安装失败，常见原因是：

- CUDA 版本不匹配
- 缺少 `ninja`（`pip install ninja`）
- GCC 版本太低

### 10.4 其他常用工具库

```bash
# 训练监控
pip install wandb         # Weights & Biases
pip install tensorboard   # TensorBoard

# 模型量化
pip install bitsandbytes  # QLoRA 依赖

# 数据处理
pip install pandas pyarrow

# 进度条
pip install tqdm

# 配置管理
pip install pyyaml omegaconf
```

### 10.5 bitsandbytes 安装注意事项

bitsandbytes 是 QLoRA 的核心依赖，安装时常见问题：

```bash
pip install bitsandbytes

# 验证
python -c "import bitsandbytes; print(bitsandbytes.__version__)"
```

如果报 CUDA 相关错误，可能是：

- 系统 CUDA 版本和 bitsandbytes 编译版本不一致
- `LD_LIBRARY_PATH` 没配对

## 12. Docker 容器化环境

生产训练强烈建议使用容器化环境，这是保证可复现性的最有效手段。

### 12.1 为什么要用容器

- 环境完全隔离，不受宿主机影响
- 版本锁定，不会"环境漂移"
- 部署一致，开发和生产用同一镜像
- 方便分发和协作

### 12.2 安装 Docker 和 NVIDIA Container Toolkit

```bash
# 安装 Docker

curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | \
  sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg
curl -s -L https://nvidia.github.io/libnvidia-container/$distribution/libnvidia-container.list | \
  sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
  sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list

sudo apt update
sudo apt install -y nvidia-container-toolkit
sudo nvidia-ctk runtime configure --runtime=docker
sudo systemctl restart docker
```

### 12.3 验证容器内 GPU 可用

```bash

```

## 13. 多机多卡分布式环境配置

### 13.1 什么时候需要多机

- 单机 GPU 数不够
- 模型太大，单机显存放不下
- 需要更大的有效 batch size

### 13.2 多机通信前提

所有训练节点之间必须能互相通信：

1. **网络互通**：节点间端口可达
2. **免密 SSH**（torchrun / pdsh 模式下需要）
3. **共享存储**：数据和模型权重所有节点可见
4. **相同环境**：所有节点的驱动、CUDA、Python、PyTorch 版本完全一致

### 13.3 配置免密 SSH

```bash
# 在主节点生成密钥
ssh-keygen -t rsa -N "" -f ~/.ssh/id_rsa

# 将公钥分发到所有节点（包括自己）
ssh-copy-id user@node0
ssh-copy-id user@node1
ssh-copy-id user@node2
ssh-copy-id user@node3

# 验证
ssh user@node1 "hostname"
```

### 13.4 用 torchrun 启动多机训练

```bash
# 在每台机器上执行（以 2 机 8 卡为例）

# node0（主节点）
torchrun \
  --nnodes=2 \
  --nproc_per_node=4 \
  --node_rank=0 \
  --master_addr=node0 \
  --master_port=29500 \
  train.py --config config.yaml

# node1
torchrun \
  --nnodes=2 \
  --nproc_per_node=4 \
  --node_rank=1 \
  --master_addr=node0 \
  --master_port=29500 \
  train.py --config config.yaml
```

关键参数说明：

| 参数 | 含义 |
|------|------|
| `--nnodes` | 总节点数 |
| `--nproc_per_node` | 每个节点的 GPU 数 |
| `--node_rank` | 当前节点的编号（从 0 开始） |
| `--master_addr` | 主节点的 IP 或主机名 |
| `--master_port` | 通信端口，需要所有节点可达 |

### 13.5 多机训练的环境变量

```bash
# 指定可见 GPU
export CUDA_VISIBLE_DEVICES=0,1,2,3

# NCCL 通信配置
export NCCL_SOCKET_IFNAME=eth0    # 使用的网络接口
export NCCL_IB_DISABLE=0          # 是否禁用 IB（0=启用）
export NCCL_DEBUG=WARN            # 日志级别

# Rendezvous 相关
export MASTER_ADDR=node0
export MASTER_PORT=29500
```

### 13.6 多机训练验证脚本

在正式跑训练前，建议先用简单脚本验证多机通信是否正常：

```python
# test_distributed.py
import os
import torch
import torch.distributed as dist

def main():
    dist.init_process_group(backend="nccl")
    rank = dist.get_rank()
    world_size = dist.get_world_size()
    local_rank = int(os.environ.get("LOCAL_RANK", 0))
    
    torch.cuda.set_device(local_rank)
    device = torch.device(f"cuda:{local_rank}")
    
    tensor = torch.tensor([rank], dtype=torch.float32, device=device)
    dist.all_reduce(tensor, op=dist.ReduceOp.SUM)
    
    expected = sum(range(world_size))
    if rank == 0:
        print(f"World size: {world_size}")
        print(f"AllReduce result: {tensor.item()}, expected: {expected}")
        if tensor.item() == expected:
            print("Distributed communication OK!")
        else:
            print("ERROR: Communication mismatch!")
    
    dist.destroy_process_group()

if __name__ == "__main__":
    main()
```

运行：

```bash
torchrun --nnodes=2 --nproc_per_node=4 --node_rank=0 \
  --master_addr=node0 --master_port=29500 test_distributed.py
```

## 18. 环境常见问题排查

### 18.1 CUDA / PyTorch 版本不匹配

**现象**：`torch.cuda.is_available()` 返回 `False`，或者报 CUDA 相关错误。

**排查**：

```bash
# 查看系统 CUDA 版本
nvcc --version

# 查看 PyTorch 编译时的 CUDA 版本
python -c "import torch; print(torch.version.cuda)"

# 两者应该大版本一致
```

**解决**：重新安装和系统 CUDA 匹配的 PyTorch 版本。

### 18.2 NCCL 通信超时

**现象**：多卡训练时卡住不动，最终报 NCCL timeout。

**排查清单**：

1. 检查防火墙是否阻断了节点间通信端口
2. 检查 `NCCL_SOCKET_IFNAME` 是否设置正确
3. 检查各节点的 NCCL 版本是否一致
4. 缩回单机验证是否正常
5. 开启 `NCCL_DEBUG=INFO` 看详细日志

### 18.3 共享内存不足

**现象**：`DataLoader` 报 `RuntimeError: DataLoader worker is killed by signal: Bus error`。

**解决**：

```bash
# 容器内
docker run --shm-size=64g ...

# 裸机
sudo mount -o remount,size=128G /dev/shm
```

### 18.4 Flash Attention 安装失败

**现象**：`pip install flash-attn` 报编译错误。

**排查**：

1. 确认 CUDA 版本 ≥ 11.6
2. 确认 GPU 架构是 Ampere（A100）或更新
3. 安装 `ninja`：`pip install ninja`
4. 确认 GCC 版本 ≥ 7
5. 如果还失败，尝试从 GitHub release 下载预编译 wheel

### 18.5 bitsandbytes CUDA 版本问题

**现象**：`import bitsandbytes` 报错，提示找不到 CUDA 库。

**排查**：

```bash
python -m bitsandbytes
# 会输出详细的 CUDA 检测信息
```

**常见解决**：

- 确保 `LD_LIBRARY_PATH` 包含 CUDA lib 路径
- 使用和系统 CUDA 匹配的 bitsandbytes 版本

### 18.6 磁盘空间不足

**现象**：训练中突然报 IO Error 或 checkpoint 保存失败。

**预防**：

```bash
# 训练前检查空间
df -h /output
df -h /data

# 估算空间需求：
# - 模型权重：约 2 * 参数量 bytes（bf16）
#   例如 7B 模型 ≈ 14GB
# - 每个 checkpoint：LoRA adapter 通常几百 MB
#   全量微调 checkpoint ≈ 模型大小
# - 优化器状态：全量微调时约等于模型大小的 2-3 倍
# - 日志和 TensorBoard 事件文件：通常不大
```

### 18.7 权限问题

**现象**：模型文件或数据文件无法读取，或者 checkpoint 无法写入。

**排查**：

```bash
ls -la /models/qwen2.5-7b-instruct/
ls -la /data/
ls -la /output/

# 确保当前用户有读写权限
# 容器内通常是 root，但挂载目录可能有宿主机权限限制
```

## 19. 环境版本锁定与复现

### 19.1 为什么必须锁版本

"我的环境能跑，你的跑不了"是团队最常见的问题之一。

根因几乎都是：

- Python 包版本不一致
- CUDA 版本不一致
- 驱动版本不一致
- 系统库版本不一致

### 19.2 用 requirements.txt 锁定

```bash
# 导出当前环境
pip freeze > requirements.txt

# 在新环境安装
pip install -r requirements.txt
```

但 `pip freeze` 导出的是全量依赖，可能包含很多不需要的包。更好的做法是手动维护一份核心依赖文件：

```text
# requirements-train.txt
torch==2.4.0
transformers==4.46.0
datasets==3.0.0
peft==0.13.0
trl==0.12.0
accelerate==1.0.0
deepspeed==0.15.0
flash-attn==2.6.0
bitsandbytes==0.44.0
safetensors==0.4.0
wandb==0.18.0
tensorboard==2.18.0
ms-swift==3.0.0
```

### 19.3 用 Docker 锁定完整环境

这是最彻底的方案。将上面所有步骤写进 Dockerfile，构建后打上版本 tag：

```bash
docker build -t llm-train:v1.0-cu124-pt2.4 .
docker tag llm-train:v1.0-cu124-pt2.4 registry.example.com/llm-train:v1.0-cu124-pt2.4
docker push registry.example.com/llm-train:v1.0-cu124-pt2.4
```

## 22. 本册总结

### 22.1 核心结论

- 训练环境不是"装个 Python 包"，而是一个多层栈，每一层都可能出问题
- 自底向上搭建、逐层验证是唯一靠谱的方式
- PyTorch 和 CUDA 版本匹配是最常见的坑
- 生产环境必须用容器化 + 版本锁定
- 多机训练的通信配置是环境搭建中最复杂的部分

### 22.2 你应该具备的能力

读完这册后，你应该能：

- 从一台裸机搭建出完整的训练环境
- 写出一个可用于生产的训练 Dockerfile
- 配置多机多卡的分布式训练通信
- 排查常见的环境问题
- 锁定和复现训练环境

这册是全套教材的"第零步"。如果你没有一个能跑的环境，前面七册讲的一切都只是纸上谈兵。
