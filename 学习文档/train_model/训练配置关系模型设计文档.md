# 大模型调优 —— 训练配置关系模型设计文档

> 版本：v1.0 | 时间计划：Q3，8 月中旬前上线第一版

---

## 一、目标与背景

### 1.1 目标

让**不同部门的新同学**用同一个模板快速跑起训练任务，对大模型进行调优，将更多时间聚焦在**基础模型选择、数据集质量**等业务本身，而非环境搭建、资源配置等基础设施事项。

### 1.2 现状与痛点

训练平台当前采用**过程式**方式让用户完成训练任务——将训练生命周期拆分成集群、存储、镜像等独立模块，由用户自行组装。开放性大、概念多，导致以下问题：

| 痛点分类 | 具体表现 | 影响 |
|---------|---------|------|
| **入门门槛高** | 新同学刚使用平台时无从下手，不知道从哪开始 | 频繁咨询，上手周期长 |
| **环境理解成本** | 用户需了解 conda、镜像等知识；常见问题："环境怎么到训练就没有了" | 大量时间陷入与业务无关的事项 |
| **资源配置易错** | 资源配置不正确导致任务中断（如内存太小 OOM），用户不清楚原因 | 排查成本高，影响训练进度 |
| **环境不复用** | 后训练场景下环境高度相似（SFT/RL 框架趋于统一），但平台未统一支持环境复用 | 重复搭建环境，效率低下 |

**关键洞察**：大模型后训练（微调/全量调优）在某些阶段**趋于统一**——训练数据集格式统一、训练框架统一。SFT 和 RL 各有主流框架，**环境的复用能极大加速训练任务进展**。

### 1.3 平台能统一的部分

```
┌──────────────────────────────────────────────────┐
│              平台可以统一管控                       │
│  ┌──────────────┬──────────────┬───────────────┐  │
│  │ 训练数据集格式 │  训练环境     │  训练代码      │  │
│  │ (alpaca/msg) │ (镜像/CUDA)  │ (框架内置)    │  │
│  └──────────────┴──────────────┴───────────────┘  │
├──────────────────────────────────────────────────┤
│              用户需要关注的                         │
│  ┌──────────────┬──────────────┬───────────────┐  │
│  │ 选哪个基座模型 │  准备什么数据  │  调什么超参数  │  │
│  └──────────────┴──────────────┴───────────────┘  │
└──────────────────────────────────────────────────┘
```

## 二、方案总览

### 2.1 核心思路：三维度决策模型

**模型类型 + 训练类型 + 设备类型** 共同决定最终的训练配置（TrainConfig）。用户只需做出这三个选择，平台自动匹配出完整的运行环境、框架、超参数模板，用户在此基础上只需关注模型和数据。

```
                         ┌─────────────────────────────────┐
                         │     用户做 3 个选择               │
                         └─────────────────────────────────┘
                                       │
               ┌───────────────────────┼───────────────────────┐
               ▼                       ▼                       ▼
      ┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
      │   模型类型        │    │   训练类型        │    │   设备类型       │
      │   Model Type     │    │   Train Type     │    │   Device Type   │
      │                  │    │                  │    │                 │
      │ · Qwen2          │    │ · SFT-全量微调    │    │ · GPU (A800)    │
      │ · Qwen3          │    │ · SFT-LoRA       │    │ · NPU (910B)    │
      │ · DeepSeek       │    │ · RL (后续)       │    │                 │
      └────────┬─────────┘    └────────┬─────────┘    └────────┬────────┘
               │                       │                       │
               └───────────────────────┼───────────────────────┘
                                       ▼
                    ┌──────────────────────────────────────┐
                    │     TrainConfig（训练配置）            │
                    │                                      │
                    │  自动确定：                            │
                    │   · 框架 & 版本（仅展示）              │
                    │   · 镜像 & CUDA 版本                  │
                    │   · 执行命令入口                      │
                    │   · 资源配额（用户仅选节点数）          │
                    │   · 默认超参数模板                    │
                    │   · template                         │
                    │   · 数据集格式要求                    │
                    │   · 监控方式                          │
                    │                                      │
                    │  用户填写/调整：                       │
                    │   · 基座模型（从模型仓库选）            │
                    │   · 训练数据集路径                    │
                    │   · 节点数                            │
                    │   · 可选超参数覆盖                    │
                    └──────────────┬───────────────────────┘
                                   │
                                   ▼
                    ┌──────────────────────────────────────┐
                    │  提交弹性训练任务                      │
                    │  平台自动注入环境变量 → 拉起 Pod       │
                    └──────────────────────────────────────┘
```

> **入口说明**：若有前台模型广场，用户绑定模型类型后可直接跳过"选择模型类型"步骤，从模型广场一键进入训练配置。

### 2.2 三维度各自的职责

| 维度 | 决定了什么 | 关键约束 |
|------|-----------|---------|
| **模型类型** | template（对话模板）、默认超参数范围、基座模型路径 | 模型类型决定 template 值，template 范围由框架版本决定 |
| **训练类型** | 微调方法（full/lora）、DeepSpeed 策略、显存需求、数据集格式 | SFT-全量需要更多资源和 ZeRO-3；LoRA 资源更少用 ZeRO-2 |
| **设备类型** | 可用框架、镜像版本、GPU/NPU 型号、资源配额上限 | NPU 当前仅支持 ms-swift；GPU 支持 swift + llama-factory |

### 2.3 组合有效性矩阵

并非所有组合都合法，平台根据以下矩阵过滤无效选项：

| 模型类型 | 训练类型 | 设备类型 | 框架 | 支持 |
|---------|---------|---------|------|------|
| Qwen2 | SFT-全量 | GPU | ms-swift 3.x | ✅ |
| Qwen2 | SFT-LoRA | GPU | ms-swift 3.x | ✅ |
| Qwen2 | SFT-全量 | GPU | llama-factory 0.9.x | ✅ |
| Qwen2 | SFT-LoRA | GPU | llama-factory 0.9.x | ✅ |
| Qwen2 | SFT-全量 | NPU | ms-swift 3.x | ✅ |
| Qwen2 | SFT-LoRA | NPU | ms-swift 3.x | ✅ |
| Qwen2 | SFT-* | NPU | llama-factory | ❌ |
| Qwen3 | SFT-全量 | GPU | ms-swift 3.x | ✅ |
| Qwen3 | SFT-LoRA | GPU | ms-swift 3.x | ✅ |
| Qwen3 | SFT-全量 | GPU | llama-factory 0.9.x | ✅ |
| Qwen3 | SFT-LoRA | GPU | llama-factory 0.9.x | ✅ |
| * | RL-* | * | * | 🔜 后续 |

> **过滤链路**：设备类型 → 过滤可用框架 → 框架过滤可用 template/模型 → 训练类型决定超参数模板。

### 3.1 模型广场（模型调优入口）

顶部菜单栏新增「**模型调优**」tab，用户进入后可以看到：

### 3.2 训练任务列表

训练任务列表页新增「**模型调优**」按钮，可直接创建基于模型的任务（区别于现有的过程式自定义任务）。

### 3.3 存储简化

逐步淡化**个人存储、空间存储**概念，用户只需关注一个核心路径：

```
/home/kas/kas_checkpoint/{username}/
```

模型产物、训练日志统一写入此路径，再通过容器开发环境查看。

## 四、后台设计

### 4.1 整体架构

后台将训练入口和环境拆开管理，可扩展性较强。模型来源于 KAS 存储，只需模型与训练环境匹配上即可训练。

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  ModelType   │     │  TrainType   │     │  DeviceType  │
│  (模型类型)   │     │  (训练类型)   │     │  (设备类型)   │
└──────┬──────┘     └──────┬──────┘     └──────┬──────┘
       │                   │                   │
       └───────────┬───────┴───────────┬───────┘
                   ▼                   ▼
          ┌────────────────────────────────────┐
          │        TrainConfig（训练配置）       │
          │  绑定关系建立后，其他内容自动覆盖       │
          ├────────────────────────────────────┤
          │  1. framework     (框架信息-展示)    │
          │  2. task_type     (SFT/RL)         │
          │  3. model_template(模板映射)        │
          │  4. environment   (镜像/CUDA)       │
          │  5. execution     (命令入口)         │
          │  6. device_env    (GPU/NPU)         │
          │  7. resources     (资源配额)         │
          │  8. hyperparams   (超参数集)         │
          │  9. publish       (发布状态)         │
          └──────────────┬─────────────────────┘
                         │ 1:N
                         ▼
          ┌────────────────────────────────────┐
          │    TrainTask（用户训练任务实例）       │
          ├────────────────────────────────────┤
          │  config_id (关联的训练配置)           │
          │  user_id                            │
          │  model_path (用户选的模型)            │
          │  datasets[] (用户选的数据集)          │
          │  node_count (用户选的节点数)          │
          │  custom_params {} (用户覆盖的超参)    │
          │  status / created_at                │
          └────────────────────────────────────┘
```

### 4.2 模型广场后台

**核心价值**：让用户快速了解哪些模型可以进行调优、哪些不支持。有了绑定关系后，用户创建任务时其他内容自动覆盖。

| 管理项 | 说明 |
|-------|------|
| 模型列表管理 | 维护已集成的模型，非模型仓库全量 |
| 模型-训练配置绑定 | 每个模型绑定对应的 TrainConfig |
| 模型信息展示 | 模型介绍、训练参数说明、数据集格式说明 |

> **策略选择**：给每个模型绑定一个训练环境，虽然绑定操作繁琐，但使用上限制更少。不同设备（GPU/NPU）的环境分开绑定。

## 五、TrainConfig 属性详解

### 5.1 框架信息（仅展示）

框架字段仅作为展示用途，让用户了解底层使用的训练框架，**无需用户自行安装或配置**。

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `framework_name` | string | 框架名称 | `ms-swift` |
| `framework_version` | string | 版本号 | `3.5.0` |
| `display_label` | string | 前端展示文案 | `ms-swift v3.5.0` |

当前支持的框架：

| 框架 | 版本 | 适用场景 |
|------|------|---------|
| **ms-swift** | 3.5.0（持续更新） | GPU + NPU 均支持，Qwen 系列原生支持好 |
| **llama-factory** | 0.9.2 | 仅 GPU，多模型 template 覆盖面广 |

### 5.2 支持任务类型

| 任务类型 | 方法 | `TRAIN_TYPE` 值 | 说明 | 状态 |
|---------|------|----------------|------|------|
| **SFT** | 全量微调 | `full` | 更新模型所有参数，效果更好但资源消耗大 | ✅ 已支持 |
| **SFT** | LoRA | `lora` | 低秩适配，仅训练少量参数，显存友好 | ✅ 已支持 |
| **RL** | GRPO / PPO | - | 强化学习对齐 | 🔜 后续 |

> `TRAIN_TYPE` 由用户选择微调方法时决定，不放入训练镜像超参数，而是作为独立的环境变量传入。

### 5.3 支持调参大模型（模型类型 → Template 映射）

模型类型决定了框架内使用的 **template**（对话格式模板），template 的可选范围由框架决定。

**ms-swift 当前支持的 template**（以 Qwen 系列为主，传空可自动识别）：

| Template 值 | 对应模型系列 | 说明 |
|-------------|-------------|------|
| `qwen` | Qwen 系列通用 | 传空时框架自动从模型配置中读取 |
| `qwen2_5` | Qwen2.5 系列 | 文本模型 |
| `qwen3` | Qwen3 系列 | 文本模型 |
| `qwen_vl` | Qwen-VL | 视觉语言模型 |
| `qwen2_vl` | Qwen2-VL | 视觉语言模型 |
| `qwen2_5_vl` | Qwen2.5-VL | 视觉语言模型 |
| `deepseek` | DeepSeek 系列 | 文本模型 |
| `deepseek_coder` | DeepSeek Coder | 代码模型 |

> **第一版**：以 Qwen 为主，template 传空即可（框架自动识别）。后续扩展更多模型时，由 `MODEL_TYPE` 环境变量控制。

### 5.4 环境（镜像配置）

每个训练配置绑定一个特定镜像。CUDA 版本和系统版本通过**镜像备注**字段录入并展示。

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `image_url` | string | 镜像地址 | `registry.cn/train/swift:3.5.0-cu121` |
| `image_remark` | string | 镜像备注（含 CUDA/系统版本） | `CUDA 12.1 / Ubuntu 22.04 / PyTorch 2.3` |
| `cuda_version` | string | 从备注中提取展示 | `12.1` |
| `os_version` | string | 从备注中提取展示 | `Ubuntu 22.04` |

### 5.5 执行命令入口

训练代码已内置于镜像中，通过环境变量传参。自定义代码场景需额外指定工作目录。

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `command` | string | 启动命令 | `bash run_train.sh` |
| `working_dir` | string | 工作目录（代码相对识别路径） | `/workspace/ms-swift` |
| `is_custom_code` | boolean | 是否为自定义代码 | `false` |

> 自定义代码场景：`working_dir` 指向用户代码挂载目录，作为代码相对路径的基准。

### 5.6 支持设备环境

| 设备类型 | 说明 | 可用框架 | 典型型号 |
|---------|------|---------|---------|
| **GPU** | NVIDIA GPU | ms-swift、llama-factory | A800-80G、H800-80G |
| **NPU** | 华为昇腾 NPU | ms-swift（待验证） | Ascend 910B |

### 5.7 默认资源配额

用户仅需选择**节点数**，其余根据训练配置自动填充。后续如果有非弹性池，再放开资源配置。

| 字段 | 类型 | 用户可选 | 默认值 | 说明 |
|------|------|---------|-------|------|
| `node_count` | int | ✅ | `1` | 节点数量 |
| `cpu` | string | ❌ | `64C` | CPU 配额 |
| `memory` | string | ❌ | `512Gi` | 内存配额 |
| `gpu_type` | string | ❌ | `A800-80G` | GPU 型号 |
| `gpu_count_per_node` | int | ❌ | `8` | 每节点 GPU 数 |
| `ib_count` | int | ❌ | `1` | IB 网卡数（多机通信） |

> 当 `node_count > 1` 时，平台自动配置分布式训练环境变量（`MASTER_ADDR`、`MASTER_PORT`、`WORLD_SIZE`、`NODE_RANK` 等）。

### 5.8 超参数

超参数分为**固定参数**（由平台/配置决定）和**可调超参数**（用户可在前端修改），均通过环境变量注入容器。

#### 5.8.1 固定参数（平台自动注入）

以下参数由训练配置和用户选择自动决定，不需要用户在超参数面板中调整：

| swift 原始参数 | 环境变量 | 来源 | 说明 |
|---------------|---------|------|------|

## 六、数据集格式规范

ms-swift 对不同任务类型支持不同的输入格式。第一版以 **alpaca 格式**为默认，后续可选择一种标准格式统一。

> 参考文档：[ms-swift 自定义数据集](https://swift.readthedocs.io/zh-cn/latest/Customization/自定义数据集.html)

### 6.1 SFT - 文本生成（单轮对话）

默认使用 **alpaca 格式**，最直观。文件格式为 `.jsonl`，每行一条样本：

```json
{
    "system": "你是一个有帮助的助手",
    "instruction": "请根据下列问题提取用户目的。\n\n网上值机成功的二维码在机场怎么使用？",
    "input": "",
    "output": "用户目的：了解如何在机场使用网上值机成功后获得的二维码。"
}
```

| 字段 | 必填 | 说明 |
|------|------|------|
| `system` | 可选 | 系统提示词 |
| `instruction` | **必填** | 用户输入/指令 |
| `input` | 可选 | 附加输入，与 instruction 拼接 |
| `output` | **必填** | 模型期望输出 |

### 6.2 SFT - 文本生成（多轮对话）

使用 **messages 标准格式**：

```json
{
    "messages": [
        {"role": "system", "content": "你是一个有帮助的助手"},
        {"role": "user", "content": "浙江的省会在哪？"},
        {"role": "assistant", "content": "浙江的省会在杭州。"},
        {"role": "user", "content": "那广东呢？"},
        {"role": "assistant", "content": "广东的省会在广州。"}
    ]
}
```

### 6.3 SFT - 图片理解（视觉语言模型）

在 messages 格式基础上，添加 `images`、`videos`、`audios` 字段，content 中用 `<image>`、`<video>`、`<audio>` 占位符标记位置。推荐使用**绝对路径**。

**图片理解**：

```json
{
    "messages": [
        {"role": "user", "content": "<image><image>两张图片有什么区别"},
        {"role": "assistant", "content": "前一张是小猫，后一张是小狗"}
    ],
    "images": ["/xxx/x.jpg", "/xxx/x.png"]
}
```

**语音理解**：

```json
{
    "messages": [
        {"role": "user", "content": "<audio>语音说了什么"},
        {"role": "assistant", "content": "今天天气真好呀"}
    ],
    "audios": ["/xxx/x.mp3"]
}
```

**混合多模态**：

```json
{
    "messages": [
        {"role": "system", "content": "你是个有用无害的助手"},
        {"role": "user", "content": "<image>图片中是什么，<video>视频中是什么"},
        {"role": "assistant", "content": "图片中是一个大象，视频中是一只小狗在草地上奔跑"}
    ],
    "images": ["/xxx/x.jpg"],
    "videos": ["/xxx/x.mp4"]
}
```

### 6.4 格式汇总

| 任务场景 | 推荐格式 | 文件类型 | 说明 |
|---------|---------|---------|------|
| SFT - 文本单轮 | alpaca | `.jsonl` | instruction + output，第一版默认 |
| SFT - 文本多轮 | messages | `.jsonl` | 多轮对话标准格式 |
| SFT - 图片理解 | messages + images | `.jsonl` | 多模态占位符 + 资源路径 |
| SFT - 视频理解 | messages + videos | `.jsonl` | 多模态占位符 + 资源路径 |
| DPO - 文本生成 | messages (ChatML) | `.jsonl` | 含 chosen/rejected 对 |
| CPT - 文本生成 | 纯文本 | `.txt` | 连续预训练用 |

## 八、Packing 与数据集缓存

开启 packing 后，框架首次运行会对数据集进行预处理（tokenize + 拼接），产生缓存文件。

| 场景 | 问题 | 解决方案 |
|------|------|---------|
| 每次重启重新制作数据集 | packing 预处理耗时长，重启后缓存丢失 | 缓存路径挂载到持久化存储 |
| 缓存路径管理 | 不同任务的缓存会冲突 | 按任务/数据集 hash 隔离缓存目录 |

```
缓存路径规则:
/home/kas/cache/packing/{model_type}_{dataset_hash}/
```

## 九、新模型 & 框架升级集成流程

### 9.1 新开源模型接入

```
新模型发布（如 Qwen3-32B）
  │
  ├─① 确认框架 template 支持
  │    ├─ ms-swift 已内置 → 直接复用
  │    └─ 未内置 → 自定义代码注册 or 等框架更新
  │
  ├─② 上传模型权重到模型仓库（KAS 存储）
  │
  ├─③ 创建 TrainConfig（草稿状态）
  │    ├─ 绑定 model_type、template
  │    ├─ 参考官方推荐值设定默认超参
  │    └─ 绑定现有镜像（框架未升级则复用）
  │
  ├─④ 标准验证流程（见 9.3）
  │
  └─⑤ 发布 → 用户在模型广场可见
```

### 9.2 框架版本升级

```
框架新版本（如 swift 3.4.1 → 3.5.0）
  │
  ├─① 构建新镜像
  │    ├─ 基于新版本 + 对应 CUDA/PyTorch
  │    └─ 填写镜像备注
  │
  ├─② 创建新 TrainConfig（草稿）
  │    ├─ 绑定新镜像、更新 framework.version
  │    ├─ 验证所有已支持模型的 template 兼容性
  │    └─ 检查环境变量/参数名是否有变更
  │         （如 swift 3.4.1 → 3.5.0 参数名可能调整）
  │
  ├─③ 标准验证流程
  │
  ├─④ 发布新版本
  │
  └─⑤ 旧版本生命周期
       ├─ deprecated = true + 提示迁移
       └─ 观察期后 is_published = false
```

## 十一、关键设计决策 FAQ

### Q1: 给每个模型绑定环境 vs 按任务类型统一走一个环境？

**选择：按模型绑定环境**。虽然绑定操作繁琐，但使用上限制更少。不同模型可能需要不同的框架版本、不同的 template、不同的默认参数。GPU 和 NPU 的环境分开绑定。

### Q2: 模型广场是否有必要存在？

如果通过选择模型类型来匹配训练配置，模型广场作为**独立入口不是必须的**。但模型广场的价值在于：

- 让用户直观看到哪些模型可调优、哪些不支持
- 提供模型介绍、参数说明、数据集格式文档
- 一键创建任务，减少选择步骤

建议**保留模型广场作为快捷入口**，同时训练任务列表也提供「模型调优」创建按钮。

### Q3: 非弹性池场景怎么处理？

第一版所有任务下发弹性池，资源配额由训练配置自动填充。后续如果有非弹性池需求，再放开资源配置让用户自选集群和资源。

### Q4: 框架参数名变更怎么兼容？

swift 3.4.1 到 3.5.0 可能存在参数名调整。通过**镜像内置启动脚本**做适配层——环境变量命名保持稳定，脚本内部根据框架版本翻译为对应参数。

## 十三、要点总结

| **快速集成** | 标准验证流水线，新模型/新框架 3~5 天完成验证接入 |
| **弹性优先** | 第一版全部走弹性池，后续按需开放资源配置 |
