## 第一章 大模型基础认知

### 1.1 什么是大模型

大模型（Large Language Model, LLM）本质上是一个**超大规模的神经网络**，通过海量文本数据学习到了语言的规律。你可以把它理解为：

```
┌──────────────────────────────────────────┐
│            大模型 = 一个函数              │
│                                          │
│   输入：一段文字（prompt）                │
│   输出：接下来最可能的文字（completion）   │
│                                          │
│   函数内部：数十亿到数千亿个"参数"        │
│   这些参数就是模型"学到的知识"            │
└──────────────────────────────────────────┘
```

**关键概念**：

| 概念 | 通俗解释 | 类比 |
|------|---------|------|
| **参数（Parameters）** | 模型内部的数值，决定了模型的行为 | 人脑中的"突触连接强度" |
| **权重（Weights）** | 和参数含义相同，常互换使用 | 同上 |
| **模型大小** | 用参数数量衡量，如 7B = 70 亿参数 | 大脑的"体积" |
| **Token** | 模型处理文本的最小单位，大约 1 个汉字 ≈ 1~2 个 token | 拼音中的一个音节 |

### 1.2 常见的大模型家族

| 模型系列 | 开发方 | 特点 |
|---------|-------|------|
| **Qwen（通义千问）** | 阿里 | 中文能力强，开源版本丰富（Qwen2、Qwen2.5、Qwen3） |
| **DeepSeek** | 深度求索 | 推理能力突出，MoE 架构 |
| **LLaMA** | Meta | 英文能力好，社区生态完善 |
| **ChatGLM** | 智谱 | 中英双语，GLM 架构 |

### 1.3 模型命名规则

以 `Qwen2.5-7B-Instruct` 为例：

```
Qwen2.5  -  7B     -  Instruct
  │         │          │
  系列版本   参数量      变体类型
             │
             7B = 70亿参数
             14B = 140亿
             72B = 720亿
```

**变体类型**：

| 后缀 | 含义 | 用途 |
|------|------|------|
| **Base** | 基础模型，只会"续写" | 需要进一步训练才能对话 |
| **Instruct / Chat** | 指令微调模型，能"对话" | 可直接拿来用，也可以进一步微调 |
| **GPTQ / AWQ** | 量化版本，体积更小 | 推理部署用，不适合训练 |

## 第二章 训练的本质——让模型变得更好

### 2.1 训练到底在做什么

训练的本质是：**通过数据，调整模型的参数，让模型的输出更接近我们期望的结果。**

```
          ┌───────────┐
          │  训练数据   │ ← 你准备的问答对 / 对话 / 文本
          └─────┬─────┘
                │
                ▼
┌───────────────────────────────┐
│          训练过程              │
│                               │
│  1. 模型读入一条数据           │
│  2. 模型产生预测输出           │
│  3. 对比预测和期望，计算"损失"  │ ← loss（越小越好）
│  4. 根据损失调整模型参数       │ ← 反向传播 + 梯度下降
│  5. 重复上述步骤数万~数百万次  │
│                               │
└───────────────────────────────┘
                │
                ▼
          ┌───────────┐
          │  更好的模型 │ ← 参数被更新了
          └───────────┘
```

### 2.2 一些你会反复听到的术语

| 术语 | 解释 | 通俗类比 |
|------|------|---------|
| **Loss（损失）** | 模型预测和标准答案的差距，训练目标是让 loss 越来越小 | 考试分数的"扣分"——越少越好 |
| **Epoch（轮次）** | 整个数据集被模型看一遍，算 1 个 epoch | 课本从头到尾读 1 遍 |
| **Step（步）** | 模型处理一个 batch 的数据并更新一次参数 | 做 1 道练习题 |
| **Batch Size** | 每一步同时处理的数据条数 | 每次做几道题 |
| **Learning Rate（学习率）** | 每一步参数调整的幅度 | 每次改错改多少 |
| **Gradient（梯度）** | 指示参数应该往哪个方向调整 | "错误方向指南针" |
| **Checkpoint** | 训练过程中保存的模型快照 | 游戏存档 |

### 2.3 训练的数学直觉（不需要真的算）

不用担心数学，只需理解一个核心循环：

```
                    ┌──────────────────────┐
                    │ 用当前参数做预测       │
                    └──────────┬───────────┘
                               │
                    ┌──────────▼───────────┐
                    │ 算出 loss（预测 vs 真实）│
          ┌────────┤                       │
          │        └───────────────────────┘
          │
          │        ┌───────────────────────┐
          │        │ 计算梯度（loss 对参数的 │
          └───────►│ 偏导数 = 调整方向）     │
                   └──────────┬────────────┘
                              │
                   ┌──────────▼────────────┐
                   │ 更新参数               │
                   │ 新参数 = 旧参数         │
                   │        - 学习率 × 梯度  │
                   └──────────┬────────────┘
                              │
                              └──→ 回到第一步，重复
```

**核心公式**（一行就够）：

```
新参数 = 旧参数 - 学习率 × 梯度
```

学习率太大 → 参数跳来跳去，训练不稳定
学习率太小 → 收敛太慢，训练很久效果没变化

## 第三章 预训练 vs 后训练——你的任务属于哪种

大模型训练分两大阶段，你几乎只会接触到**后训练**：

```
┌─────────────────────────────────────────────────────────┐
│                    大模型训练全景图                        │
│                                                         │
│   ┌───────────────────────────────────────────────┐     │
│   │  阶段一：预训练（Pre-Training）                │     │
│   │                                               │     │
│   │  · 用 TB 级文本（互联网、书籍、代码等）         │     │
│   │  · 从零训练一个基座模型                        │     │
│   │  · 需要数千张 GPU，训练数周到数月              │     │
│   │  · 成本：数百万到数千万美元                    │     │
│   │  · 通常由大厂/实验室完成                       │     │
│   │                                               │     │
│   │  产出：Base 模型（如 Qwen2-7B）               │     │
│   └───────────────────┬───────────────────────────┘     │
│                       │                                 │
│                       ▼                                 │
│   ┌───────────────────────────────────────────────┐     │
│   │  阶段二：后训练（Post-Training）    ← 你在这里  │     │
│   │                                               │     │
│   │  · 用业务数据对已有模型进行调优                │     │
│   │  · 让模型更好地完成特定任务                    │     │
│   │  · 需要 1~几十张 GPU，训练数小时到数天          │     │
│   │  · 成本：几十到几千元                          │     │
│   │  · 每个业务团队都可以做                        │     │
│   │                                               │     │
│   │  产出：Instruct / Chat / 领域模型             │     │
│   └───────────────────────────────────────────────┘     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**一句话区分**：
- **预训练**：教模型"识字和理解世界"（你不需要做）
- **后训练**：教模型"按你的要求做事"（你要做的就是这个）

## 第四章 后训练三大范式详解

### 4.1 全景图

```
                        后训练
                          │
          ┌───────────────┼───────────────┐
          │               │               │
          ▼               ▼               ▼
    ┌──────────┐   ┌──────────┐   ┌──────────┐
    │   SFT    │   │   RL     │   │   CPT    │
    │ 监督微调  │   │ 强化学习  │   │ 继续预训练│
    └──────────┘   └──────────┘   └──────────┘
    最常用，必学     进阶，后续支持   特定场景使用
```

### 4.2 SFT（Supervised Fine-Tuning）—— 监督微调

**什么是 SFT？** 用"问答对"数据教模型——给它看标准答案，让它学会模仿。

```
数据格式示例（每一条都是"一个问题 + 一个标准答案"）：

用户问：杭州有什么好玩的地方？
标准答：杭州的著名景点有西湖、灵隐寺、千岛湖...

用户问：帮我写一段 Python 排序代码
标准答：def sort_list(lst): ...
```

**SFT 有两种方式**：

#### 全量微调（Full Fine-Tuning）

更新模型的**所有参数**。

```
模型参数（70亿个）
████████████████████████████████
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
全部参与更新
```

| 优点 | 缺点 |
|------|------|
| 效果更好，模型变化更彻底 | 需要大量显存（一般 8 张 80G 的卡起步） |
| 适合数据量大、任务差异大的场景 | 训练时间更长 |
| | 产出的模型和原模型一样大 |

#### LoRA（Low-Rank Adaptation）微调

只在模型上"外挂"一小组新参数，只训练这一小部分。

```
模型原始参数（70亿个，冻结不动）
████████████████████████████████  ← 不更新

外挂 LoRA 参数（仅几百万个）
▓▓▓▓  ← 只更新这些
```

| 优点 | 缺点 |
|------|------|
| 显存需求小得多（1~2 张卡就能跑 7B 模型） | 效果可能稍弱（对于复杂任务） |
| 训练速度快 | 需要调 LoRA 特有参数（rank、alpha） |
| 产出的 LoRA 权重只有几十 MB | 需要和基座模型一起加载使用 |
| 可以为不同任务训练不同 LoRA，灵活切换 | |

**选哪个？** 新手建议先从 **LoRA** 开始，资源需求少、速度快、试错成本低。

### 4.3 RL（Reinforcement Learning）—— 强化学习

用"人类偏好"或"规则奖励"来进一步优化模型行为。典型方法有 RLHF（PPO）、DPO、GRPO 等。

```
┌────────────────────────────────────────┐
│  SFT 模型输出两个回答 A 和 B            │
│  人类（或规则）判断 A 比 B 更好          │
│  模型学会在类似问题上输出更像 A 的回答   │
└────────────────────────────────────────┘
```

> RL 在训练平台上目前规划为后续支持，新手暂时不需要关注。

### 4.4 CPT（Continual Pre-Training）—— 继续预训练

用大量**领域文本**（如医学论文、法律文书）继续训练 Base 模型，让它掌握领域知识。

```
适用场景：模型对某个专业领域"不懂"，需要先灌输领域知识
数据格式：纯文本，不需要问答对
数据量：通常需要 GB 级别
```

### 4.5 三种范式对比

| | SFT | RL | CPT |
|---|---|---|---|
| **目的** | 让模型学会执行特定指令 | 让模型输出更符合人类偏好 | 让模型掌握领域知识 |
| **数据格式** | 问答对（instruction + output） | 偏好对（chosen + rejected） | 纯文本 |
| **数据量** | 几百~几万条 | 几千~几万条 | 几 GB~几 TB |
| **难度** | 入门级 | 进阶 | 中等 |
| **平台支持** | ✅ 已支持 | 🔜 后续 | 视需求 |
| **你应该先学** | **这个** | 之后再说 | 特定场景 |

## 第五章 训练框架——帮你跑起来的"引擎"

### 5.1 什么是训练框架

训练框架是一套封装好的代码工具，帮你完成"读数据 → 喂给模型 → 计算 loss → 更新参数 → 保存模型"这整个流程。你不需要自己写这些底层代码。

```
┌─────────────────────────────────────────────────┐
│                训练框架帮你做的事                  │
│                                                  │
│  你只需要提供：                                    │
│  · 基座模型（从仓库选）                            │
│  · 数据集（按格式准备）                            │
│  · 超参数（有默认值，按需调）                      │
│                                                  │
│  框架自动处理：                                    │
│  · 数据加载和 tokenize                            │
│  · 模型加载和分布式切分                            │
│  · 训练循环（前向 → loss → 反向 → 更新）           │
│  · 混合精度训练（bf16）                            │
│  · DeepSpeed 并行策略                             │
│  · Checkpoint 定期保存                            │
│  · 训练日志和 TensorBoard                         │
└─────────────────────────────────────────────────┘
```

### 5.2 当前使用的框架

| 框架 | 版本 | 一句话介绍 | 适用场景 |
|------|------|-----------|---------|
| **ms-swift** | 3.5.0 | 阿里出品，Qwen 系列原生支持最好 | GPU + NPU 均支持 |
| **llama-factory** | 0.9.2 | 社区驱动，模型覆盖面广 | 仅 GPU |

> **你不需要自己安装框架**。训练平台已经把框架代码打包进了镜像中，选完训练配置后，框架信息只是展示给你看的，让你知道底层用了什么。

### 5.3 框架与模型的关系——Template

每个大模型有自己的**对话模板（Template）**，决定了输入文本如何拼接成模型能理解的格式。

以 Qwen2 的 ChatML 模板为例：

```
<|im_start|>system
你是一个有帮助的助手<|im_end|>
<|im_start|>user
杭州有什么好玩的？<|im_end|>
<|im_start|>assistant
杭州的著名景点有西湖...<|im_end|>
```

**为什么要关心 Template？** 因为如果模板不对，模型看到的数据格式就"不认识"，训练效果会很差。

好消息是：在训练平台上，**你选了模型类型之后，template 会自动匹配**，一般不需要你手动设置。

| 模型 | Template | 说明 |
|------|----------|------|
| Qwen 系列 | `qwen` / 自动识别 | 传空即可 |
| Qwen2.5 | `qwen2_5` | 传空也可以自动识别 |
| Qwen3 | `qwen3` | 传空也可以自动识别 |
| DeepSeek | `deepseek` | 需手动指定 |

## 第六章 训练数据集——最影响效果的东西

### 6.1 为什么数据集最重要

在大模型训练中有一句公认的话：**"Garbage in, garbage out"**（垃圾进，垃圾出）。

模型的能力上限由基座模型决定，但在这个上限内，**数据质量直接决定训练效果**。再好的模型、再贵的 GPU、再精细的超参数，都弥补不了差数据。

```
训练效果 =
  ┌──────────────────────────────────┐
  │ 基座模型能力（天花板）              │ ← 你选的模型
  │   ×                              │
  │ 数据质量（最关键的乘数）            │ ← 你准备的数据 ★★★
  │   ×                              │
  │ 超参数调优（微调）                 │ ← 你调的参数
  └──────────────────────────────────┘
```

### 6.2 SFT 数据集格式

#### 格式一：Alpaca 格式（单轮对话，推荐新手使用）

文件类型：`.jsonl`（每行一个 JSON 对象）

```json
{"instruction": "请将以下英文翻译成中文：Hello, how are you?", "input": "", "output": "你好，你好吗？"}
{"instruction": "写一首关于春天的七言绝句", "input": "", "output": "春风拂柳绿丝绦，细雨润花香满郊。燕子衔泥新巢筑，桃红杏白映溪桥。"}
{"instruction": "解释一下什么是机器学习", "input": "", "output": "机器学习是人工智能的一个子领域，它使计算机系统能够从数据中学习和改进，而无需进行明确编程..."}
```

| 字段 | 必填 | 说明 |
|------|------|------|
| `instruction` | **必填** | 用户的指令/问题 |
| `input` | 可选 | 附加输入内容（和 instruction 拼接） |
| `output` | **必填** | 期望的模型回答 |
| `system` | 可选 | 系统提示词（如"你是一个翻译助手"） |

#### 格式二：Messages 格式（多轮对话）

```json
{
    "messages": [
        {"role": "system", "content": "你是一个旅行顾问"},
        {"role": "user", "content": "我想去杭州玩三天，怎么安排？"},
        {"role": "assistant", "content": "建议行程：第一天西湖...第二天灵隐寺...第三天千岛湖..."},
        {"role": "user", "content": "第一天具体怎么安排？"},
        {"role": "assistant", "content": "上午9点从断桥出发..."}
    ]
}
```

#### 格式三：多模态格式（图片/视频理解）

```json
{
    "messages": [
        {"role": "user", "content": "<image>请描述这张图片的内容"},
        {"role": "assistant", "content": "图片中是一只橘色的猫坐在窗台上，阳光照在它身上..."}
    ],
    "images": ["/data/images/cat.jpg"]
}
```

### 6.3 数据集质量检查清单

在开始训练之前，用这个清单检查你的数据：

| 检查项 | 好的标准 | 常见问题 |
|-------|---------|---------|
| **格式正确** | 每行是合法的 JSON，字段名正确 | 多余逗号、缺少引号、字段名拼错 |
| **内容完整** | instruction 和 output 都不为空 | 空值、null、"无" |
| **质量一致** | output 的风格和质量一致 | 有的回答详细，有的一句话 |
| **无重复** | 无完全重复的数据 | copy-paste 导致大量重复 |
| **长度合理** | 不超过模型的 max_length | 超长文本会被截断，浪费数据 |
| **数量适当** | SFT 通常 500~50,000 条 | 太少学不到，太多可能过拟合 |

### 6.4 数据量参考

| 场景 | 推荐数据量 | 说明 |
|------|-----------|------|
| 快速验证 / 冒烟测试 | 50~100 条 | 只看训练流程是否通，不看效果 |
| 简单任务（分类/提取） | 500~2,000 条 | 任务格式统一，变化不大 |
| 通用对话微调 | 5,000~20,000 条 | 需要覆盖多种场景 |
| 复杂任务（代码/推理） | 10,000~50,000 条 | 需要更多样本教模型复杂推理 |

## 第七章 训练环境——GPU、镜像与基础设施

### 7.1 为什么需要 GPU

大模型训练的核心计算是**矩阵乘法**——数十亿参数同时参与运算。GPU（图形处理器）天生擅长大规模并行计算，比 CPU 快几十到几百倍。

| 硬件 | 适合的事 | 训练大模型 |
|------|---------|-----------|
| CPU | 串行逻辑运算（if/else） | 太慢，不实际 |
| **GPU** | 大规模并行矩阵运算 | **主力** |
| NPU | 华为昇腾专用加速卡 | 可用，框架适配中 |

### 7.2 常见 GPU 型号

| GPU 型号 | 显存 | 特点 | 适合 |
|---------|------|------|------|
| **A800-80G** | 80 GB | 当前主力训练卡 | 7B~72B 模型训练 |
| **H800-80G** | 80 GB | 更新一代，算力更强 | 同上，速度更快 |
| A100-40G | 40 GB | 显存较小 | 7B 以下或 LoRA |
| Ascend 910B | 64 GB | 华为 NPU | 国产化场景 |

### 7.3 显存（VRAM）——最关键的资源

显存决定了你能训练多大的模型。显存不够 = 训练直接报错（OOM）。

**粗略估算**（bf16 精度下）：

| 模型大小 | 全量微调显存需求 | LoRA 显存需求 |
|---------|---------------|-------------|
| 1.5B | ~16 GB | ~8 GB |
| 7B | ~80 GB（1×A800） | ~24 GB |
| 14B | ~160 GB（2×A800） | ~48 GB |
| 72B | ~640 GB（8×A800） | ~160 GB（2×A800） |

> 以上为粗略参考。实际显存取决于 batch size、sequence length、DeepSpeed 策略等。

### 7.4 什么是镜像（Image）

镜像就是一个**打包好的运行环境**——操作系统 + CUDA 驱动 + Python + 训练框架 + 所有依赖，全部装好、测试通过、打包成一个"快照"。

```
┌─────────────────────────────────┐
│          训练镜像                │
│  ┌───────────────────────────┐  │
│  │  Ubuntu 22.04             │  │ ← 操作系统
│  │  CUDA 12.1                │  │ ← GPU 驱动库
│  │  Python 3.10              │  │
│  │  PyTorch 2.3              │  │ ← 深度学习框架
│  │  ms-swift 3.5.0           │  │ ← 训练框架
│  │  Flash-Attention 2        │  │ ← 加速库
│  │  DeepSpeed                │  │ ← 分布式训练库
│  │  run_train.sh             │  │ ← 启动脚本
│  └───────────────────────────┘  │
└─────────────────────────────────┘
```

**为什么要用镜像？**

| 不用镜像 | 用镜像 |
|---------|-------|
| 自己装 CUDA、PyTorch、框架... | 一个镜像全搞定 |
| 不同机器环境不一致 | 所有机器跑同一个镜像，100% 一致 |
| "在我机器上能跑"经典问题 | 打包好的环境到处都能跑 |
| 环境坏了得从头装 | 重新拉镜像即可恢复 |

> **在训练平台上，你不需要关心镜像**。选完模型类型和训练方法后，平台自动匹配好镜像。

### 7.5 CUDA 是什么

CUDA 是 NVIDIA 提供的 GPU 编程框架。可以理解为 **GPU 的"驱动程序"**——没有它，PyTorch 无法调用 GPU 进行计算。

```
你的训练代码
    │
    ▼
PyTorch（深度学习框架）
    │
    ▼
CUDA（GPU 编程接口）
    │
    ▼
GPU 硬件（A800/H800）
```

常见 CUDA 版本：11.8、12.1、12.4。不同版本的 CUDA 和 PyTorch 版本需要匹配，这也是为什么要用镜像——**镜像里已经帮你配好了**。

## 第八章 超参数——你能调的"旋钮"

### 8.1 什么是超参数

超参数是训练开始前你需要设定的参数。它们不是模型学到的（那些叫参数/权重），而是你告诉训练过程"该怎么学"的配置。

```
类比：
  参数（模型学到的） = 学生的知识
  超参数（你设定的） = 教学方法
    · 每天学几个小时？      → epoch
    · 每次做几道题？        → batch_size
    · 犯错后改多少？        → learning_rate
    · 用什么学习策略？      → deepspeed
```

### 8.2 核心超参数详解

#### num_train_epochs（训练轮次）

**含义**：整个数据集过几遍。

```
epoch=1: 数据集从头到尾看 1 遍
epoch=3: 数据集从头到尾看 3 遍
epoch=5: 数据集从头到尾看 5 遍
```

| 设置 | 效果 |
|------|------|
| 太少（1） | 模型没学够，欠拟合 |
| **推荐（2~5）** | 充分学习，效果好 |
| 太多（>10） | 模型"死记硬背"，过拟合——训练数据上表现好，新数据上表现差 |

**默认值**：`3`

#### learning_rate（学习率）

**含义**：每一步参数调整的幅度。**最重要的超参数，没有之一。**

```
学习率大（1e-3）:   ●━━━━━━━━━━→  步子太大，来回震荡
学习率适中（1e-4）:  ●──→──→──→●   稳步逼近最优
学习率小（1e-6）:   ●→→           步子太小，训练太慢
```

| 训练方式 | 推荐学习率 | 说明 |
|---------|-----------|------|
| LoRA | **1e-4**（0.0001） | LoRA 参数少，可以用稍大的学习率 |
| 全量微调 | **1e-5**（0.00001） | 参数多，学习率太大会"灾难性遗忘" |

> **灾难性遗忘**：学习率过大导致模型把之前学到的知识全忘了——比如训练后模型不会说中文了。

#### per_device_train_batch_size（单卡 batch size）

**含义**：每张 GPU 每次处理多少条数据。

| 设置 | 效果 |
|------|------|
| 太小（1） | GPU 利用率低，训练慢 |
| **推荐（2~8）** | 平衡效率和显存 |
| 太大（>16） | 显存可能不够 → OOM |

**默认值**：LoRA 用 `4`，全量用 `2`

#### gradient_accumulation_steps（梯度累积步数）

**含义**：累积多少个 batch 的梯度后，再统一更新一次参数。

```
如果 batch_size=4, accumulation=4
那么"等效 batch size" = 4 × 4 = 16

好处：在显存有限时，通过累积模拟更大的 batch size
```

**默认值**：LoRA 用 `4`，全量用 `8`

#### max_length（最大序列长度）

**含义**：输入+输出的最大 token 数。超出的部分会被截断。

| 设置 | 显存占用 | 适用场景 |
|------|---------|---------|
| 2048 | 较少 | 短文本任务（分类、提取） |
| **4096** | 适中 | 通用对话（推荐默认） |
| 8192+ | 大 | 长文本任务（文档理解） |

**默认值**：`4096`

#### deepspeed（分布式策略）

**含义**：多卡训练时，如何在多张 GPU 之间分配模型和数据。

| 策略 | 原理 | 显存节省 | 适用 |
|------|------|---------|------|
| **ZeRO-2** | 分割优化器状态和梯度 | 中等 | **LoRA 推荐** |
| **ZeRO-3** | 分割优化器状态、梯度和参数 | 大 | **全量微调推荐** |

```
ZeRO-2（LoRA 推荐）:
  GPU 0: [完整模型] + [1/8 优化器]
  GPU 1: [完整模型] + [1/8 优化器]
  ...
  ↑ 每张卡都有完整模型，优化器切分

ZeRO-3（全量推荐）:
  GPU 0: [1/8 模型] + [1/8 优化器]
  GPU 1: [1/8 模型] + [1/8 优化器]
  ...
  ↑ 模型本身也切分到各卡上
```

### 8.3 LoRA 专属参数

如果你用 LoRA 微调，还需要了解两个参数：

#### lora_rank（LoRA 秩）

**含义**：LoRA "外挂"参数矩阵的大小。越大 → 参数越多 → 表达能力越强，但显存也越多。

| 设置 | 效果 |
|------|------|
| 8~16 | 参数极少，适合简单任务 |
| **64（推荐）** | 平衡效果和效率 |
| 128~256 | 接近全量微调效果，但显存增大 |

#### lora_alpha（LoRA 缩放因子）

**含义**：控制 LoRA 参数的影响力度。通常设为 rank 的 2 倍。

**推荐**：`lora_alpha = 2 × lora_rank`，即 rank=64 时 alpha=128

### 8.4 新手推荐配置速查表

| 参数 | LoRA（推荐新手） | 全量微调 |
|------|----------------|---------|
| train_type | `lora` | `full` |
| num_train_epochs | `3` | `3` |
| learning_rate | `1e-4` | `1e-5` |
| batch_size | `4` | `2` |
| gradient_accumulation | `4` | `8` |
| deepspeed | `zero2` | `zero3` |
| max_length | `4096` | `4096` |
| lora_rank | `64` | - |
| lora_alpha | `128` | - |
| dtype | `bf16` | `bf16` |

> 以上默认值已经在训练平台中预置，**你不改也完全可以直接跑**。

## 第九章 分布式训练——多机多卡协同

### 9.1 为什么需要多卡

一张 A800（80GB 显存）装不下一个 72B 的全量微调任务。怎么办？——**多张卡一起干**。

```
单机单卡:     [GPU 0] ← 小模型 or LoRA 能搞定
单机多卡:     [GPU 0][GPU 1]...[GPU 7] ← 一台机器 8 张卡
多机多卡:     机器A:[GPU×8] + 机器B:[GPU×8] ← 16 张卡一起
```

### 9.2 关键概念

| 概念 | 含义 | 类比 |
|------|------|------|
| **节点（Node）** | 一台物理机器 | 一间教室 |
| **World Size** | 参与训练的总 GPU 数 | 总学生人数 |
| **Node Rank** | 当前节点编号（从 0 开始） | 教室编号 |
| **Local Rank** | 当前节点内的 GPU 编号 | 教室内的座位号 |
| **MASTER** | 主节点，负责协调 | 班主任 |

### 9.3 你需要做什么

**几乎什么都不需要做。** 在训练平台上，你只需选择**节点数**：

- **1 节点**（默认）：单机 8 卡训练，适合大部分场景
- **2+ 节点**：多机训练，平台自动配置 `MASTER_ADDR`、`MASTER_PORT`、`WORLD_SIZE` 等

```
你做的：   选择节点数 = 2

平台做的：
  · 启动 2 台机器，每台 8 张 GPU
  · 自动注入 MASTER_ADDR=10.0.0.1
  · 自动注入 MASTER_PORT=29500
  · 自动注入 WORLD_SIZE=16
  · 机器 A: NODE_RANK=0
  · 机器 B: NODE_RANK=1
  · 通过 IB 网卡建立高速通信
```

### 9.4 IB 网卡

InfiniBand（IB）是一种**超高速网络**，多机训练时各节点间需要频繁交换梯度数据。普通以太网太慢，IB 网卡带宽可达 200~400 Gbps，是多机训练的标配。

> 平台默认每个节点分配 1 块 IB 网卡，你不需要配置。

## 第十章 训练监控与调试

### 10.1 Loss 曲线——最重要的监控指标

训练过程中最需要关注的就是 **loss 曲线**。

```
loss
  │
3.0├─●
  │   \
2.0├────●──
  │        \──●──
1.0├──────────────●──●──●──●
  │                          ── loss 趋于平稳
0.5├────────────────────────────●──●
  │
  └────────────────────────────────── step
    0    500   1000  1500  2000  2500
```

**如何判断训练正不正常？**

| Loss 表现 | 含义 | 怎么办 |
|-----------|------|-------|
| 持续下降，逐渐趋平 | **正常**，模型在学习 | 继续等 |
| 一开始下降，后面震荡 | 学习率可能偏大 | 降低 learning_rate |
| 完全不下降 | 学习率太小 or 数据有问题 | 检查数据格式 & 调大 lr |
| 直接爆炸（变成 NaN/Inf） | 学习率太大 or 数据异常 | 降低 lr，检查数据 |
| 下降后又上升 | 过拟合 | 减少 epoch，增加数据量 |

### 10.2 TensorBoard

TensorBoard 是当前版本使用的训练监控工具，可以可视化查看：

- **train/loss**：训练损失曲线
- **train/learning_rate**：学习率变化曲线
- **train/epoch**：当前训练到第几轮

训练日志自动写入 `{output_dir}/runs/` 目录，平台内嵌 TensorBoard 面板查看。

## 第十一章 训练产物——Checkpoint 的管理与使用

### 11.1 什么是 Checkpoint

Checkpoint（检查点）是训练过程中定期保存的模型快照。相当于游戏存档，方便：

- 训练中断后从上次存档继续
- 回退到之前某个版本
- 对比不同阶段的模型效果

### 11.2 Checkpoint 保存策略

| 参数 | 默认值 | 含义 |
|------|-------|------|
| `save_steps` | 500 | 每 500 步保存一次 |
| `save_total_limit` | 3 | 最多保留 3 个 checkpoint（旧的自动删除） |

```
训练目录结构:
/home/kas/kas_checkpoint/{username}/
  ├── checkpoint-500/        ← 第 500 步的存档（可能已被自动删除）
  ├── checkpoint-1000/       ← 第 1000 步的存档
  ├── checkpoint-1500/       ← 第 1500 步的存档
  ├── checkpoint-2000/       ← 第 2000 步的存档（最新）
  └── runs/                  ← TensorBoard 日志
       └── events.out.tfevents.xxx
```

### 11.3 全量微调 vs LoRA 的产物区别

| | 全量微调产物 | LoRA 产物 |
|---|---|---|
| **包含什么** | 完整模型所有参数 | 只有 LoRA adapter 参数 |
| **大小** | 和原模型一样大（7B ≈ 14GB） | 几十~几百 MB |
| **使用方式** | 直接加载就能推理 | 需要和基座模型一起加载 |
| **存储开销** | 大 | 小 |

### 11.4 训练完成后做什么

```
训练完成
    │
    ├─① 查看 loss 曲线 → 确认训练正常
    │
    ├─② 加载最终 checkpoint → 做推理测试
    │    · 用一些测试问题检查模型回答质量
    │    · 对比训练前后的效果
    │
    ├─③ 效果满意 → 部署上线
    │
    └─④ 效果不满意 → 分析原因
         · 数据质量？→ 清洗数据重新训练
         · 数据量不够？→ 扩充数据
         · 超参数不对？→ 调整 lr/epoch 重试
```

## 第十三章 常见问题排查手册

### 13.1 OOM（Out of Memory）—— 显存不足

**现象**：任务启动后不久就报错退出，日志中出现 `CUDA out of memory`。

**原因与解决**：

| 原因 | 解决方案 |
|------|---------|
| batch_size 设太大 | 降低 `per_device_train_batch_size`（如从 4 改成 2） |
| max_length 设太大 | 降低 `max_length`（如从 8192 改成 4096） |
| 全量微调大模型 | 换成 LoRA，或增加节点数 |
| 模型太大 | 换更小的模型（如 72B → 7B） |

### 13.2 Loss 不下降

**现象**：训练在跑，但 loss 始终不变或在高位震荡。

| 可能原因 | 排查方法 | 解决 |
|---------|---------|------|
| 学习率太小 | 查看日志里的 lr 值 | 增大 learning_rate |
| 数据格式不对 | 检查 jsonl 格式是否正确 | 修复数据格式 |
| 数据质量差 | 抽查几条数据看内容 | 清洗数据 |
| Template 不匹配 | 检查是否选了正确的模型类型 | 确认模型类型 |

### 13.3 Loss 变成 NaN 或 Inf

**现象**：训练中途 loss 突然变成 NaN 或无穷大。

| 解决方案 |
|---------|
| 降低学习率（如 1e-4 → 5e-5） |
| 检查数据中是否有异常值（空值、超长文本） |
| 确认使用了 bf16 精度（而不是 fp16，后者更容易溢出） |

### 13.4 训练突然中断

**现象**：任务运行一段时间后消失或状态变为 failed。

| 可能原因 | 排查 |
|---------|------|
| 内存不足（不是显存，是系统内存） | 查看日志有无 `Killed` 字样 |
| 弹性池资源被抢占 | 等待重新调度，或增加优先级 |
| 镜像/环境问题 | 查看日志中是否有 import 报错 |
| 磁盘空间不足 | checkpoint 太多占满了磁盘 |

### 13.5 训练完成但效果不好

| 症状 | 可能原因 | 优化方向 |
|------|---------|---------|
| 模型回答格式乱 | 数据格式不统一 | 统一数据中的 output 格式 |
| 模型只会复读 | 数据太少或太重复 | 增加数据多样性 |
| 模型"忘记"原有能力 | 学习率过大 / epoch 过多 | 降低 lr，减少 epoch |
| 特定场景不行 | 训练数据没覆盖该场景 | 补充对应场景的数据 |
