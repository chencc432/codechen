# 上篇：训练框架

## 一、训练框架全景图

### 1.1 什么是训练框架

训练框架是**介于底层算力（PyTorch/GPU）和业务需求之间的工具层**。它封装了数据处理、模型加载、训练循环、分布式策略、checkpoint 管理等通用能力，让你只需关注模型选择、数据准备和超参数调整。

```
┌──────────────────────────────────────────────────────────┐
│                      你的训练任务                          │
│  · 选模型 · 准备数据 · 调超参数                            │
└─────────────────────────┬────────────────────────────────┘
                          │
┌─────────────────────────▼────────────────────────────────┐
│                    训练框架（本文重点）                     │
│  ms-swift / LLaMA-Factory / Axolotl / torchtune ...      │
│  · 数据加载 & tokenize     · 多种微调方法                  │
│  · Template 管理           · 训练循环                      │
│  · Checkpoint 保存         · 分布式策略集成                 │
└─────────────────────────┬────────────────────────────────┘
                          │
┌─────────────────────────▼────────────────────────────────┐
│                    底层训练库                               │
│  HuggingFace Transformers / PEFT / TRL / DeepSpeed       │
└─────────────────────────┬────────────────────────────────┘
                          │
┌─────────────────────────▼────────────────────────────────┐
│                    深度学习框架                             │
│  PyTorch + CUDA/NCCL                                     │
└─────────────────────────┬────────────────────────────────┘
                          │
┌─────────────────────────▼────────────────────────────────┐
│                    硬件                                    │
│  GPU（A800/H800） / NPU（Ascend 910B）                    │
└──────────────────────────────────────────────────────────┘
```

### 1.2 主流框架一览

| 框架 | 开发方 | Star | 特点 | 适用设备 |
|------|-------|------|------|---------|
| **ms-swift** | 阿里 ModelScope | 8k+ | Qwen 原生支持，GPU+NPU | GPU, NPU |
| **LLaMA-Factory** | 社区 | 40k+ | WebUI、模型覆盖广 | GPU |
| **Axolotl** | 社区 | 8k+ | 配置灵活、YAML 驱动 | GPU |
| **torchtune** | Meta（PyTorch 官方） | 5k+ | 原生 PyTorch、轻量 | GPU |
| **Unsloth** | 社区 | 20k+ | 极致速度优化、低显存 | GPU |
| **xtuner** | 上海 AI Lab | 4k+ | InternLM 生态、多模态强 | GPU |
| **TRL** | HuggingFace | 10k+ | RLHF/DPO 专用 | GPU |

> 本文重点介绍 **ms-swift** 和 **LLaMA-Factory**，它们是当前平台集成的两个框架。

| Template | 模型系列 | 模态 | 说明 |
|----------|---------|------|------|
| `qwen` | Qwen 通用 | 文本 | 自动识别，传空即可 |
| `qwen2_5` | Qwen2.5 | 文本 | 可自动识别 |
| `qwen3` | Qwen3 | 文本 | 可自动识别 |
| `qwen2_vl` | Qwen2-VL | 文本+图片+视频 | 多模态视觉语言 |
| `qwen2_5_vl` | Qwen2.5-VL | 文本+图片+视频 | 多模态视觉语言 |
| `qwen2_audio` | Qwen2-Audio | 文本+音频 | 语音理解 |
| `deepseek` | DeepSeek V2/V3 | 文本 | |
| `deepseek_coder` | DeepSeek Coder | 文本（代码） | |
| `llama3` | LLaMA 3 / 3.1 | 文本 | Meta 系列 |
| `chatglm4` | GLM-4 | 文本 | 智谱系列 |
| `yi` | Yi | 文本 | 零一万物 |
| `internlm2` | InternLM 2 | 文本 | 书生系列 |

> 完整列表参考 [swift 文档 - 支持的模型](https://swift.readthedocs.io/zh-cn/latest/)。新模型发布后通常 1~2 周内完成 Template 适配。

### 2.4 支持的训练任务

| 任务类型 | swift 命令 | 说明 |
|---------|-----------|------|
| SFT 全量微调 | `swift sft --train_type full` | 更新全部参数 |
| SFT LoRA | `swift sft --train_type lora` | 仅训练 LoRA 参数 |
| SFT QLoRA | `swift sft --train_type lora --quant_bits 4` | 4bit 量化 + LoRA |
| DPO | `swift rlhf --rlhf_type dpo` | 直接偏好优化 |
| GRPO | `swift rlhf --rlhf_type grpo` | 分组相对策略优化 |
| PPO | `swift rlhf --rlhf_type ppo` | 近端策略优化 |
| SimPO | `swift rlhf --rlhf_type simpo` | 简化偏好优化 |
| ORPO | `swift rlhf --rlhf_type orpo` | 无需参考模型的偏好优化 |
| KTO | `swift rlhf --rlhf_type kto` | Kahneman-Tversky 优化 |
| CPT | `swift pt` | 继续预训练 |

### 2.5 命令行示例

#### SFT-LoRA 训练

```bash
swift sft \
  --model /path/to/qwen2-7b-instruct \
  --train_type lora \
  --dataset /path/to/train.jsonl \
  --output_dir /path/to/output \
  --num_train_epochs 3 \
  --learning_rate 1e-4 \
  --per_device_train_batch_size 4 \
  --gradient_accumulation_steps 4 \
  --deepspeed zero2 \
  --max_length 4096 \
  --lora_rank 64 \
  --lora_alpha 128 \
  --torch_dtype bf16
```

#### SFT-全量微调

```bash
swift sft \
  --model /path/to/qwen2-7b-instruct \
  --train_type full \
  --dataset /path/to/train.jsonl \
  --output_dir /path/to/output \
  --num_train_epochs 3 \
  --learning_rate 1e-5 \
  --per_device_train_batch_size 2 \
  --gradient_accumulation_steps 8 \
  --deepspeed zero3 \
  --max_length 4096 \
  --torch_dtype bf16
```

#### DPO 训练

```bash
swift rlhf \
  --rlhf_type dpo \
  --model /path/to/qwen2-7b-instruct \
  --train_type lora \
  --dataset /path/to/dpo_data.jsonl \
  --output_dir /path/to/output \
  --num_train_epochs 2 \
  --learning_rate 5e-5 \
  --deepspeed zero2 \
  --lora_rank 64
```

### 2.6 数据格式支持

| 格式名 | 适用任务 | 结构 |
|-------|---------|------|
| **alpaca** | SFT 单轮 | `{"instruction": "...", "input": "...", "output": "..."}` |
| **messages** | SFT 多轮 / 多模态 | `{"messages": [{"role": "user", "content": "..."},...]}` |
| **dpo** | DPO / ORPO | `{"messages": [...], "chosen": "...", "rejected": "..."}` |
| **kto** | KTO | `{"messages": [...], "label": true/false}` |
| **text** | CPT | `{"text": "一段连续文本..."}` |

### 2.7 版本演进

| 版本 | 重大变化 |
|------|---------|
| 2.x → 3.0 | 架构重构，命令行接口大幅变化，参数名调整 |
| 3.0 → 3.4 | 增加 GRPO/SimPO/ORPO 等 RL 方法 |
| 3.4 → 3.5 | 参数名微调（如部分环境变量名变更），性能优化 |

> **注意**：2.x 和 3.x 的命令行接口**不兼容**。如果参考网上教程，务必确认 swift 版本。

### 3.5 配置文件示例

LLaMA-Factory 推荐使用 **YAML 配置文件**（也支持命令行参数）：

#### SFT-LoRA（YAML）

```yaml
### model
model_name_or_path: /path/to/qwen2-7b-instruct
template: qwen

### method
stage: sft
do_train: true
finetuning_type: lora
lora_rank: 64
lora_alpha: 128
lora_target: all

### dataset
dataset: /path/to/train.jsonl
cutoff_len: 4096

### output
output_dir: /path/to/output
logging_steps: 10
save_steps: 500
save_total_limit: 3

### train
per_device_train_batch_size: 4
gradient_accumulation_steps: 4
num_train_epochs: 3.0
learning_rate: 1.0e-4
warmup_ratio: 0.05
bf16: true
deepspeed: examples/deepspeed/ds_z2_config.json

### eval
val_size: 0.05
per_device_eval_batch_size: 4
eval_strategy: steps
eval_steps: 500
```

#### 启动命令

```bash
llamafactory-cli train config.yaml
```

## 四、其他主流框架简介

### 4.1 Axolotl

| 项目 | 信息 |
|------|------|
| GitHub | https://github.com/axolotl-ai-cloud/axolotl |
| 特点 | 高度可配置，YAML 驱动，社区活跃 |
| 适合 | 需要精细控制训练流程的高级用户 |

**优势**：
- 配置文件非常灵活，几乎所有 HuggingFace 参数都可以设置
- 原生支持 Flash Attention、sample packing
- 支持多数据集混合训练（数据配比可调）

**示例配置片段**：

```yaml
base_model: /path/to/qwen2-7b-instruct
model_type: AutoModelForCausalLM
tokenizer_type: AutoTokenizer

adapter: lora
lora_r: 64
lora_alpha: 128
lora_target_linear: true

datasets:
  - path: /path/to/data.jsonl
    type: alpaca

sequence_len: 4096
sample_packing: true
pad_to_sequence_len: true

micro_batch_size: 4
gradient_accumulation_steps: 4
num_epochs: 3
learning_rate: 1e-4
optimizer: adamw_torch
bf16: true
deepspeed: deepspeed_configs/zero2.json
```

### 4.2 torchtune

| 项目 | 信息 |
|------|------|
| GitHub | https://github.com/pytorch/torchtune |
| 特点 | PyTorch 官方出品，原生纯净，无 HuggingFace 依赖 |
| 适合 | 追求简洁、可控性强的团队 |

**优势**：
- 代码极简，容易理解和定制
- 原生 PyTorch，不依赖 HuggingFace Transformers
- 内存优化好（原生支持 activation checkpointing）

**劣势**：
- 模型支持较少（主要 LLaMA/Mistral/Gemma/Qwen2）
- 功能不如 swift/llama-factory 丰富

### 4.3 Unsloth

| 项目 | 信息 |
|------|------|
| GitHub | https://github.com/unslothai/unsloth |
| 特点 | 速度极快（2~5 倍加速），显存占用极低 |
| 适合 | 资源有限（消费级 GPU）、追求速度 |

**优势**：
- 手写 Triton kernel 优化，LoRA 训练速度提升 2~5 倍
- 显存占用减少 50~70%
- 单张 3090/4090 就能跑 7B 模型的 LoRA

**劣势**：
- 仅支持 LoRA/QLoRA（不支持全量微调）
- 模型支持有限
- 多机分布式支持较弱

### 4.4 TRL（Transformer Reinforcement Learning）

| 项目 | 信息 |
|------|------|
| GitHub | https://github.com/huggingface/trl |
| 特点 | HuggingFace 官方，RLHF/DPO 的参考实现 |
| 适合 | RL 对齐任务、学术研究 |

**定位**：TRL 更多是一个**库**而非端到端框架。ms-swift 和 LLaMA-Factory 在 RL 部分底层都依赖了 TRL。

### 4.5 xtuner

| 项目 | 信息 |
|------|------|
| GitHub | https://github.com/InternLM/xtuner |
| 特点 | InternLM 生态，多模态训练能力强 |
| 适合 | InternLM 系列模型、多模态场景 |

## 七、SFT——监督微调

### 7.1 原理

SFT 的核心思想很简单：**给模型看"标准答案"，让它学会模仿**。

```
训练数据:  问题 Q → 答案 A（人工标注的标准答案）

训练过程:
  模型看到 Q → 预测出 A' → 对比 A' 和 A → 算出 loss → 调整参数
  重复数万次...
  最终模型学会：看到类似 Q 时，输出类似 A
```

### 7.2 全量微调 vs LoRA——本质区别

```
                    全量微调                          LoRA
              ┌─────────────────┐           ┌─────────────────┐
              │ 原始模型参数      │           │ 原始模型参数      │
              │ ████████████████│           │ ████████████████│
              │ ████████████████│           │ ████████████████│ ← 冻结
              │ ████████████████│           │ ████████████████│
              │ ^^^^^^^^^^^^^^^^│           │                 │
              │ 全部参与更新      │           │ + LoRA 参数      │
              └─────────────────┘           │   ▓▓▓▓          │ ← 只更新这些
                                            └─────────────────┘

参数量:       70 亿（7B 模型）                 约 4000 万（7B + rank64）
显存:         ~80 GB                          ~24 GB
训练速度:     较慢                             快 2~3 倍
效果:         理论上限更高                      略低，但大多数场景足够
产物大小:     ~14 GB                           ~100 MB
```

### 7.3 SFT 的数据格式

| 场景 | 格式 | 示例 |
|------|------|------|
| 单轮对话 | alpaca | `{"instruction": "翻译", "output": "Translation"}` |
| 多轮对话 | messages | `{"messages": [{"role":"user",...},{"role":"assistant",...}]}` |
| 多模态 | messages + images/videos | 在 messages 基础上加 images/videos 字段 |

### 7.4 SFT 的典型超参数

| 参数 | LoRA 推荐 | 全量推荐 | 为什么 |
|------|----------|---------|-------|
| learning_rate | `1e-4` | `1e-5` | 全量参数多，学习率要更保守 |
| batch_size | `4` | `2` | 全量显存占用更大 |
| deepspeed | `zero2` | `zero3` | 全量需要切分模型参数 |
| epochs | `2~5` | `2~3` | 全量更容易过拟合 |

### 7.5 什么时候用 SFT

| 场景 | 适合 SFT |
|------|---------|
| 让模型学会特定格式输出（JSON、表格等） | ✅ 非常适合 |
| 让模型掌握领域知识的问答 | ✅ 适合 |
| 让模型学会特定风格（客服口吻、学术口吻等） | ✅ 适合 |
| 让模型拒绝有害请求 | ⚠️ 可以，但 RL 更好 |
| 让模型回答更符合人类偏好 | ⚠️ SFT 做粗调，RL 做精调 |

## 八、LoRA 家族——参数高效微调

### 8.1 LoRA 原理

LoRA 的核心数学思想：大模型参数的更新矩阵是**低秩的**，不需要完整的矩阵来表示。

```
原始权重矩阵 W（d × d）
  更新 ΔW = B × A

其中：
  A 的大小: r × d    （r << d，r 就是 lora_rank）
  B 的大小: d × r

原本需要 d² 个参数
现在只需要 2 × d × r 个参数

例如 d=4096, r=64:
  原本: 4096 × 4096 = 16,777,216 参数
  LoRA: 2 × 4096 × 64 = 524,288 参数（仅 3%）
```

```
           输入 x
             │
             ▼
    ┌────────────────────┐
    │   原始权重 W        │ ← 冻结，不更新
    │   (d × d)          │
    └────────┬───────────┘
             │ W·x
             │
    ┌────────┴───────────┐
    │   + LoRA 增量       │
    │   B·A·x            │ ← 只训练 A 和 B
    │   (低秩矩阵)        │
    └────────┬───────────┘
             │
             ▼
           输出 y = W·x + B·A·x
```

### 8.2 LoRA 关键参数

| 参数 | 含义 | 推荐值 | 影响 |
|------|------|-------|------|
| `lora_rank` (r) | 低秩矩阵的秩 | `64` | 越大表达能力越强，参数量和显存越多 |
| `lora_alpha` (α) | 缩放因子 | `128`（2×rank） | 控制 LoRA 的影响力度 |
| `lora_target` | 应用到哪些层 | `all`（推荐） | q/k/v/o/gate/up/down 等 |
| `lora_dropout` | Dropout 比例 | `0.05` | 防止过拟合 |

> 经验法则：`alpha / rank` 的比值（通常设为 2）控制了 LoRA 的"学习强度"。

### 8.3 QLoRA —— 量化 + LoRA

QLoRA 在 LoRA 基础上，先把基座模型**量化到 4-bit**，大幅降低显存：

```
              LoRA                          QLoRA
    ┌──────────────────┐        ┌──────────────────┐
    │ 基座模型 (BF16)   │        │ 基座模型 (4-bit)  │ ← 量化后体积减少 75%
    │ ~14 GB (7B)      │        │ ~4 GB (7B)       │
    │ + LoRA 参数       │        │ + LoRA 参数       │
    │ ~100 MB          │        │ ~100 MB          │
    └──────────────────┘        └──────────────────┘
    总显存: ~24 GB               总显存: ~10 GB
```

| | LoRA | QLoRA |
|---|---|---|
| 基座精度 | BF16（16-bit） | NF4（4-bit） |
| 7B 显存 | ~24 GB | ~10 GB |
| 速度 | 更快 | 稍慢（量化反量化开销） |
| 效果 | 更好 | 略有损失（多数场景可接受） |
| 适合 | 有充足 GPU 资源 | 显存受限（如单卡 24G） |

### 8.4 DoRA —— LoRA 的改进版

DoRA（Weight-Decomposed Low-Rank Adaptation）将权重分解为**幅度（magnitude）和方向（direction）**两部分，分别处理：

```
标准 LoRA:     W' = W + B·A
DoRA:          W' = m · (W + B·A) / ||W + B·A||

m = 可学习的幅度向量
效果通常比标准 LoRA 更好，计算开销略增
```

### 8.5 AdaLoRA —— 自适应分配 rank

AdaLoRA 让不同层自动学习不同的 rank 值——重要的层分配更大的 rank，不重要的层减小 rank，在固定参数预算下获得更好效果。

### 8.6 LoRA 家族选型

| 方法 | 特点 | 推荐场景 |
|------|------|---------|
| **LoRA** | 经典、稳定、生态完善 | **默认选择，大多数场景** |
| **QLoRA** | 极低显存 | 显存不足时的备选方案 |
| DoRA | 效果更好，但较新 | 追求效果提升时尝试 |
| AdaLoRA | 自动分配 rank | 学术实验 |

## 九、DPO——直接偏好优化

### 9.1 为什么需要 DPO

SFT 教模型"正确答案是什么"，但有些问题没有唯一正确答案——只有"更好"和"更差"的回答。DPO 就是教模型**分辨好坏**。

```
SFT:   问题 → 标准答案（唯一）
DPO:   问题 → 好的回答 vs 差的回答（偏好对）
```

### 9.2 原理

DPO 的训练数据是**偏好对（preference pairs）**——同一个问题，提供一个好的回答（chosen）和一个差的回答（rejected）：

```
┌──────────────────────────────────────────┐
│  问题: "如何保持健康？"                    │
│                                          │
│  Chosen（好的回答）:                       │
│  "保持健康需要均衡饮食、规律运动和         │
│   充足睡眠。建议每天运动30分钟..."         │
│                                          │
│  Rejected（差的回答）:                     │
│  "多吃维生素就行了。"                      │
│                                          │
└──────────────────────────────────────────┘

DPO 的训练目标:
  增大 P(chosen) 的同时减小 P(rejected)
  让模型学会：遇到类似问题时，倾向于输出 chosen 风格的回答
```

### 9.3 数据格式

```json
{
    "messages": [
        {"role": "user", "content": "如何保持健康？"}
    ],
    "chosen": {
        "role": "assistant",
        "content": "保持健康需要均衡饮食、规律运动和充足睡眠..."
    },
    "rejected": {
        "role": "assistant",
        "content": "多吃维生素就行了。"
    }
}
```

### 9.4 DPO vs SFT

| | SFT | DPO |
|---|---|---|
| **数据** | 问答对（Q→A） | 偏好对（Q→chosen+rejected） |
| **目标** | 模仿标准答案 | 区分好坏回答 |
| **适用** | 教模型"做什么" | 教模型"什么更好" |
| **数据标注成本** | 较低 | 较高（需要成对标注） |
| **通常顺序** | 先做 SFT | 在 SFT 之后做 DPO |

### 9.5 典型超参数

| 参数 | 推荐值 | 说明 |
|------|-------|------|
| learning_rate | `5e-6` ~ `5e-5` | 通常比 SFT 的 lr 更小 |
| epochs | `1~3` | DPO 对过拟合很敏感 |
| beta | `0.1` | DPO 特有参数，控制与参考模型的偏离程度 |

## 十、GRPO / PPO——强化学习对齐

### 10.1 PPO（Proximal Policy Optimization）

PPO 是经典的 RLHF（Reinforcement Learning from Human Feedback）方法，流程最完整但也最复杂：

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  1. SFT 模型  │───→│ 2. 奖励模型   │───→│ 3. PPO 训练   │
│  (已微调好的)  │    │  (RM, 单独训练)│    │  (用 RM 打分)  │
└──────────────┘    └──────────────┘    └──────────────┘

三步流水线:
  Step 1: 先用 SFT 训练一个基础模型
  Step 2: 用偏好数据训练一个奖励模型（给回答打分）
  Step 3: 用 PPO 算法，让 SFT 模型的输出得到更高的奖励分
```

**PPO 的复杂性**：
- 需要同时维护 4 个模型（Policy / Reference / Reward / Critic）
- 训练不稳定，超参数敏感
- 显存需求大（4 个模型同时在 GPU 上）

### 10.2 GRPO（Group Relative Policy Optimization）

GRPO 是 DeepSeek 提出的简化版 RL 方法，**不需要单独训练奖励模型**：

```
┌─────────────────────────────────────────────────┐
│                    GRPO 流程                      │
│                                                  │
│  1. 对一个问题，让模型生成 N 个回答（一组）        │
│  2. 用规则或 AI 对这 N 个回答打分                  │
│  3. 组内排名：好的回答得正奖励，差的得负奖励       │
│  4. 根据奖励更新模型                              │
│                                                  │
│  核心优势：                                       │
│  · 不需要训练奖励模型（省一步 + 省显存）           │
│  · 组内相对排名，奖励信号更稳定                   │
│  · 比 PPO 更容易训练                              │
└─────────────────────────────────────────────────┘
```

```
问题: "1+1等于几？"

模型生成一组回答:
  回答 1: "1+1=2"         → 正确 → 正奖励 +1
  回答 2: "1+1=3"         → 错误 → 负奖励 -1
  回答 3: "1+1=2，因为..." → 正确且有推理 → 正奖励 +2
  回答 4: "不知道"         → 无用 → 负奖励 -2

模型学到：应该像回答 3 那样，给出正确答案并带推理过程
```

### 10.3 PPO vs DPO vs GRPO

| | PPO | DPO | GRPO |
|---|---|---|---|
| **需要奖励模型** | ✅ 需要单独训练 | ❌ 不需要 | ❌ 不需要 |
| **需要偏好数据** | 训练 RM 时需要 | ✅ 需要 | ❌ 不需要（用规则打分） |
| **训练复杂度** | 最高（4 个模型） | 低（1~2 个模型） | 中（1 个模型 + 采样） |
| **训练稳定性** | 较差 | 好 | 较好 |
| **显存需求** | 最大 | 中等 | 中等 |
| **效果** | 理论上限最高 | 好 | 好（数学/代码任务突出） |
| **适用场景** | 需要极致对齐 | 有偏好数据时 | 有明确评价规则时 |
| **平台支持** | 🔜 后续 | 🔜 后续 | 🔜 后续 |

## 十一、CPT——继续预训练

### 11.1 什么是 CPT

CPT（Continual Pre-Training）是在已有的 Base 模型基础上，用**大量领域文本**继续预训练，让模型掌握领域知识。

```
          通用 Base 模型                  领域增强模型
        ┌──────────────┐              ┌──────────────┐
        │ 知道很多通用知识│              │ 通用知识 +     │
        │ 但不懂你的领域 │  ── CPT ──→  │ 领域专业知识   │
        └──────────────┘              └──────────────┘

        "什么是民法典？"               "民法典是2020年通过的
         → 可能回答模糊                  中华人民共和国法律..."
                                       → 准确且专业
```

### 11.2 CPT vs SFT

| | CPT | SFT |
|---|---|---|
| **目的** | 灌入领域知识 | 学会执行指令 |
| **数据格式** | 纯文本（无需问答对） | 问答对（instruction + output） |
| **数据量** | 大（GB~TB 级） | 小（千~万条） |
| **输入模型** | Base 模型 | Base 或 Instruct 模型 |
| **典型流程** | Base → CPT → SFT → RL | Instruct → SFT |

### 11.3 数据格式

CPT 的数据最简单——纯文本：

```json
{"text": "民法典是中华人民共和国的一部法律，于2020年5月28日由第十三届全国人民代表大会第三次会议通过..."}
{"text": "合同法规定，当事人订立合同，采取要约、承诺方式..."}
{"text": "物权法是规范财产关系的民事基本法律..."}
```

### 11.4 什么时候需要 CPT

| 场景 | 需要 CPT？ |
|------|-----------|
| 通用对话微调 | ❌ 直接 SFT 即可 |
| 模型完全不懂你的领域 | ✅ 先 CPT 再 SFT |
| 需要模型掌握大量专业术语和知识 | ✅ CPT |
| 有大量未标注的领域文档 | ✅ CPT 可以利用无标注数据 |

## 十二、KTO / ORPO——新兴对齐方法

### 12.1 KTO（Kahneman-Tversky Optimization）

KTO 简化了 DPO 的数据要求——**不需要成对的偏好数据**，只需要知道一个回答"好不好"：

```
DPO 数据:  问题 → (好的回答, 差的回答)    ← 需要成对
KTO 数据:  问题 → 回答 → 好/差（二元标签）  ← 只需单个标签

KTO 数据示例:
{"messages": [{"role": "user", "content": "..."}], "response": "...", "label": true}
{"messages": [{"role": "user", "content": "..."}], "response": "...", "label": false}
```

**优势**：数据标注成本比 DPO 低得多——不需要对同一个问题准备两个回答再比较。

### 12.2 ORPO（Odds Ratio Preference Optimization）

ORPO 把 SFT 和偏好优化**合并成一步**——在 SFT 的 loss 中直接加入偏好优化项：

```
传统流程:  SFT → DPO（两步）
ORPO:      SFT+偏好 一步完成

优势：
  · 训练效率高（少跑一轮）
  · 不需要参考模型（Reference Model），省显存
```

### 12.3 SimPO（Simple Preference Optimization）

SimPO 是 DPO 的简化版，去掉了参考模型（Reference Model），用模型自身的长度归一化概率作为隐式奖励：

```
DPO:    需要 Policy Model + Reference Model
SimPO:  只需 Policy Model（省一半显存）
```

### 12.4 新兴方法选型

| 方法 | 数据要求 | 是否需要参考模型 | 训练复杂度 | 效果 |
|------|---------|----------------|-----------|------|
| DPO | 偏好对 | 需要 | 低 | 好 |
| KTO | 二元标签 | 需要 | 低 | 接近 DPO |
| ORPO | 偏好对 | 不需要 | 最低（和 SFT 合并） | 好 |
| SimPO | 偏好对 | 不需要 | 低 | 好 |
| GRPO | 不需要偏好数据 | 不需要 | 中（需要采样） | 好 |

## 十三、任务类型选型指南

### 13.1 决策流程图

```
你想让模型做什么？
    │
    ├── 让模型"学会做某件事"（格式/风格/知识）
    │   │
    │   ├── 有问答对数据
    │   │   └──→ SFT（先从 LoRA 开始）
    │   │
    │   └── 只有大量无标注文档
    │       └──→ CPT → 再 SFT
    │
    ├── 让模型"回答更好"（质量/安全/偏好）
    │   │
    │   ├── 有成对偏好数据（chosen + rejected）
    │   │   └──→ DPO
    │   │
    │   ├── 只有好/差二元标签
    │   │   └──→ KTO
    │   │
    │   └── 有明确的评价规则（如答案正确性）
    │       └──→ GRPO
    │
    └── 不确定
        └──→ 先 SFT，效果不够再加 DPO/GRPO
```

### 13.2 典型训练流水线

#### 场景 A：最简单（大多数业务场景）

```
Instruct 模型 ──→ SFT (LoRA) ──→ 部署
                    │
                    用业务问答对数据
```

#### 场景 B：追求更高质量

```
Instruct 模型 ──→ SFT (LoRA) ──→ DPO ──→ 部署
                    │                │
                    问答对数据        偏好对数据
```

#### 场景 C：领域知识增强

```
Base 模型 ──→ CPT ──→ SFT ──→ 部署
               │        │
               领域文档    问答对数据
```

#### 场景 D：数学/代码等有明确评价标准的任务

```
Instruct 模型 ──→ SFT ──→ GRPO ──→ 部署
                    │        │
                    问答数据   规则打分（如答案正确性）
```

### 13.3 综合对照表

| 任务类型 | 数据格式 | 数据量 | 资源需求 | 难度 | 效果提升维度 | 平台支持 |
|---------|---------|-------|---------|------|------------|---------|
| **SFT-LoRA** | 问答对 jsonl | 500~50K | 低（1~2 卡） | 入门 | 格式/风格/知识 | ✅ |
| **SFT-全量** | 问答对 jsonl | 1K~50K | 高（8 卡起） | 入门 | 格式/风格/知识（更彻底） | ✅ |
| **DPO** | 偏好对 jsonl | 1K~20K | 中 | 中级 | 回答质量/安全性 | 🔜 |
| **GRPO** | 问题 + 评价规则 | 1K~10K | 中 | 中级 | 推理/正确性 | 🔜 |
| **PPO** | 偏好对 + RM | 1K~20K | 最高 | 高级 | 全面对齐 | 🔜 |
| **KTO** | 二元标签 | 1K~20K | 中 | 中级 | 回答质量 | 🔜 |
| **ORPO** | 偏好对 jsonl | 1K~20K | 低（和 SFT 合并） | 中级 | 回答质量 | 🔜 |
| **CPT** | 纯文本 | GB~TB | 高 | 中级 | 领域知识 | 视需求 |

### 13.4 新手推荐路径

```
第 1 周:  SFT-LoRA
          · 用默认参数跑通
          · 学会看 loss 曲线
          · 检查训练产物

第 2~3 周: SFT 调参
          · 调 lr / epoch / rank
          · 对比不同数据量的效果
          · 尝试全量微调

第 4 周+:  进阶
          · 尝试 DPO
          · 了解 GRPO
          · 探索多模态训练
```

### A. 各框架训练命令速查

| 框架 | SFT-LoRA | SFT-全量 | DPO |
|------|----------|---------|-----|
| **ms-swift** | `swift sft --train_type lora` | `swift sft --train_type full` | `swift rlhf --rlhf_type dpo` |
| **LLaMA-Factory** | `llamafactory-cli train --stage sft --finetuning_type lora` | `llamafactory-cli train --stage sft --finetuning_type full` | `llamafactory-cli train --stage dpo` |
| **Axolotl** | `accelerate launch -m axolotl.cli.train config.yaml` | 同上，YAML 中 adapter 设为 null | 同上，YAML 中配置 rl: dpo |

### B. 数据格式速查

| 任务 | 格式 | 关键字段 |
|------|------|---------|
| SFT 单轮 | alpaca | `instruction`, `output` |
| SFT 多轮 | messages | `messages[{role, content}]` |
| SFT 多模态 | messages + media | `messages` + `images`/`videos`/`audios` |
| DPO | preference | `messages` + `chosen` + `rejected` |
| KTO | binary | `messages` + `response` + `label` |
| CPT | text | `text` |
