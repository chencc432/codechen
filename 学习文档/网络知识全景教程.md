# 网络知识全景教程

## 前言

这不是一份“背概念”的网络笔记，而是一份面向真实生产环境的网络教程。

很多人学网络，停留在这些关键词上：

- OSI 七层模型
- TCP 三次握手
- VLAN
- 路由器
- 防火墙
- 负载均衡

这些知识当然重要，但如果只知道这些碎片，到了真实生产环境还是会发懵。因为生产中的网络从来不是几个单独的协议名词，而是一整套连续的系统：

用户请求从哪里来，怎么解析域名，怎么选路，怎么到默认网关，怎么经过交换机和路由器，在哪里做 NAT，在哪里做安全策略，在哪里被负载均衡，怎么进入 Kubernetes，怎么命中 Pod，怎么走回包路径，AI 训练时为什么 GPU 明明很贵却在等网络，AI 推理时为什么入口网关和模型路由才是瓶颈。

这份教程的目标，就是把这些知识全部串起来。

你会看到的不是“课本式概念堆砌”，而是：

- 从主机到交换机再到路由器的完整视角
- 从园区网、数据中心到云和 Kubernetes 的统一理解
- 从网关、NAT、负载均衡到服务治理的边界划分
- 从传统业务流量到 AI 训练和推理流量的差异
- 从理论原理到生产设计、排障和演进方法

如果你能真正读懂这份教程，你对网络的理解会从“会几个术语”升级为“能看懂一张生产网络拓扑，能理解流量怎么走，能知道问题大概会出在哪一层”。

## 第一章：先建立正确的网络心智模型

### 1.1 网络到底解决什么问题

网络的本质不是“把两台机器连起来”，而是让分布在不同位置、不同系统、不同网络域中的计算资源，能够以可控、可扩展、可观测、可治理的方式互相通信。

说得更具体一点，网络主要解决这几类问题：

1. 让数据从源头到达目标。
2. 让数据在规模变大后仍然可控。
3. 让不同部门、不同租户、不同业务之间可以隔离。
4. 让流量在高峰期也尽量稳定。
5. 让故障发生时能够快速定位和收敛影响。
6. 让系统在扩容、迁移、容灾时仍然可演进。

所以网络从来不只是“连通性”问题，它同时还是：

- 边界问题
- 性能问题
- 安全问题
- 容量问题
- 运维问题

### 1.2 为什么课本学完还是不会看生产网络

因为课本通常是“分模块讲知识”，而生产环境需要的是“链路思维”。

举个最常见的访问场景：

用户在浏览器里输入 `https://api.example.com`，点击回车之后，发生的事情不是只有 TCP 三次握手。

真实路径可能是：

```text
浏览器
-> 本地 DNS 缓存 / 本地递归 DNS
-> 权威 DNS / GSLB
-> 得到某个公网 VIP
-> 本机路由表判断下一跳
-> 默认网关
-> 接入交换机 / 汇聚 / 出口路由
-> 防火墙 / NAT / Anti-DDoS
-> CDN / WAF / 四层或七层负载均衡
-> 入口网关 / Ingress
-> Kubernetes Service
-> Pod
-> 应用进程
-> 数据库 / 缓存 / 其他微服务
```

这就是为什么只会几个协议名词，到了生产里不够用。你需要的不只是记忆，而是路径还原能力。

### 1.3 生产网络的四个面

理解网络时，至少要区分四个“面”：

1. 控制面  
决定“该怎么转发”，比如路由协议、ARP 学习、BGP 路由发布、EVPN 控制平面。

2. 数据面  
真正负责“把包转出去”，比如交换芯片查表、路由查 FIB、NAT 会话匹配、负载均衡转发。

3. 管理面  
运维人员登录设备、下发配置、采集监控、查看日志、做变更的那一面。

4. 策略面  
负责“能不能过、优先级如何、需要走哪条路径”，比如 ACL、QoS、Security Group、NetworkPolicy、WAF 规则。

很多人说“网络坏了”，其实要先问清：坏的是哪一个面。

## 第二章：把一次请求完整走一遍

### 2.1 一次访问为什么是学习网络的最好入口

学网络最有效的方法，不是先背协议，而是先完整跟踪一条请求路径。

假设一台办公电脑要访问公司 Kubernetes 集群里的一个业务系统：

```text
员工电脑
-> 办公网 DNS
-> 出口网关
-> 企业防火墙
-> 云上负载均衡
-> Ingress Controller
-> Service
-> Pod
-> 应用进程
-> 下游数据库
```

这一条路径几乎把网络里的大多数关键知识都串上了：

- DNS 决定请求先去哪里
- 主机路由决定第一跳是谁
- 网关负责跨网段转发
- 防火墙决定策略是否允许
- NAT 决定地址如何转换
- 负载均衡决定后端实例怎么选
- Kubernetes 网络决定 Pod 怎么收到流量
- 回包路径决定会话能否完整闭环

### 2.2 为什么回包路径和去包路径一样重要

很多初学者只看“请求发出去了没”，但生产里更致命的是“回包能不能正确回来”。

例如：

- 出口 NAT 做在 A 防火墙上，请求从 A 走出去
- 回包却因为路由策略从 B 防火墙回来
- B 上没有对应 NAT 会话状态
- 结果请求看起来是“偶发超时”

这类问题本质不是“路由不通”，而是状态和路径不对称。

所以你以后看网络，一定要同时看：

- 去程路径
- 回程路径
- 状态在哪保存
- NAT 在哪发生
- 安全策略在哪生效

## 第三章：主机网络基础，很多故障的起点都在这里

### 3.1 网卡、MAC、IP 到底是什么关系

每台主机连上网络，至少要解决三件事：

1. 我是谁  
这对应主机的 IP 地址，以及二层上的 MAC 地址。

2. 对方是谁  
这对应 DNS 解析、ARP 解析、邻居发现等机制。

3. 我要怎么过去  
这对应路由表和默认网关。

MAC 地址主要在二层网络里使用，IP 地址主要在三层网络里使用。不要把它们混成一回事。

### 3.2 ARP 为什么这么基础却经常出问题

在 IPv4 环境里，主机要向同一网段的目标发包时，需要先知道目标的 MAC 地址；如果目标不在本地网段，就需要知道默认网关的 MAC 地址。ARP 就是在做这个映射。

因此很多“网络不通”，第一现场其实不是路由协议，而是 ARP：

- ARP 没学到
- ARP 老化异常
- 网关 MAC 漂移
- 二层环路导致 ARP 混乱
- 安全策略拦截了 ARP 欺骗后正常业务也受影响

### 3.3 默认网关到底是什么

默认网关不是“一个神秘配置项”，它本质上是主机访问非本地网段时的第一跳三层设备。

主机的基本判断逻辑是：

1. 目标 IP 是否在本地子网里？
2. 如果在本地，就直接 ARP 对方。
3. 如果不在本地，就 ARP 默认网关，把包交给默认网关。

这就是为什么默认网关配置错了，会出现“同网段能通、跨网段不通”的典型现象。

### 3.4 路由表是主机的交通规则

主机并不是“只会走默认网关”，它也有自己的路由表。

核心规则是：最长前缀匹配。

例如：

- `10.0.0.0/24` 走网关 A
- `10.0.0.0/16` 走网关 B
- `0.0.0.0/0` 走默认网关 C

当目标是 `10.0.0.8` 时，会优先匹配 `/24`，而不是 `/16` 或默认路由。

这条规则在主机、路由器、云路由表、Kubernetes 节点上都非常重要。

### 3.5 DNS 不只是“把域名翻译成 IP”

DNS 是整个系统的控制面入口之一。它不承载业务流量，但它决定业务流量去哪里。

现实中的 DNS 影响远比想象中大：

- 应用访问慢，可能是 DNS 解析慢
- 切流失败，可能是 TTL 太长
- 多活流量错位，可能是 GSLB 策略不合理
- Kubernetes 服务发现异常，可能是 CoreDNS 或上游转发有问题

所以以后排障时，不要只看 IP 和端口，一定先问一句：名字解析是否正常。

### 3.6 TCP 为什么是生产网络的核心协议

TCP 真正重要的地方不只是“三次握手”这四个字，而是它承担了可靠传输、流量控制、拥塞控制、重传等机制。

在生产环境里，TCP 常见关注点包括：

- 建连耗时
- 重传率
- 窗口大小
- RTT
- 拥塞控制算法
- 长连接保活
- TIME_WAIT / CLOSE_WAIT 状态异常

很多“应用卡顿”，并不是链路彻底断了，而是 TCP 表现很差。

### 3.7 TLS 和 QUIC 为什么改变了网络行为

现代应用几乎都会加密，因此不能只学明文 HTTP。

你需要理解：

- TLS 终止点在哪里
- 证书和 SNI 如何影响七层转发
- HTTPS 下抓包为什么看不到业务内容
- QUIC 基于 UDP，但提供类似 TCP 的可靠机制
- HTTP/3 会改变传统四层设备对流量的可见性

这会直接影响：

- 负载均衡设计
- 安全检测位置
- 性能优化方式
- 可观测性能力

## 第四章：二层网络，为什么今天依然绕不过去

### 4.1 交换机到底在做什么

交换机主要根据 MAC 地址表转发二层帧。它的核心行为是：

1. 学习源 MAC 地址来自哪个端口。
2. 查询目标 MAC 是否已知。
3. 已知就单播转发，未知就泛洪。

这意味着，二层网络天然会面临三个问题：

- 广播
- 未知单播泛洪
- 环路

### 4.2 VLAN 的真正价值

VLAN 的本质是把一个大的二层广播域逻辑切分成多个小广播域。

它的主要价值是：

- 限制广播范围
- 做基础隔离
- 便于按部门、业务、租户分段

但你必须记住：VLAN 不是完整安全边界。

它只能解决“二层隔离”和广播域划分的问题，不能替代真正的访问控制、身份认证和安全策略。

### 4.3 Trunk 和 Access 是什么关系

Access 口通常承载一个 VLAN，适合终端接入。

Trunk 口通常承载多个 VLAN，适合交换机之间、交换机与虚拟化主机之间的链路。

很多现场故障就卡在这里：

- 业务 VLAN 没放通
- Native VLAN 不一致
- 交换机与虚拟化平台对 VLAN Tag 的理解不一致
- 容器宿主机桥接配置错误

### 4.4 二层环路为什么杀伤力巨大

二层不像三层那样天然靠 TTL 防止无限转发，所以一旦形成环路，会出现：

- 广播风暴
- MAC 地址表抖动
- CPU 飙高
- 局部网络瞬间雪崩

这也是 STP、RSTP、MSTP 至今仍然重要的原因：它们不是为了“让网络更快”，而是为了防止二层拓扑失控。

### 4.5 为什么大规模网络不适合无限扩展二层

因为二层域越大，广播越多、故障域越大、排障越难、环路风险越高、MAC 学习压力越大。

所以生产网络的基本趋势是：

- 小二层
- 大三层
- 边界清晰
- 故障域可控

## 第五章：三层网络，网络规模化的真正起点

### 5.1 路由器和三层交换机的本质

它们本质上都在做三层转发，只是应用场景和性能模型可能不同。

一个三层设备收到包后，核心判断过程是：

1. 查目标 IP。
2. 做最长前缀匹配。
3. 找到下一跳和出接口。
4. 若需要，继续解析下一跳 MAC。
5. 重新封装二层头并转发出去。

### 5.2 静态路由、默认路由和动态路由的边界

静态路由适合：

- 小规模环境
- 固定路径
- 出口缺省路由
- 特定业务强制走某链路

动态路由适合：

- 网络规模较大
- 路径需要自动收敛
- 多设备多区域拓扑
- 多链路冗余

默认路由适合：

- 指向未知目的地的统一出口
- 办公网、服务器默认出网
- 分支机构统一回总部

### 5.3 ECMP 和策略路由的生产意义

ECMP 是等价多路径，适用于多条代价相同的路径同时转发流量。

它的价值在于：

- 提高链路利用率
- 提供一定冗余
- 适合 Spine-Leaf 等多路径网络

策略路由 PBR 则不是按“目标网段”做决定，而是可以按源地址、协议、端口等条件选路。

它适合：

- 让特定业务走专线
- 让备份流量走低优先级出口
- 多出口按业务分流

但 PBR 也会增加排障复杂度，尤其在回包路径不对称时更明显。

### 5.4 为什么三层更适合做边界

因为三层天然适合：

- 控制广播域
- 清晰定义路由边界
- 做 ACL 和策略
- 缩小故障域
- 便于冗余和收敛

这就是为什么越是生产网络，越强调“边界三层化”。

## 第六章：动态路由，网络扩大以后必须依赖协议

### 6.1 OSPF 的生产价值

OSPF 不是只为了考试，它在企业内网、数据中心 Underlay、传统园区网里都非常常见。

它的核心价值是：

- 自动发现邻居
- 通过链路状态数据库理解拓扑
- 计算最短路径
- 在链路变化时自动收敛

真实生产里你更应该关注这些问题：

- 区域划分是否合理
- 是否有过多 LSA 泛洪
- 是否出现邻居抖动
- 汇总是否正确
- 路由黑洞是否因为汇总不当产生

### 6.2 BGP 为什么这么重要

BGP 是互联网的骨架协议，也是现代大规模网络最常见的策略型路由协议之一。

它擅长的不是“最快收敛”，而是：

- 大规模路由分发
- 灵活策略控制
- 跨自治系统协同
- 对多出口、多上游、多租户环境的良好适配

在生产里，BGP 不只用于互联网出口，还常用于：

- IDC Fabric Underlay
- EVPN 控制平面
- 云专线互联
- 大规模机房间互联

### 6.3 学 BGP 必须学会看属性

很多 BGP 故障不是邻居没起来，而是属性不对。

最常见的几个属性：

- `AS-PATH`
- `Local Preference`
- `MED`
- `Next-Hop`
- `Community`

很多“路由有了但流量不走”或者“明明发布了却不生效”，问题都在这些属性上。

## 第七章：网关，网络世界里最容易被误解的词

### 7.1 为什么“网关”不能只理解成路由器

在不同语境里，“网关”代表完全不同的角色。

最常见的几类网关有：

1. 默认网关  
主机跨网段时的第一跳。

2. VLAN 网关 / SVI 网关  
交换机上某个 VLAN 的三层接口。

3. 冗余网关  
VRRP、HSRP 这类提供虚拟网关 IP 的机制。

4. NAT 网关  
负责私网到公网的地址转换。

5. 应用网关  
API Gateway、Ingress Gateway、服务入口控制点。

6. 安全网关  
防火墙、代理、零信任访问入口。

### 7.2 生产设计里为什么网关一定要分层

因为不同网关承担的职责不同：

- 默认网关解决第一跳转发
- 冗余网关解决高可用
- NAT 网关解决地址转换
- 入口网关解决七层治理
- 安全网关解决访问控制

如果把这些职责全堆在同一层，最终会导致：

- 架构混乱
- 状态耦合
- 性能瓶颈集中
- 排障困难

### 7.3 冗余网关不是万能高可用

VRRP/HSRP 这类网关冗余解决的是“第一跳不要单点”，但它不自动解决：

- 上行链路故障
- 上游路由黑洞
- 防火墙状态漂移
- NAT 会话一致性

所以做高可用时，不能以为有了虚拟网关就万事大吉。

## 第八章：NAT，生产网络里最常见的复杂度来源

### 8.1 NAT 到底做了什么

NAT 的核心是改地址，常见形式包括：

- SNAT：改源地址
- DNAT：改目标地址
- PAT：多个连接复用不同端口做地址复用

NAT 常见于：

- 私网主机统一出网
- 公网服务映射到内网
- 多租户地址复用
- 云上 NAT Gateway

### 8.2 为什么 NAT 一定要结合状态来看

现代 NAT 几乎总是跟连接跟踪状态绑定的。

这意味着你必须理解：

- 会话从哪里建立
- 状态保存在哪台设备
- 回包是否经过同一台设备
- 端口是否耗尽
- 会话超时是否合理

很多“偶发超时”“连接数打满”“长连接中断”不是应用自己坏了，而是 NAT 状态资源吃满了。

### 8.3 NAT 的价值和副作用

它的价值包括：

- 节约公网地址
- 隐藏内网地址结构
- 统一出网
- 便于集中控制

它的副作用包括：

- 排障复杂
- 非对称路径敏感
- 某些协议兼容性差
- 会话状态成为瓶颈
- 端口资源可能耗尽

所以 NAT 不是“越多越安全”，而是“越多越复杂”。

## 第九章：负载均衡，生产架构中的核心流量分发层

### 9.1 为什么负载均衡不是“平均分配请求”

真正的负载均衡要解决的不是平均，而是“按业务目标做正确分发”。

它要同时考虑：

- 后端健康状态
- 连接数
- 会话粘性
- 证书终止
- URL 或域名路由
- 限流与熔断
- 灰度发布
- 可观测性

因此负载均衡既是转发设备，也是控制点。

### 9.2 L4 和 L7 分别适合什么

L4 负载均衡主要看 IP 和端口，优点是：

- 性能高
- 转发快
- 适合通用 TCP/UDP
- 对协议内容不敏感

L7 负载均衡主要看 HTTP/TLS/应用头部，优点是：

- 可按域名、路径、Header、Cookie 路由
- 更适合微服务和 Web 业务
- 便于做认证、限流、重试、灰度

缺点是：

- 开销更大
- 配置更复杂
- 对协议理解更深

### 9.3 负载均衡在现代架构里的多重角色

一个现代互联网请求，可能依次经过：

```text
DNS / GSLB
-> CDN
-> WAF
-> 公网 SLB
-> Ingress Gateway
-> Service Mesh Gateway
-> 服务实例
```

这说明“负载均衡”并不是一台单独设备，而是一整串分层的流量入口体系。

### 9.4 会话保持什么时候需要，什么时候反而有害

以下场景常需要会话保持：

- Session 在单机内存里
- WebSocket 长连接
- 某些推理会话需要保持上下文

以下场景则更倾向去会话化：

- 微服务无状态
- 自动扩缩容频繁
- 灰度和故障切换要求高
- 需要提升弹性调度能力

所以会话保持不是默认答案，而是业务设计选择。

## 第十章：DNS，不承载业务，但支配业务入口

### 10.1 DNS 是整个系统的方向盘

很多人把 DNS 当成“配个域名而已”，但真实生产里，DNS 决定：

- 用户第一次会连到哪里
- 就近接入是否生效
- 多活切流是否及时
- 服务发现是否可靠
- 故障切换是否平滑

### 10.2 递归解析和权威解析要分开理解

递归 DNS 帮客户端去问，最后把答案拿回来。

权威 DNS 负责告诉世界：这个域名到底该指向谁。

生产里常见问题包括：

- TTL 太长，切流不及时
- CNAME 链过长，解析耗时增加
- 本地缓存未刷新
- 私有 DNS 与公网 DNS 混淆
- CoreDNS 转发上游配置不合理

### 10.3 DNS 和负载均衡的关系

DNS 常常负责“把流量先送到哪个入口”，而负载均衡负责“入口之后如何继续分发”。

这两个层面都重要，但职责不同：

- DNS 更偏控制面调度
- LB 更偏数据面分发

## 第十一章：安全与策略，网络不是只要能通

### 11.1 ACL、防火墙、安全组到底怎么区分

ACL 更像基础包过滤规则，通常基于源、目的、协议、端口等做匹配。

状态防火墙会跟踪会话，因此它不只是“看单个包”，而是能识别“这是某个已建立连接的回包”。

云上的 Security Group 和 NACL 也要区分：

- Security Group 通常更像实例级、状态型规则
- NACL 通常更像子网级、无状态或更底层的规则

### 11.2 为什么很多问题不是没路由，而是被策略挡了

典型现象包括：

- `ping` 通，但业务端口不通
- 白名单放了 IP，但没放端口
- 出站策略允许，回包被拦截
- Kubernetes Pod 能解析域名，但访问外部失败
- 多云互联物理连通，策略域没有放通

所以安全与连通一定要一起看。

### 11.3 零信任和微隔离为什么越来越重要

现代网络不再默认“内网就可信”。原因很简单：

- 网络边界越来越模糊
- 业务跑在云、容器、边缘多个环境
- 人员、服务、设备的身份都在变化

因此越来越多的网络设计，会把“基于位置的信任”转成“基于身份和最小权限的信任”。

## 第十二章：QoS 与性能，带宽只是表面数字

### 12.1 判断网络性能不能只看带宽

一个“好网络”至少还要看：

- 时延
- 抖动
- 丢包
- 重传
- 乱序
- 队列长度
- 尾时延

为什么很多业务“平均延迟不高，但用户体验很差”？因为用户真正感知的是尾时延，而不是平均值。

### 12.2 大象流和老鼠流为什么要区别对待

大象流常见于：

- 备份
- 数据复制
- 分布式存储
- AI 训练同步

老鼠流常见于：

- Web 请求
- RPC
- 服务注册与心跳
- 小文件传输

如果不做区分，大象流很容易挤占队列，导致老鼠流尾时延恶化。

### 12.3 微突发为什么常常肉眼看不到

很多链路监控按秒采样，看起来带宽并没打满，但交换机缓存里已经因为瞬时微突发出现排队和丢包。

这类问题常见于：

- 高并发 RPC
- 分布式存储写入
- AI 训练中的同步收敛阶段

所以真正做性能分析时，不能只看平均吞吐图。

## 第十三章：数据中心网络，从传统三层到 Spine-Leaf

### 13.1 为什么核心-汇聚-接入不适合无限扩展

传统三层分层架构在中小规模环境很好理解，但到了大规模数据中心，会逐渐暴露问题：

- 横向扩展有限
- 东西向流量越来越多
- 汇聚点容易形成瓶颈
- 扩容不够平滑

现代数据中心越来越倾向 Spine-Leaf 架构，核心原因是：让任意机架之间都能通过相似的多跳路径互通，并通过 ECMP 实现横向扩展。

### 13.2 Underlay 和 Overlay 必须同时理解

Underlay 是底层真实 IP 网络，负责提供可达性。

Overlay 是叠加网络，负责提供更灵活的租户隔离、二层延伸或逻辑抽象。

VXLAN 就是典型的 Overlay 技术，它把原有二层帧封装进 UDP 里，在三层网络上运输。

### 13.3 VXLAN 和 EVPN 为什么一起出现

VXLAN 解决的是“怎么封”，EVPN 解决的是“怎么学”和“怎么控”。

如果只有 VXLAN 没有好的控制平面，网络会退回到大量洪泛学习的模式，规模和运维体验都会变差。

EVPN 的价值就在于：

- 更优雅地分发 MAC/IP 可达信息
- 减少洪泛
- 支持多租户和大规模扩展
- 更适合现代数据中心自动化

### 13.4 Anycast Gateway 为什么重要

Anycast Gateway 的核心思想是：同一个网关地址可以出现在多个叶子节点上，主机无论挂在哪个叶子，第一跳体验都一致。

它的好处是：

- 简化主机网关设计
- 减少跨机架回流
- 更适合虚拟化和容器大规模迁移

## 第十四章：云网络，抽象得更高，但底层逻辑并没有消失

### 14.1 VPC 到底是什么

VPC 不是“云上的交换机”，而是云厂商提供的逻辑隔离网络环境。

你通常能看到这些抽象对象：

- VPC / VNet
- Subnet
- Route Table
- Security Group
- NACL
- Internet Gateway
- NAT Gateway
- Peering
- Transit Gateway

这些虽然名字不同，但背后仍然对应传统网络中的隔离、路由、出口、边界和互联问题。

### 14.2 为什么云网络看起来简单，排障却并不简单

因为云把很多底层细节藏起来了，但并没有消除复杂性，只是把复杂性迁移到了抽象控制层。

常见故障点包括：

- 子网路由表没指对
- 安全组没放通
- NACL 拒绝了回包
- NAT Gateway 没有正确关联
- 私网 DNS 解析错了
- 跨账号 VPC 互联缺少授权或传播配置

### 14.3 云网络和传统网络该怎么统一理解

不要把它们理解成两套完全无关的世界。

更好的理解方式是：

- 传统网络解决的是物理和协议层面的转发问题
- 云网络在这些能力之上，提供了更高层的逻辑抽象和托管控制

你学会传统网络后，再看云网络，会轻松很多。

## 第十五章：Kubernetes 网络，必须看到数据包到底怎么走

### 15.1 Kubernetes 网络模型的核心前提

Kubernetes 有一个极其重要的前提：理论上每个 Pod 都应该有自己的 IP，并且 Pod 之间默认可以直接通信。

这个设计的好处是：

- 容器像独立主机一样思考
- 应用不需要感知 NAT 级联
- 服务发现模型更自然

但代价是：底层网络实现会复杂得多。

### 15.2 Pod 到 Pod 是怎么通信的

这取决于 CNI 插件。

不同实现思路大致分三类：

1. 纯三层路由型  
例如某些 Calico 模式，节点知道 Pod 网段如何路由。

2. Overlay 型  
例如 Flannel VXLAN，跨节点时做封装。

3. eBPF 加速型  
例如 Cilium，在转发、策略和可观测性上更现代。

所以“Pod 之间能通信”这句话背后，实际实现方式可能完全不同。

### 15.3 Service 到底是什么

Service 不是一个真实网卡口，不是某台机器，也不是某个独立进程。

它本质上是一套“虚拟服务入口 + 后端 Pod 集合 + 转发规则”的抽象。

`ClusterIP` 的关键意义是：给一组变化中的 Pod 提供一个稳定入口。

### 15.4 kube-proxy 在做什么

传统理解里，kube-proxy 负责在节点上安装规则，让访问 Service IP 的流量被转发到实际 Pod。

它可能基于：

- `iptables`
- `IPVS`
- `eBPF` 相关替代实现

不同实现方式在以下方面会有差异：

- 性能
- 规则规模
- 调试方式
- 可观测性

### 15.5 外部流量怎么进入集群

最常见路径是：

```text
客户端
-> 公网或内网 LoadBalancer
-> Ingress Controller
-> Service
-> Pod
```

这里面每一层都不能混：

- `LoadBalancer` 解决的是基础入口暴露
- `Ingress` 解决的是 HTTP/HTTPS 七层路由
- `Service` 解决的是集群内部稳定服务入口

### 15.6 Kubernetes 网络最常见的生产问题

你以后看到 K8s 网络问题，优先从这些方向想：

- Pod IP 是否正常分配
- 节点间 Pod 网段是否可达
- CNI 是否异常
- `kube-proxy` 规则是否正确
- `CoreDNS` 是否稳定
- 出网 SNAT 是否正常
- MTU 是否一致
- `NetworkPolicy` 是否误伤
- Ingress、LB、NodePort 是否叠加出复杂路径

### 15.7 为什么 Kubernetes 网络一定要结合底层网络一起看

因为 Kubernetes 不是凭空运行的。它最终还是落在：

- 宿主机网卡
- Linux 路由
- iptables / nftables / eBPF
- VPC 路由
- 数据中心交换网络

所以你如果底层网络基础没打通，K8s 网络会越看越玄学。

## 第十六章：Service Mesh，不是替代网络，而是叠加治理

### 16.1 Mesh 改变了什么

Service Mesh 往往通过 Sidecar 或其他代理形式，把服务间流量接管到一个额外的数据路径里。

于是原来：

```text
应用 A -> 应用 B
```

变成了：

```text
应用 A -> 本地代理 -> 对端代理 -> 应用 B
```

这样做的好处是：

- mTLS
- 超时控制
- 重试
- 熔断
- 灰度发布
- 流量镜像

但代价也很真实：

- 延迟增加
- 资源开销增加
- 排障层次增加
- 配置复杂度增加

### 16.2 为什么 Mesh 不能替代底层网络

因为 Mesh 解决的是服务治理问题，不解决基础连通性问题。

如果底层 DNS、路由、MTU、出口 SNAT、节点间互通本身有问题，Mesh 只会把现象放大，而不是替你修好网络。

## 第十七章：可观测性与排障，网络工程的核心能力之一

### 17.1 网络排障的第一原则：先定义现象

不要一上来就说“网络坏了”，而要先明确：

- 是完全不通还是偶发不通
- 是只有某个端口不通还是所有端口不通
- 是只有白天慢还是一直慢
- 是单节点有问题还是整个区域都有问题
- 是首包慢、建连慢、吞吐差，还是偶发超时

现象描述不清，后面所有排查都会跑偏。

### 17.2 一套实用的排障顺序

生产里可以优先按这条顺序来：

1. 确认影响范围  
单机、单服务、单节点、单网段、单可用区还是全局。

2. 确认最近变更  
配置变更、扩容、升级、策略调整、证书替换、LB 切流。

3. 先看 DNS  
尤其是“域名访问异常但 IP 直连正常”的问题。

4. 看主机本地  
IP、路由、端口监听、连接状态、防火墙。

5. 看路径  
网关、路由、ACL、NAT、LB、回包。

6. 看应用层  
TLS、证书、超时、连接池、上游依赖。

### 17.3 最常用的网络排障工具

下面这些命令你必须熟：

- `ip addr`
- `ip route`
- `ip neigh`
- `ss -antup`
- `ping`
- `traceroute`
- `mtr`
- `tcpdump`
- `curl -v`
- `dig`
- `nslookup`
- `nc`
- `iperf3`
- `ethtool`
- `conntrack`
- `kubectl get`
- `kubectl describe`
- `kubectl logs`

### 17.4 证据比经验更重要

经验能帮助你更快猜到可能原因，但最后必须靠证据收敛。

所以成熟的网络排障一定包括：

- 抓包
- 会话状态
- 路由表
- 监控图
- 日志
- 流量路径还原

## 第十八章：AI 训练网络，为什么它和普通业务网络差别巨大

### 18.1 AI 训练网络首先要理解通信模式

普通 Web 业务大多是客户端到服务端的南北向请求。

而 AI 训练常见的是大量 GPU 节点之间的东西向通信，尤其是在分布式训练时。

当使用：

- 数据并行
- 张量并行
- 流水并行
- 专家并行

时，网络就不再只是“传请求”，而是在参与整个训练吞吐和收敛效率。

### 18.2 为什么 AllReduce 会把网络变成关键瓶颈

因为很多训练阶段需要各节点交换梯度或中间结果。

这类集合通信通常有几个特征：

- 带宽需求极高
- 延迟敏感
- 对抖动敏感
- 对丢包敏感
- 同步性强，慢节点会拖累全局

所以 AI 训练网络的目标不是“能通”，而是“稳定、低抖动、低尾延迟、高吞吐”。

### 18.3 RoCE、PFC、ECN 为什么会被频繁提到

因为当训练规模上来后，普通以太网 TCP 方案在某些场景下可能无法满足高性能需求，于是很多方案会走向 RDMA over Converged Ethernet，也就是 RoCE。

但这里面你一定要有清醒认知：

- RoCE 不只是换张网卡
- 它依赖交换机、队列、缓冲、拥塞控制的配套设计
- PFC 虽然可以减少丢包，但也可能引入拥塞扩散和 head-of-line blocking
- ECN / DCQCN 等机制常常需要一起考虑

所以 AI 训练网络不是“上更贵的设备就行”，而是一个完整系统工程。

### 18.4 InfiniBand 和以太网 RoCE 怎么理解

简化理解可以这样看：

- InfiniBand：更偏专用高性能互联，生态成熟于 HPC 和部分高端训练场景
- RoCE：在以太网体系内追求高性能，更容易与现有数据中心体系结合

选型时不能只比峰值带宽，还要看：

- 成本
- 运维团队能力
- 交换网络一致性
- 工具链和故障定位能力
- 与现有网络体系的兼容性

### 18.5 AI 训练网络的核心设计原则

常见原则包括：

- 尽量低阻塞甚至无阻塞
- 控制过订阅比
- 训练流量与存储流量分离
- 管理面与业务面分离
- 关注尾时延而不是只看平均吞吐
- 从 GPU 利用率反推网络设计是否合理

## 第十九章：AI 推理网络，重点不再只是 GPU 互联

### 19.1 推理网络和训练网络的思维方式完全不同

推理更像在线服务系统，因此重点会变成：

- 客户端接入体验
- 入口扩展能力
- 连接管理
- 限流与熔断
- 模型路由
- 缓存命中
- 区域亲和

训练看重东西向 GPU 互联，推理更看重南北向入口和服务链稳定性。

### 19.2 一个典型的大模型推理请求会怎么走

```text
客户端
-> DNS / GSLB
-> CDN / WAF
-> API Gateway / L7 LB
-> 鉴权与限流
-> 模型路由层
-> 推理调度层
-> GPU 实例
-> KV Cache / 向量库 / RAG 检索
-> 流式返回
```

这里每一层都会影响用户体验。

例如：

- API Gateway 配置不合理，会导致入口瓶颈
- 模型路由错误，会导致热点实例打爆
- 流式输出超时配置不合理，会导致长连接异常断开
- 向量数据库网络延迟高，会拖慢整个 RAG 响应

### 19.3 推理网络的几个关键点

做 AI 推理网络时，至少要重点看：

- L4 和 L7 入口如何分层
- 是否需要会话保持
- 模型是否按区域或机房亲和调度
- 推理层与检索层之间是否产生额外跨区网络开销
- 流式输出连接数是否会压垮入口网关
- 缓存层是否足够靠近推理节点

### 19.4 RAG 架构为什么对网络提出新要求

RAG 并不是“多查一次数据库”这么简单。

它通常意味着：

```text
用户请求
-> 网关
-> 编排服务
-> 向量检索
-> 重排
-> 大模型推理
-> 结果生成
-> 流式输出
```

这条链路变长以后，任何一跳的网络时延和抖动都会被放大到最终用户体验上。

## 第二十章：把整个网络世界真正串起来

### 20.1 一张统一的知识地图

到这里，你应该把网络理解成下面这张总图：

```text
用户请求
-> DNS
-> 主机路由
-> 默认网关
-> 二层交换 / 三层路由
-> NAT / 安全策略
-> 负载均衡 / 网关
-> 数据中心或云网络
-> Kubernetes Service / Ingress
-> Pod / 应用
-> 数据库 / 缓存 / 下游服务
-> AI 检索 / 推理 / 训练互联
-> 回包路径
```

这条路径里的每一个环节，都是网络知识体系的一部分。它们不是孤立的。

### 20.2 真正成熟的网络理解是什么

真正成熟的网络理解，不是会背多少术语，而是能回答这些问题：

- 这个请求现在走到了哪一层？
- 谁在决定下一跳？
- 谁在保存状态？
- 谁在做地址转换？
- 谁在做安全控制？
- 谁在决定后端实例？
- 谁在造成时延、抖动或瓶颈？
- 这个设计的单点、故障域、扩容边界和回滚点在哪里？

如果你能持续这样思考，你就已经不是在学“零散网络知识”，而是在建立网络架构能力。

### 20.3 学网络最有效的方式

建议按下面顺序反复训练自己：

1. 先看一条真实访问链路。  
2. 尝试画出请求怎么走。  
3. 把路径上的每个设备和组件职责写出来。  
4. 思考哪些地方可能出故障。  
5. 思考如何验证、如何回滚、如何扩容。  

只要你每学一个概念，都能把它放回真实路径里，这套知识就会越来越牢。

## 结语

网络不是某一门孤立课程，而是整个现代计算系统的血管系统。

传统业务、微服务、云平台、Kubernetes、AI 训练、AI 推理，看起来像不同世界，但本质上都离不开那几个核心问题：

- 流量从哪里来
- 流量往哪里去
- 中间经过谁
- 谁在做控制
- 谁在做转发
- 谁在做状态保持
- 谁在决定性能上限

当你真正把这些问题想清楚，网络就不再是一堆抽象协议，而会变成一张完整、可推理、可设计、可排障的系统地图。
